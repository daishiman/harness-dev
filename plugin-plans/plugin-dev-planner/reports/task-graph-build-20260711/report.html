実行フローの構造 — 内ループと外ループ

本質課題route モードの肝は、状態の正本を canonical graph から切り離すこと。graph を実行中に書き換えると graph_hash pin が毎遷移で壊れるため、構造 SSOT (task-graph.json)・可変状態 SSOT (task-state.json)・観測ビュー (task-progress.md) を三層に分離する。
解決アプローチ内ループは ready-set を機械算出して並列に検証し、結果を状態へ書き戻して再投影する定常回転。外ループは検証 FAIL を discovered-task として planner へ還流し、新 revision へ repin してから内ループへ再入する。
どう活かすかこの二重ループにより、部分進捗の可視化と『FAIL しても graph を汚さず改善を織り込む』還流が両立する。dispatch/write-back/discovered-emit の所有区分は harness-creator との producer/consumer 境界で固定されている。

実行は 2 つのループが入れ子で回る。内ループは 1 周ごとに『着手可能な node 集合』を確定して並列検証する定常回転、外ループは検証 FAIL を吸収して graph 自体を進化させる還流である。

内ループ (ready-set 駆動の定常回転)

  1. compute-ready-set.py が依存を満たし未完了の node 集合 (ready-set) を算出する。
  2. 各 node を独立 context の SubAgent へ dispatch し、二値受入基準で並列検証する。
  3. 検証結果を task-state.json へ done として書き戻す (canonical graph は不変)。
  4. project-task-status.py が状態を投影し task-graph-status.json / task-progress.md を再生成 → 次の ready-set 算出へ戻る。

外ループ (FAIL 還流による graph 進化)

  1. 内ループの検証が FAIL すると、その内容を discovered-task として planner の inbox へ emit する。
  2. planner drain が discovered-task を回収し、form (additive な追記) として plan へ反映する。
  3. graph を新 revision へ再導出し、graph_hash を更新して repin する。
  4. repin 済み graph で内ループへ再入する (pin 済み graph は直接更新しない=不変則)。
不変則 canonical graph を状態台帳として直接編集しない。status 更新は consumer 単一 writer による task-state 更新と投影で満たす。discovered-task は pin 済み graph を書き換えず、外ループで新 hash へ反映する。

この外ループが実際に何周回り、各周で何を還流したかを次に見る。

外ループ 3 周の詳細

本質課題外ループが実際に回った 3 周は、いずれも『内容の齟齬を検証 node が捕捉 → planner が additive に修復 → repin』という同一パターンで、graph 構造は 2 周目以降 不変を保った。
解決アプローチ各周の起点 FAIL・還流 form・drain 結果・graph 変化を突き合わせ、還流が graph_hash の安定性を壊さずに内容だけを進化させたことを確認する。

3 周のうち graph 構造 (node 数) が変わったのは Round 1 のみ (57→58)。Round 2/3 は phase 文書の内容修正に閉じ、graph は不変だった。すなわち外ループは『必要なときだけ構造を増やし、多くは内容を additive に締める』還流として働いた。

graph_hash の安定性
3 周を経ても最終 graph_hash は sha256:c636b6aa… の単一 pin に収束。Round 1 の node 追加は repin で新 hash へ移行し、以降の内容修正は hash を動かさなかった。pin の一致は依存関係整合の機械証跡になっている。

還流の到達点である 2 本の route build 結果へ進む。

route build 結果 (C01 / C02)

本質課題2 route はいずれも新規 component 追加ではなく、既存 skill への --mode update additive 差分。差分の実体は C17-C19 の実行契約強化で、既存の 13-phase task-graph.json を byte 不変に保つ後方互換設計が要だった。
解決アプローチC01 (producer) が実行契約 envelope とゲートを追加し、C02 (consumer=evaluator) がその契約を genuine 判定する semantic_check を追加する producer→consumer の順序で build した。

route は依存順 (C02 depends_on C01) で実行された。C01 が producer 側の実行契約と schema/renderer を確立し、C02 がそれを評価軸へ配線する。両者とも status=success で完走した。

C01 が追加した実行契約 (C17-C19)

  • C17 TaskExecutionEnvelope: node が title 単独でなく task_spec_ref / phase_policy_ref / component_route / acceptance / inputs / knowledge / verify を実質携帯し、追加質問なしに着手できる実行契約を optional additive で schema へ追加。
  • C18 三層状態 parity: project-task-status.py が構造 (graph)・状態 (task-state)・観測ビューの三層 parity を検査。単一 writer=consumer TG-C09 に確定 (elegant で二重 writer を根治)。
  • C19 cross-cycle knowledge: plan-ledger の predecessor_cycle_id と check-cycle-knowledge.py で、過去 cycle 知識を source_ref / freshness_checked_at / decision / reason 付きで有界注入。

C02 が追加した評価軸 (S8 / S9)

  • S8 execution-envelope: envelope が下流着手に十分な情報を実質携帯しているかを genuine 判定 (C17 の意味検査)。
  • S9 cycle-knowledge: 過去 cycle 知識が関連知見だけを有界注入しているかを genuine 判定 (C19 の意味検査)。
  • S8/S9 は S5-S7 と同一 shape の llm-only advisory (conditions=[]) で、C1-C4 の機械 verdict accounting (G1-G11) には非混入。
build 中の 2 つの逸脱 (開示済) (1) C19 の新規 plan-ledger.schema.json が repo-root .gitignore の広域ルール **/plan-*.json に名前衝突し未追跡化 (ローカル緑/CI 赤の gitignored source leak)。同一 plugin の先例に倣い negation 1 行で rescue。(2) C02 build の fork design-evaluate が session 上限で中断 → 機械検証で代替し、残差は elegant-review の 3 analyst 独立レビューで補完した。

これらの route が乗る 58 node の依存構造を次に俯瞰する。

依存グラフの構造 (13 phase DAG)

本質課題58 node の canonical graph は、13 phase の spine に各 phase の検証タスク (x-node) がぶら下がり、build 対象の 2 phase (P05/P09) が component route (C01/C02) へ写像される三種構成。
解決アプローチ全 58 node を羅列するのではなく、13 phase の骨格と node の 3 種別を示して構造の読み筋を与える。route への写像はダッシュエッジで表す。

graph の背骨は P01 requirements から P13 release までの 13 phase。各 phase は複数の検証 node (P0N-x-NN) を fan-out し、それらが phase 本体へ収束する。実装/QA を担う P05・P09 だけが component route へ写像される。下図は 13 phase を 8 つの論理ステージ群 に束ねた骨格で、P05→C01・P09→C02 の写像を該当ステージに注記した (1 図の node 上限に合わせた集約表現・全 phase を被覆)。

phase node (P01–P13)
13 phase の共通ライフサイクル契約。各 node は phase_ref で 1 つの phase policy を参照する。
検証 node (P0N-x-NN)
1 検証可能成果物の実行単位。phase 本体から fan-out し、二値受入基準で done 判定される (全 58 node の大半)。
route 写像 node
P05-C01-01 → run-plugin-dev-plan、P09-C02-01 → assign-plugin-plan-evaluator。build 対象 component への写像を持つ唯一の node 種別。
構造の要点
13 phase の直列 spine は不変のガードレール、可変なのは各 phase の検証 node 群。この分離により『固定 phase policy × 可変 task spec』が両立し、graph 全体は非循環 (DAG) を保つ。

構造が健全でも監査経路の欠陥は残りうる。次に思考リセット後のエレガント検証の所見を見る。

エレガント検証 (30 思考法) の所見

本質課題build 完走後の思考リセットが、機械ゲート緑では見えない high 3 件を捕捉した。いずれも『成果は出ているが記録・主張・所有が実体とズレている』という監査系の欠陥で、Goodhart 的な見かけの緑を剥がした。
解決アプローチPhase1 リセット俯瞰 → Phase2 3 並列独立分析 (論理構造 10 + メタ発想 9 + システム戦略 11 = 30 思考法・skip 0) → Phase3 改善実行。周回 1 で 4 条件全 PASS に到達した。
どう活かすか検出された high 3 件はいずれも恒久ガード (schema/validator/assert) 込みで根治し、次サイクルの再発を機械層で封じた。

30 思考法を 3 analyst へ並列配分し、思考リセット後の先入観なし俯瞰から重大度順に統合した。high 3 件はいずれも『build は success だが監査経路が断絶している』型で、機械テストの緑だけでは捕捉できなかった。

見かけの緑を剥がす
3 件はいずれも pytest 緑・completion_gate ok の状態で潜んでいた。特に knowledge 全損は『C19 (cross-cycle knowledge) を導入した当の cycle が自らの教訓を 1 件も残せていない』という自己矛盾で、思考リセットなしには見落とされていた。

最終 4 条件 (dispatcher 独立再検証済)

回帰確認は dispatcher 実走で独立に取り、plugin-dev-planner 824 passed / harness-creator 361 passed を確認した。medium 以下の所見 (form 冪等 skip の内容差分・fork agent 未伝播・安定 CLI 節の renderer 未掲載等) は Phase3 で B1-B6 として全解消済み。

未実証だった 3 点の決着

本質課題本サイクル前の plan review で runtime 実証が保留だった 3 点 (plan 再現性・テスト網羅・移行 gate) について、今回の build がどこまで決着させ、何を次サイクルへ送ったかを整理する。

残課題と次アクション

本質課題本 build のスコープ外に、baseline から持ち越しの検査赤と、次案件へ送る実生成例・軽量経路が残る。いずれも本サイクルの成果を毀損しない周辺項目である。
解決アプローチ残課題を『別 PR で解消する既存赤』と『次サイクル/次案件へ送る拡張』に分け、担当と状態を task として明示する。
  • 未完了[ ]legacy plan 2 件の既存検査赤 (lint-goal-seek / lint-script-frontmatter) を解消する。baseline HEAD で同一再現する pre-existing 条件で本 build boundary 外。別 PR で対応。(次 PR)
  • 未完了[ ]engine:task-graph の実生成例 (新規 plugin を task-graph route で 0 から生成) を次案件として実施する。(次案件)
  • 未完了[ ]MD-1 checklist-verify 軽量経路 / MD-8 plan-time renderability 一括検査 + knowledge_refs 必須キー / SS-5 bootstrap→target 移行 gate を実装する (Loop B knowledge へ起票済)。(次サイクル)
  • 完了[x]C02 fork design-evaluate の session 上限中断分は elegant-review 3 analyst の独立レビューで代替済 (残差なし)。(elegant-review)
本サイクルのスコープ確定 残課題はいずれも周辺項目で、58/58 done・4 条件 PASS の成果を毀損しない。既存赤は別 PR、拡張は次サイクル/次案件へ委譲することでスコープを閉じた。
