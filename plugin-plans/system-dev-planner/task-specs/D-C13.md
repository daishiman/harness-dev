# D-C13: System plan lock lifecycle manager

`taskType=implementation`、`visualEvidence=NON_VISUAL`。本タスクの単一責務は、C09 が確定した caller repository 内で system plan session lock の lifecycle を管理することである。readiness 判定、plan 生成、promotion、hook 配線は変更しない。

## Phase 1: 要件定義

- **目的**: lock が誰のものか、いつ有効か、どの feature を守るかを機械判定できる要件へ固定する。
- **実行タスク**: `run-system-dev-plan/SKILL.md`、C07 hook、C09 resolver、`.dev-graph/locks` 設定を読み、現行 owner 不在を `rg` で再確認する。AC-1 atomic acquire、AC-2 owner限定 renew/release、AC-3 TTL expiry、AC-4 audit cleanup、AC-5 malformed same-repo fail-closed、AC-6 containment、AC-7 deterministic receiptを確定する。
- **参照資料**: `plugins/system-dev-planner/skills/run-system-dev-plan/SKILL.md`、`plugins/system-dev-planner/hooks/guard-implementation-readiness.py`、`plugins/system-dev-planner/references/repo-local-runtime-contract.md`。
- **成果物**: lock field/operation/exit-code 要件表。
- **完了条件**: taskType、NON_VISUAL、7 AC、スコープ外が明記され、B-C09 未完了時は着手しない。
- **統合テスト連携**: C07 が読む filename と JSON field を fixture 契約として共有する。

## Phase 2: 設計

- **目的**: stdlib-only の単一 script と repo-local state の境界を設計する。
- **実行タスク**: `acquire|renew|release` を定義し、lock に `repository_id/run_id/session_owner/feature_id/feature_digest/acquired_at/heartbeat_at/expires_at` の exact-set を持たせる。acquire は fixed filename の exclusive create、renew は全 owner identity 照合後の atomic replace、release は同照合後のみ削除する。expired cleanup はこれらの操作内で監査 receipt を先に固定してから行う。exit code は `0=success`、`1=held/state conflict`、`2=usage/policy/IO fail-closed` とする。
- **参照資料**: C09 の `guard_relative_path()` と repository identity、C07 の lock scan 契約。
- **成果物**: operation matrix、state transition、error matrix。
- **完了条件**: 書込み先が C09 で解決した caller repository root 配下の `.dev-graph/locks/` に限定され、global temp/network依存がない。
- **統合テスト連携**: C07/C01 は script API を利用し、lock JSON を独自生成しない。

## Phase 3: 設計レビュー

- **目的**: 排他・安全・回復の4条件を実装前に審査する。
- **実行タスク**: 同時acquire、renew中断、wrong owner release、時計境界、malformed JSON、symlink escapeをレビューする。単純案「hook が lock を兼務」は責務混在のため却下する。
- **参照資料**: Phase 1 AC、Phase 2 state transition。
- **成果物**: PASS/MINOR/MAJOR 判定と指摘追跡表。
- **完了条件**: MAJOR 0件。B-C09/C09境界が未確定、または malformed lock の扱いがfail-openなら NO-GO。
- **統合テスト連携**: Phase 4 の negative cases が全ACと1:1対応する。
