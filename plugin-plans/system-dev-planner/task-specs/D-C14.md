# D-C14: System build handoff and registration receipt boundary

`taskType=implementation`、`visualEvidence=NON_VISUAL`。本タスクの単一責務は、staging packageからversioned handoffを決定論生成し、system-dev-plannerとdev-graphのregistration ownershipを契約化することである。exact-13 package生成、C12全体validation、C11 promotion、dev-graph mutation自体は担当しない。

## Phase 1: 要件定義

- **目的**: dangling output promiseを、生成物・schema・owner・entry gateを持つ実行可能契約へ変換する。
- **実行タスク**: C01 output contract、handoff draft、feature package/atomic promotion契約、C09 containmentを確認する。AC-1 versioned schema、AC-2 deterministic producer、AC-3 manifest hash包含、AC-4 exact-13 source refs、AC-5 registration owner分離、AC-6 promotion/receipt gate、AC-7 path containmentを明記する。
- **参照資料**: `system-build-handoff-draft.json`、`feature-execution-package-contract.md`、`atomic-promotion-contract.md`、C09 resolver。
- **成果物**: handoff field/owner/entry-gate要件表。
- **完了条件**: 7 ACとスコープ外が明記され、B-C09未完了時は着手しない。C14は生成前チェックを自己完結させ、C12をimportしない。
- **統合テスト連携**: build/runtimeとも base emit → C14 handoff生成 → C12最終検査の順に固定する。

## Phase 2: 設計

- **目的**: 自己参照digestを作らず、handoffとmanifestを決定論的に結合する。
- **実行タスク**: schemaに `schema_version/status/source/repository_id/feature_package_id/parent_feature/artifacts/entry_gates/registration/authorization` を定義する。producerはbase artifactsを検査→handoffをcanonical JSONでwrite→handoff SHAをmanifest filesへ追加→manifest canonical digestを再計算する。handoff自身へ最終manifest digest値を埋め込まず、`staging-manifest.json#canonical_digest` locatorで参照してdigest cycleを避ける。
- **参照資料**: C09 containment、staging-manifest/canonical digest規則、dev-graph registration contract。
- **成果物**: JSON schema、producer IO contract、ownership matrix、digest algorithm。
- **完了条件**: 同一bytesから同一handoff/manifestが生成され、manifestが自身をhash対象にせずhandoffを含む。
- **統合テスト連携**: B-C12は生成後manifest exact-set、schema、digestを検証するconsumerとなる。

## Phase 3: 設計レビュー

- **目的**: handoff生成、promotion、registration applyの責務混在を防ぐ。
- **実行タスク**: system-dev-plannerはregistration request/handoff、dev-graphはall-or-none applyと`dev-graph-registration-receipt.json`、system build側はreceipt検証後の実装authorizationを所有することをレビューする。producerがreceiptを自己発行する案は却下する。
- **参照資料**: Phase 2 ownership matrix。
- **成果物**: PASS/MINOR/MAJOR verdictと指摘追跡表。
- **完了条件**: MAJOR 0件。receipt owner不明、digest cycle、partial registration許容、C14→C12 順序の逆転があればNO-GO。
- **統合テスト連携**: Phase 4でowner境界ごとのnegative caseを作る。
