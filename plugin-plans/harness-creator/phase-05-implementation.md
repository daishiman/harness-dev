# P05 — implementation (実装設計)

## 目的
P04 で確定した C01-C08 の受入基準を Green にする最小実装を、関数シグネチャ・アルゴリズムの具体度で確定する。producer 側 (`plugin-plans/plugin-dev-planner/phase-05-implementation.md`) が既に確定した実インターフェース (`compute-ready-set.py` の呼出規約・`canonicalize()` の規約・`check-task-state-schema.py` の `validate_task_state()`/`check_graph_hash_pin()` 規約) と整合させる。

## 背景
producer 側 phase-05 により `compute-ready-set.py` の `main()` は `<plan_dir>/task-graph.json` を固定パスで読む単一位置引数 CLI (`compute-ready-set.py <plan_dir>`) であり、`--task-graph`/`--task-state` 等の flag は持たず、node の `state` は graph 内に既に埋め込まれている前提で動作することが判明している。この実インターフェースに合わせ、C01 の「マージ→cross-plugin 呼出」設計を確定する。

## 前提条件
- P04 の test-design が完了している。
- `plugin-plans/plugin-dev-planner/phase-05-implementation.md` の `compute-ready-set.py`/`derive-task-graph.py` 実装設計を前提として引き継ぐ (再実装しない・呼出規約のみ整合させる)。

## ドメイン知識

- **C01 実装設計 (新規)**: `scripts/dispatch-ready-set.py` に `merge_state(graph: dict, state: dict) -> dict` を新設する (`graph["nodes"]` の各 node について `state.get(node["id"], {}).get("state", node.get("state", "pending"))` を `node["state"]` へ上書きした浅コピーを返す・純関数)。`main()` のアルゴリズム: (1) `--task-graph`/`--task-state` を読み込み `merge_state()` でマージした graph を得る、(2) `tempfile.TemporaryDirectory()` 配下に `task-graph.json` としてマージ済み graph を書き出す (**この一時ファイルは compute-ready-set.py への入力受け渡し専用の非永続コピーであり、canonical serialization 契約 [constraints #3] が対象とする「正準構造として保存・比較される成果物」には該当しない。subprocess 終了後に自動破棄する**)、(3) `subprocess.run([sys.executable, "<planner_root>/skills/run-plugin-dev-plan/scripts/compute-ready-set.py", <tmpdir>], capture_output=True, text=True)` を実行する (producer 側 `ready_set()` アルゴリズムを再実装しない・cross-plugin subprocess 呼出のみ)、(4) subprocess の stdout (`{"ready_set": [...], "conflicts": [...]}`) を JSON parse し、`blocked` (`merged graph` 上で `state=="blocked"` の node id 一覧・単純フィルタでありアルゴリズムの複製ではない) を追加した `{"ready_batch": ready_set, "conflicts": conflicts, "blocked": [...], "graph_hash_pin": <下記>, "source": "compute-ready-set.py"}` を stdout へ出す。subprocess の returncode が非 0 なら exit1。**周回 graph_hash 再検証 (C10-d・F10)**: (2) の一時ファイル書き出し前に、`task_state.get("graph_hash")` が設定済みなら `verify_graph_hash_pin(task_graph_path: str, pinned: str, planner_root: str) -> bool` を呼ぶ。この関数は producer 側 `derive-task-graph.py --print-graph-hash <task-graph.json>` (read-only サブコマンド・FC-4) を subprocess 呼出しして現 task-graph.json の canonical hash を stdout で受け取り `pinned` と比較する (derive の write-mode main は起動せず task-graph.json を上書きしない・再実装しない)。一致なら出力 `graph_hash_pin="verified"` で処理継続、不一致なら `{"graph_hash_pin": "mismatch", ...}` を stdout へ出して exit1 (build 開始時 1 回の C07 pin 検証を素通りした実行中の graph 変更混入を dispatch 周回ごとに fail-closed で拒否)。pin 未設定 (初回) は `graph_hash_pin=None` で照合をスキップする。本 script は read-only を維持し照合のみで書き込みを持たない。

`canonicalize(graph: dict)` が対象とする `nodes`/`edges` の graph 形状ではなく、C16 schema の `nodes` 配列は node 単位の list であり sort key が異なる) を持つため producer 側関数をそのまま呼び出せない。constraints #3 の趣旨 (独自シリアライザの乱立防止) - **C07 実装設計 (新規)**: `scripts/manage-build-lease.py` は build 開始前安全性ゲートを担う。**lock 中身と孤児 steal (F2)**: `.build.lock` は空ファイルではなく `{"started_at": <UTC ISO8601>, "pid": os.getpid(), "host": socket.gethostname()}` JSON を書き込む。`acquire_lock(lock_path: Path, now: datetime, lock_ttl_seconds: int) -> str` のアルゴリズム: `os.open(lock_path, os.O_CREAT | os.O_EXCL)` で排他生成を試み成功なら中身を書いて `"acquired"` を返す。`FileExistsError` の場合は既存 lock の中身を読み、`is_stale = (now - started_at) > lock_ttl_seconds` (heartbeat 途絶) `or (host == 現ホスト and not pid_alive(pid))` (`pid_alive` は `os.kill(pid, 0)` の成否) を評価し、`is_stale` なら自分の中身で上書き再取得して `"stolen"` を返す (dispatcher クラッシュ後の残留 lock による恒久 lockout=C10-a resume の自己締め出しを防ぐ)、生存 lock なら `"already-held"` を返す。`renew_lock(lock_path: Path, now: datetime) -> None` は保持中 lock の `started_at` を `now` へ更新する (heartbeat)。`release_lock(lock_path: Path) -> None` は `lock_path.unlink(missing_ok=True)`。`find_expired_leases(task_state: dict, now: datetime) -> list[str]` は `nodes` を走査し `state == "running"` かつ `lease_expires_at < now` の `id` 一覧を返す (純関数・実書込みを持たない。dispatcher の `--renew-lease` heartbeat で延長された lease は将来時刻となり本関数の対象にならないため、正当な長時間 running は回収されない=F1)。`main()` のアルゴリズム: `--lock-action release` は `release_lock()` のみで exit0。`--lock-action force-release` は状態不問で無条件 `release_lock()` し `{"lock": "released"}` を出力し exit0 (人間の手動救済)。`--lock-action renew` は `renew_lock()` し `{"lock": "renewed"}` を出力し exit0 (dispatcher の heartbeat)。`--lock-action acquire` は (1) `resolve_build_dir(target_plugin_slug, cycle_id)` (C02 import) 配下の `.build.lock` へ `acquire_lock()` を試み、結果が `"already-held"` なら `{"lock": "already-held"}` を出力し exit1、`"acquired"|"stolen"` なら続行、(2) `find_expired_leases()` で孤児 task_id を検出し、各 task_id について `subprocess.run([sys.executable, "sync-task-state.py", "--task-id", tid, "--to-state", "pending", "--reap-lease", ...])` を呼び出し C02 へ実書込みを委譲する (C07 自身は task-state.json を直接書かない・単一 writer 規約の維持)、(3) `task_state.get("graph_hash")` が未設定なら producer 側 `derive-task-graph.py --print-graph-hash <task-graph.json>` (read-only・FC-4) を subprocess 呼出しで算出し `sync-task-state.py --pin-graph-hash <値>` で初回 pin、既設定なら producer 側 `check-task-state-schema.py` の `check_graph_hash_pin(task_state, task_graph)` を subprocess 呼出しして現 task-graph の再計算 hash と照合し、不一致なら `{"graph_hash_pin": "mismatch"}` を出力し `release_lock()` してから exit1 (fail-closed・実行中の graph 変更混入を拒否)、(4) 全て成功したら `{"lock": "acquired"|"stolen", "reaped_task_ids": [...], "graph_hash_pin": "pinned"|"verified"}` を出力し exit0 (lock は build 完了まで維持し dispatcher が明示的に `--lock-action release` を呼ぶまで解放しない)。

- **C03 実装設計 (新規・実 producer graph 形状に整合)**: `scripts/inject-task-inputs.py` に `resolve_inputs(graph: dict, state: dict, task_id: str) -> dict` を新設する。アルゴリズム: (1) 対象 node の `depends_on` (producer task id 一覧) を取得、(2) 各 producer の `state.get(pid, {}).get("state")` が `"done"` でなければ即座に `{"rejected": True, "reason": f"producer {pid} not done", "blocking_producer_task_id": pid}` を返す (fail-closed・最初に見つかった未完了 producer で打ち切り)、(3) 全 producer が done なら**各 producer の成果物を `node.write_scope` から解決する** (実 producer=derive-task-graph.py は `produces` エッジを生成せず成果物パスを node.write_scope に格納する。`produces` エッジが存在する場合は forward-compat で併読するが write_scope を主経路とする)。**成果物実在検査 (F5・state==done の代理述語に依存しない)**: 各 `artifact_path` を `os.path.exists()` で実在検査し、欠落があれば `{"rejected": True, "reason": "producer artifact missing", "blocking_producer_task_id": pid, "missing_artifact": artifact_path}` を返す (done state と成果物存在の乖離を fail-closed で捕捉)、(4) 各 producer の **state node の `handoff_notes` (dict: went_well/friction_points/downstream_watchouts の各 list・単一 writer C02 が実際に書く運用フィールド) を固定カテゴリ順で平坦化集約する** (flat `notes` list も後方互換で併読)。**上限の SSOT は schema (F8)**: 件数上限/文字数上限は producer 所有の `handoff-notes.schema.json` の `maxItems`/`maxLength` を `--notes-schema` (既定 producer 側 path) から読み取って使い、consumer 側で数値を再定義しない (件数上限は各カテゴリ maxItems の総和=1 producer が全カテゴリを埋めた最大件数・`--max-notes`/`--max-note-chars` は schema 値を上書きする任意フラグ)。超過なら `{"rejected": True, "reason": "notes bound exceeded"}` を返す。正常系は `{"injected_inputs": [...], "injected_notes": [...]}` を返す。テストは実 derive-task-graph.py 出力 (write_scope 成果物・handoff_notes) に対し injected_inputs/injected_notes が非空になる統合ケースで realized を固定する。
