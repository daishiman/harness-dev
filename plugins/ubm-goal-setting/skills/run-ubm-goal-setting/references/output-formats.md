## ファイル命名規則

| 種別 | パターン | 例 |
|------|----------|----|
| 週報 | `UBM - 1-週報 - YYYY-MM-DD〜YYYY-MM-DD.md` | `UBM - 1-週報 - 2026-03-23〜2026-03-29.md` |
| 月報 | `UBM - 2-月報（１ヶ月） - YYYY-MM-DD〜YYYY-MM-DD.md` | `UBM - 2-月報（１ヶ月） - 2026-03-02〜2026-03-29.md` |
| 期報 | `UBM - 3-月報（２ヶ月） - YYYY-MM-DD〜YYYY-MM-DD.md` | `UBM - 3-月報（２ヶ月） - 2026-02-09〜2026-04-05.md` |

## 数値ルール

- 数値は必ず**半角数字のみ**（「万円」「千円」不可 → 600000）
- 差分は必ず**「+」または「-」**を付けて記載（例: -300000, +2）
- 行動目標（週報＝今週／月報＝今月／期報＝今期 の優先度A/B/Cの各項目）には**期日と数値**を含め、**各項目の末尾に必ず期日トークンを付ける**
  - 期日トークンはバリデータ（validate-goal-output.py）が日付認識できる形にする: `M/D`（例 7/2）／`YYYY-MM-DD`／`毎日`／`毎朝`／`毎週`／`今週`／`〜まで` のいずれかを最低1つ含める
  - 「今月中」「8月中」など月単位の曖昧表現を単独で使わない。使う場合は具体日付トークン（`M/D` 等）を併記する
  - 箇条書きは `- ` / `- [ ]`（ハイフン）で書くのを推奨。ただし `・`（全角ビュレット）でもバリデータは受理する（Obsidianが `- `→`・` に自動変換するため両対応済み。どちらでもPASS）

## 【重要】目標・実績・差分セクションの記述ルール（厳守）

### 核心原則

**「目標 → 実績 → 差分 → 未達を挽回するための行動変化」が一目で見て分かること**が最優先。

各セクションは1つの責務のみを持つ：
- **目標**：数値・期日のみ
- **実績**：数値・完了日のみ
- **差分**：差分数値のみ（+/-記号必須）
- **行動変化**：補足説明・経緯・理由・改善方針 をすべてここに集約

実績セクションに延期理由・代替手段・経緯説明を書くと、目標 vs 実績の比較が読めなくなる。経緯・補足は必ず「未達を挽回するための行動変化」セクションに移す。

### 目標設計の流れ（上流→下流の逆算ルール・厳守）

目標は必ず **売上目標 → 成果目標 → 行動目標** の順で、上流から下流へ逆算して組み立てる。

```
[売上目標] （いくら稼ぐか・最上流）
    ↓ その売上を立てるためにどんな成果が必要か？
[成果目標] （先方判断・受諾・参加人数・合意など、コントロール不能な結果）
    ↓ その成果を出すためにどんな行動が必要か？
[行動目標] （自分でコントロール可能な動作・件数・期日）
```

#### 設計手順（必ずこの順番で対話・記述する）

1. **Step A: 売上目標の決定**
   - 今回（週／月／期）の売上目標を1つの数値で確定する
   - 売上の内訳（誰から・いくら）は対話では確認するがファイルには書かない（数値1つだけ）

2. **Step B: 売上目標から成果目標を逆算**
   - 「その売上が立つためには、誰の・どんな先方判断・合意・受諾・継続が必要か？」を問う
   - 各成果目標は売上目標のどの部分を支えるかが説明できる必要がある
   - 例：売上60万 → 森田さん業務委託継続合意1件・tenant摩擦ゼロ運用継続1件・新規受諾1件 …

3. **Step C: 成果目標から行動目標を逆算**
   - 各成果目標について「その先方判断・合意を引き出すために、自分はどんな行動を何件・いつまでにするか？」を問う
   - 各行動目標は **どの成果目標に紐づくか** を説明できる必要がある
   - 例：森田さん継続合意1件 → 1週目スコープ握りMTG実施1回・週次レポート送付4通・月20時間枠ルール提示1回 …

4. **Step D: 自作成物（成果物完成目標）の整理**
   - 行動の前提となる自作成物（資料・テンプレ・システム）は別途「成果物完成目標」として明記
   - 行動目標と区別する（行動目標 = 動作、成果物完成目標 = 完成物）

#### 設計時の問い（対話で必ず使う）

- 売上→成果：「この売上を立てるには、誰の・どの判断が必要か？」
- 成果→行動：「この合意を引き出すには、自分は何を・何件・いつまでにやるか？」
- 行動→成果：「この行動を全部やったら、本当にこの成果に到達するか？足りてるか？」（逆方向の整合性チェック）

#### NG例（流れが組み立てられていない）

- ✗ 売上目標と無関係な成果目標が並ぶ（例：売上目標は60万なのに、成果目標が「アカデミー参加1回」だけ）
- ✗ 成果目標と紐づかない行動目標が並ぶ（例：成果目標に森田さんがあるのに、行動目標に森田さん関連の動作がない）
- ✗ 行動目標を先に決めてから成果目標を後付けする（下流→上流は禁止）

#### OK例（上流→下流が一貫している）

```
[売上目標] 600000円
    ↓
[成果目標]
- 森田さん：1週目スコープ合意1件　期日4/28
- tenant：摩擦ゼロ運用継続　期日5/24
- 月1MTG7名：日程確保7件　期日5月中
    ↓
[行動目標]
- 森田さんキックオフMTG実施1回（4/28 18:00）／週次レポート送付4通（5月）  ← 森田さん成果目標を支える
- tenant4点セット作成・先方共有・週次レポート送付2通  ← tenant成果目標を支える
- 月1MTG7名へ日程打診メッセージ送付（4/28〜4/30）  ← 月1MTG成果目標を支える
```

各行動目標が **どの成果目標を支えるか** が一目で読み取れることが必須。

### 成果目標と行動目標の区別ルール（厳守）

**成果目標** = 自分でコントロールできない結果（先方判断・参加人数・受諾・売上額など）
**行動目標** = 自分でコントロールできる行為（実施・送付・参加・作成など）

「実施する」「開催する」「参加する」「送付する」など自分の行動で完結するものは行動目標側に置く。
「受諾される」「合意してもらう」「N人来る」「契約成立する」など先方判断や数の結果は成果目標側に置く。

#### なぜ分けるか（差分を別々に取るため・重要）

成果は行動の「結果」であり、行動と1対1では対応しない。行動したが成果につながらないことも、意図せず行動が良い成果を生むこともある。だから行動目標（やった/やらない）と成果目標（結果が出た/出ない）は必ず別セクションに分け、振り返りでそれぞれ独立に差分を取れるようにする。混在させると「行動はできたのに成果が出なかった／行動は少なかったが成果は出た」という学習ができなくなる。

#### 分離の実装ルール（厳守）

- 各行動目標の末尾に `→ 支える成果目標：XXX` を付け、対応する成果目標を明示する
- その `XXX` は必ず成果目標セクションにも1項目として置く（行動側だけに書いて成果側に無いのは禁止）
- 「面談する／定義する／日程を確定する／告知する／打診する／作成する」等の自分で完結する動作は行動目標に置き、成果目標には置かない
- 「合意される／継続される／受注する／N名参加する／紹介が出る」等の先方・市場の結果は成果目標に置き、行動目標には置かない
- 成果物（資料・定義・提案書）の完成は行動（作成する）として書き、その成果物が引き出す先方判断を成果目標に置く
- 関係維持・習慣など成果が測りにくい行動は、無理に成果化せず `→ 支える成果目標：（関係維持）` のように置いてよい

#### 成果→行動の完全な落とし込み（標準粒度・厳守）

成果目標を並べるだけでは不十分。**各成果目標に、それを生む行動目標を「準備→資料作成→（練習/リハ）→実施→フォロー」の標準粒度で必ず紐付ける**（1成果につき2〜4行動が目安）。行動側が「実施する」1行だけで準備・作成が抜けていると、成果につながる計画になっていない。

- 例（成果「日工：継続/次回研修の合意」）→ 行動「研修資料を作成（期日）」「研修を実施（期日）」「研修後に継続/次回をヒアリング（期日）」の各行に `→ 支える成果目標：日工 継続/次回研修の合意`
- 例（成果「勉強会：参加申込5名以上」）→ 行動「企画を作る」「会場を確保」「告知チラシ作成・告知」「申込フォーム作成」
- **振り返り（前週/前月/前期）でも同様**に、前期間の各成果を生んだ行動を行動目標・実績・差分に並べ、「行動はやったか」「成果は出たか」を別々に差分できるようにする
- 粒度の既定は「標準」（準備→作成→実施→フォローの4前後）。細かすぎる各回分解や、実施1行だけの粗さは避ける

#### 成果目標としてOKな例
- 日工さん：研修売上75,000円　期日4月中旬（売上＝結果）
- 森田さん：単価合意1件・業務委託開始1件　期日4/2（先方判断＝結果）
- 阿部さん：Claude Code提案受諾1件・月1MTG合意1件　期日4/10（先方判断＝結果）
- 田村さん：GWS提案受諾1件・方向性整理1件　期日4月中（先方判断＝結果）
- AI質問会勉強会：参加者5名以上　期日4/18（人数＝結果）
- アカデミー：実績報告1件　期日4/25（自分の行動だが「報告物として残す」結果として扱う）

#### 行動目標として書くべき例（成果目標側に書いてはいけない）
- 「研修実施1回」 → 行動目標
- 「自主開催1回」 → 行動目標
- 「MTG実施1回」 → 行動目標
- 「アカデミー参加」 → 行動目標
- 「提案実施」 → 行動目標

#### よくある混在の修正例
- ✗ NG：「AI質問会勉強会：自主開催1回」（成果目標に書いている → 開催は行動）
- ✓ OK（成果目標）：「AI質問会勉強会：参加者5名以上」
- ✓ OK（行動目標）：「AI質問会勉強会：自主開催1回」

成果目標セクションの冒頭には次の注釈を入れる：
> 成果目標は「自分でコントロールできない結果（先方判断・参加人数等）」を記述する。実施行為は行動目標側に置く。

### 要因分析セクション（重要・必須）

各「未達を挽回するための行動」セクション（全レベルの新フォーマットで「前週/前月/前期の◯◯に対して未達を挽回するための行動」という名称。前期間語（前週/前月/前期）が変わるだけで役割は同じ）では、フォーマット通りの行動を箇条書きで書いた直後に、別見出し（H3）で **要因分析** を付ける。

- 行動変化セクション本体：フォーマット通り（`- **未達項目** → 行動変化` の形）
- その下に H3 で要因分析セクションを追加：「なぜダメだったのか／なぜうまくいったのか」を分析的に記述

#### 構造例

```markdown
## 【前月の売上に対して未達を挽回するための行動】
- **未達項目1** → 次の行動変化
- **未達項目2** → 次の行動変化
- **未達項目3** → 次の行動変化

### 売上未達の要因分析（なぜ60万円が0円になったのか）

**根本要因**：
1. **要因1見出し**：状況の詳細・なぜそれが起きたか・何を見落としたか（2〜4文）
2. **要因2見出し**：同上
3. **要因3見出し**：同上

**学習**：今回から得た普遍的な教訓を1〜2文でまとめる
```

#### 上振れ実績の要因分析（目標外で達成された場合）

目標外で達成された実績（例：tenant契約成立）がある場合、未達分析と同じ形で「なぜうまくいったのか」を分析する：

```markdown
### {{上振れ項目}}（目標外上振れ）の要因分析と次への行動変化

**契約が決まった要因**：
1. **要因1**：何が機能したか・どう作用したか
2. **要因2**：同上

**次への行動変化（再現可能化）**：
- **再現アクション1** → 具体的に何をいつまでにやるか
- **再現アクション2** → 同上
```

要因分析の目的：単に「未達でした」「成功しました」で終わらせず、構造的な原因を言語化することで、次の行動の質を上げる。フォーマットだけを満たして終わると学びが蓄積されない。

### 売上系セクション（売上目標・売上実績・差分）

**そのセクションには数値1つだけ**を記述する。

- ✓ OK: `600000`
- ✓ OK: `-600000`
- ✗ NG: 表（| 売上源 | 目標 | 根拠 |）を入れる
- ✗ NG: 「### 前回売上目標の内訳」のサブセクションを足す
- ✗ NG: 説明文を足す

内訳・根拠を残したい場合は、目標設定の決定プロセスとして対話で確認するだけにとどめ、出力ファイルには記述しない。

### 成果目標・行動目標セクション

**箇条書きのみ**。Markdown 表（table）の使用は禁止。

#### 目標セクション

`- 項目名：数値・期日` の形

例：
- `- 日工さん研修：1回・75000円　期日4月中旬`
- `- 森田さん業務委託開始：1件　期日4/2`

#### 実績セクション（特に厳守）

**「目標に書いた項目すべて」を必ず実績側にも書く**。実績数値（できた件数・回数・金額）と完了日のみで記述。

- 形式：`- 項目名：実績数値・完了YYYY-MM-DD（または完了月日）` の形
- 例：
  - `- 日工さん研修：3回・完了4月中旬`
  - `- 森田さん業務委託開始：1件・完了4/16`
  - `- 海老名さん単価決定：1件・完了4/15／業務委託開始：0件`
  - `- 阿部さんGWS提案：0件`
  - `- 田村さんGWS方向性：0件`

**実績セクションに書いてはいけないもの**：
- 「→次のフェーズへ」「先方都合で延期」などの理由・背景説明
- 「実施」「未達」「次回確認」などの抽象的状態
- 「合意未達→受注」のような経緯説明
- 「○/△/×」「(参考)」などの記号
- 売上計上未確認・代替成果の経緯など

**実績に書くべきもの**：
- 目標項目それぞれに対する **件数（数値）** と **いつ完了したか（日付・期日）**
- できなかったものは `0件` と記述
- 数値が分割される場合は項目を分ける（`単価決定：1件・完了4/15／業務委託開始：0件`）

**目標外の追加実績**：
目標になかったが期間中に実現した件は、実績セクションの末尾に「（追加実績）」サブ見出しの下に箇条書きで追加してよい。形式は同じ（`- 項目名：実績数値・完了日`）。

#### 差分セクション

`- 項目名：差分数値` の形（例: `- 日工さん研修：+2回`）

- 必ず `+` または `-` を付ける
- 説明文・理由・背景は書かない

#### 説明・理由はどこに書くか

実績の延期理由・代替実現・軸転換の経緯・先方都合などは、すべて以下のセクションに集約する：

- 「【前{週/月/期}の売上に対して未達を挽回するための行動】」
- 「【前{週/月/期}の売上以外の成果に対して未達を挽回するための行動】」
- 「【前{週/月/期}の行動に対して未達を挽回するための行動】」

これらのセクションは散文・箇条書きどちらでもよく、根本原因→次月への行動変化を記述する。

### 表形式の使用可否

| セクション | 表形式 | 箇条書き |
|---|---|---|
| 売上目標・実績・差分 | 禁止（数値1つだけ） | 不要（数値1つだけ） |
| 成果目標・実績・差分 | 禁止 | 必須 |
| 行動目標・実績・差分 | 禁止 | 必須 |
| やらないこと | 禁止 | 必須 |
| MTG設計・スケジュール | 許可 | 許可 |
| 北原・高山アドバイス | 許可（参考） | 許可 |

## 週報（1週間目標）フォーマット

週報は視認性のため `---` で2ゾーンに分ける。**ゾーンA（ロールアップ用ヘッダー・準静的＋前週実績）** と **ゾーンB（今週の実行・毎日見る所）** の2層構成。ゾーンAは月報/期報へロールアップする公式ヘッダーと前週レビュー、ゾーンBは週報固有のチェックリスト群（最重要数字・到達ライン・優先度ABC・習慣目標・判断基準・個別メッセージ・週末振り返り）を置く。

### ロールアップ契約（週報→月報/期報の集約規則）

週報のゾーンAを上位（月報/期報）へ集約する際は次の契約に従う。見出し名は改名せず、規約に沿って機械的に集約する。

- (a) 集約は見出し名キーの再マッピングで行う。期間語（今週/今月/今期、前週/前月/前期）を正規化した「【{期間}の…】」キーでルックアップし、ゾーン位置に依存せず（ゾーン非依存ルックアップ）、上位フォーマットの確定セクション順へ再ゾーニングする。本文は N:1 で集約し、見出しキーは 1:1 で対応させる。
- (b) 週報固有見出し（今週の位置づけ・今週の最重要数字・今週の大きな到達ライン・習慣目標（仕組みで動く土台）・今週送る個別メッセージ・今週の判断基準・今週末の振り返りチェックリスト）は集約対象外。上位へロールアップしない。
- (c) 月報固有見出し（確認された情報・北原さんの原則・要因分析H3・プロジェクト別タスク）は週報集約では供給されず、月報側で新規生成する。

### チェックリスト形式の徹底ルール（最優先・厳守）

週報は「済/未済を二値で判定できる」ことが最優先。以下のセクション（すべて**ゾーンB**に属する）は必ず `- [ ]` チェックボックス形式で出力する。**簡潔版を求められても、ユーザーがプロジェクト別タスクの省略を指定しても、これらのチェックボックス化は絶対に外さない。**

| セクション | 形式 |
|-----------|------|
| 今週の最重要数字 | `- [ ]` 各項目 |
| 今週の大きな到達ライン | `- [ ]` 日別（`6/8（月）：...` の形で1日1行以上） |
| 今週の行動目標（行動管理・優先順位付き） | 優先度A/B/Cの `### ` 見出し＋配下に `- [ ]` 各項目（合計3項目以上） |
| 習慣目標 | カテゴリ別 `### ` 見出し＋各カテゴリに `- [ ]` 2項目以上（最低6チェックボックス） |
| 今週やらないこと | `- [ ]` 各項目（3項目以上） |
| 今週末の振り返りチェックリスト | 3ブロックの `- [ ]` |

#### 実行管理の2方式（どちらでもよいが、行動目標のチェックボックス化は必須）

- **方式1: プロジェクト別タスク中心** — 各案件を `- [ ] [期日] [提出先・宛先] 対象物・行動` で列挙する
- **方式2: 優先度ABC＋日別到達ライン中心** — プロジェクト別タスクは省略し、優先度ABC（行動目標）と日別到達ラインのチェックボックスで実行管理する

ユーザーが「プロジェクトごとのタスクは不要」と指定した場合は**方式2**を採用する。その場合でも、優先度ABC・日別到達ライン・習慣目標・振り返りチェックリストは必ずチェックボックスで出力する。バリデーション（validate-goal-output.py）は行動目標が `- [ ]` 形式で3項目以上あることを必須チェックし、到達ライン・プロジェクト別タスクは任意（WARN）として扱う。

## 期報（2ヶ月目標）フォーマット

期報は月報（上記「月報フォーマット」）の確定セクション順をそのまま使い、以下の読み替え規則を適用する。

### 読み替え規則

- サマリー見出しは `## 【2ヶ月の目標】{{start_date}}〜{{end_date}}`（継続語彙・凍結。改名禁止）。
- 振り返り部の「前月」はすべて「前期」に読み替える（例: 【前期の売上目標】【前期の行動目標（行動管理）】）。
- 計画部の「今月」はすべて「今期」に読み替える（例: 【今期の売上以外の成果目標】【今期の行動目標（行動管理・優先順位付き）】）。
- 【今期の行動目標（行動管理・優先順位付き）】の各項目末尾にも、月報・週報と同様に期日トークン（`M/D`／`YYYY-MM-DD`／`毎日`／`毎朝`／`毎週`／`今週`／`〜まで`）を必ず付ける。月単位の曖昧表現は具体日付を併記する。箇条書きは `・`/`- ` どちらでも可（Obsidian変換対応）。
- 【確認された情報】の目標期間は「2ヶ月」とする。
- スナップショット項目（今期の累計売上実績・現在事業パートナー数・現在のグリッドパートナー数・前回/次回のアカデミーへの参加日・次回の壁打ち予定日・次回の壁打ち予定月までに目指している状態・次回の壁打ち予定日までに達成させること）は月報と同じ位置・名称で含める（期間語化しない）。
- **ヘッダーの【今期の売上目標】（月報の期アンカー項目）は期報では出力しない**（採用案A1）。期報では計画部の【今月の売上目標】→【今期の売上目標】がその役割を兼ねるため、ヘッダー側の【今期の売上目標】を重複出力しない。これで期報の見出し集合は【今期の売上目標】が一意になる。
- **【プロジェクト別タスク】は期報では出力しない**（プロジェクト別タスク方針: 月報＝必須／週報＝任意／期報＝禁止）。
- **【今月の日別タスク】（読み替えで【今期の日別タスク】に相当）は期報では出力しない**（日別タスクはプロジェクト別タスクの日付展開のため月報のみ。期報は【確定スケジュール】で代替する）。

#### 規則の優先順位と不変条件（厳守）

本規則（見出し集合・セクション順・A1・期報禁止セクション）は動的学習に優先する。旧期報に【今期の売上目標】重複や【プロジェクト別タスク】があっても、それらを再生産しない。

不変条件:
- 期報は【今期の売上目標】を1回のみ出力する（ヘッダー側を出さず、計画部の1個に統一する）。
- 期報は【プロジェクト別タスク】を出力しない。
- 期報は【今期の日別タスク】を出力しない（日別タスクは月報のみ）。

その上で、月報の確定セクション順の末尾（今期末の振り返りチェックリストの後）に、以下の期報固有セクションを加える:
