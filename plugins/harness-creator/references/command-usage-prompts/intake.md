# /intake 用途プロンプト（30思考法エレガント検証つき）

## ① スラッシュコマンド入力

まずこれを打つ。

```
/intake "<やりたいこと>" [--page-url <Notionページ>]
```

`run-skill-intake` (11 phase) が起動し、5軸ヒアリング（出力先・情報源・共有相手・真の課題・ナレッジ資産）+ user_profile 6軸推定 + 図解を経て `intake.json` / `intake.md` / Notion ページが生成される。plugin 規模の構想と判定された場合は mode P として `plugin-dev-planner` への引き渡しを推奨し、そこで停止する（スキル生成は自動起動しない）。

### オプション

`/intake` の全オプション（正本 = `commands/intake.md` の argument-hint）。①では代表として `--page-url` のみ例示:

| オプション | 何を入れるか | 用途 |
|---|---|---|
| `"<topic>"`（任意・位置引数） | やりたいことの自然文。省略時は kickoff で対話確定 | ヒアリング対象テーマ |
| `--page-url <notion-page-url>` | Notion ページ URL | 既存ページへ update 公開。解決不能なら新規作成せず停止 |
| `--page-id <notion-page-id>` | Notion ページ ID | URL の代わりに ID で出力先を指定 |
| `--database-id <notion-db-id>` | Notion DB ID | 公開先 DB の明示指定 |

## ② 構造化プロンプト（説明）

### Layer1 基本定義

- **プロジェクトID**: `intake-elegant-review`
- **最上位目的**: `/intake` が生成した `intake.json` / `summary.json` / `interview.json`（5軸 + user_profile 6軸）が、非エンジニアの真の目的・要望・対象スキルを過不足なく捉えているかを、思考リセット後に30種の思考法で多角的に検証し、エレガント（最小複雑性で最大要件充足）な状態へ改善する
- **背景**: 非エンジニアヒアリングは聞き漏れ・思い込み・対象取り違え・専門用語の押し付けが起きやすく、Gate A（Phase 8 summarizer）通過後でも真の目的とズレたまま Notion 公開・Phase 11 次アクション推奨に進むリスクが残る
- **期待成果**: 4条件（矛盾なし／漏れなし／整合性あり／依存関係整合）を全て満たすエレガントなヒアリング成果（真の目的・要望・対象スキル同定・5軸プロファイル）
- **成功基準**: 30思考法を全て適用した後、4条件が全て PASS している
- **スコープ**
  - 含む: `/intake` 実行後の `intake.json` / `summary.json` / `interview.json`（5軸 + user_profile）の検証・改善提案
  - 含まない: `run-skill-intake` 内部実装（スクリプト・schema）の変更／Notion 公開の再実行／`run-skill-create` 等スキル生成の起動

### Layer2 ドメイン定義

**用語集**

- 思考リセット = 思考のクリア（ヒアリング結果を先入観なく再読すること）であり、生成済み成果物（`intake.json` 等）の削除ではない
- エレガント = 最小複雑性（ヒアリング往復・確認質問を増やさない）で最大要件充足（5軸＋真の目的＋対象同定を過不足なく満たす）状態
- 5軸 = 出力先(output_target)／情報源(info_source)／共有相手(share_target)／真の課題(true_problem)／ナレッジ資産(knowledge_assets)
- user_profile 6軸 = 熟練度／役割／文脈／制約／動機／共有意図

**30思考法一覧**

| カテゴリ | 思考法 |
|---|---|
| 論理分析系 | 批判的思考／演繹思考／帰納的思考／アブダクション／垂直思考 |
| 構造分解系 | 要素分解／MECE／2軸思考／プロセス思考 |
| メタ・抽象系 | メタ思考／抽象化思考／ダブル・ループ思考 |
| 発想・拡張系 | ブレインストーミング／水平思考／逆説思考／類推思考／if思考／素人思考 |
| システム系 | システム思考／因果関係分析／因果ループ |
| 戦略・価値系 | トレードオン思考／プラスサム思考／価値提案思考／戦略的思考 |
| 問題解決系 | why思考／改善思考／仮説思考／論点思考／KJ法 |

**検証4条件（この用途での読み替え）**

| 条件 | 読み替え |
|---|---|
| 矛盾なし | 要望どうし、または目的と要望が矛盾しない（例: 「効率化したい」と言いつつ手作業確認を増やす要望が混在していないか） |
| 漏れなし | 5軸（output_target／info_source／share_target／true_problem／knowledge_assets）全てに `verified=true` が付き placeholder が残っていない |
| 整合性あり | `technical_level` が非技術のユーザーに専門用語（API／OAuth 等）が言い換えなしで登場せず、図解・選択肢の粒度が user_profile と一貫している |
| 依存関係整合 | 真の課題(true_problem) → 要望 → 対象スキル同定(mode A-E/P) の依存が論理的に繋がっている（逆算しても崩れない） |

**ビジネスルール**

- CONST_001: 思考リセット＝クリア。`intake.json`／`intake.md`／Notion 公開済みページは削除・改変しない。改善提案は別ファイルまたは追記ヒアリング probe として出す
- CONST_002: 30種全て使用。1つでも省略した状態は未完了とする
- CONST_003: 4条件が全て充足するまで完了扱いにしない

### Layer3 インフラ

- **Agent Team（SubAgent並列）**: 思考法カテゴリ別に Agent2（論理・構造）／Agent3（メタ・発想）／Agent4（システム・戦略・問題解決）を並列起動し、同一入力（`intake.json`／`summary.json`／`interview.json`）を渡す。文脈汚染防止のため各 SubAgent は context:fork
- **Codex（改善の外部委譲）**: 5軸の言い換え文言・追加ヒアリング probe 候補のドラフト生成を委譲可能。採否は Agent5 が最終判断する

### Layer4 共通ポリシー

- 30思考法を全て使用する（1つでも欠落があれば未完了）
- 改善提案には適用した思考法を根拠として明示する（例: 「why思考の結果、〜」）
- 4条件（矛盾なし／漏れなし／整合性あり／依存関係整合）を全て充足する
- **エスカレーション**: 改善方向がエージェント間で割れた場合（例: 対象スキル mode 判定が割れる）はユーザーに確認する。5軸の再ヒアリング（Phase 4 差し戻し）や Notion ページ再公開など大幅な構造変更はユーザー承認を要する

### Layer5 エージェント定義

- **Agent1（思考リセット俯瞰）**: `intake.json`／`summary.json`／`interview.json` をバイアスなく俯瞰読了する。「これは削除でなくクリア」を前提に、`kickoff.json` の元発話まで遡り、summarizer 要約時に切り落とされた文脈がないか一次確認する

- **Agent2（論理・構造、9種）**
  - 批判的思考: `summary.json` の「これで完成」という前提を疑い、5軸の `verified=true` が実質的な裏取り済みかを再点検する
  - 演繹思考: `user_profile`（熟練度等）から導かれるはずの語彙・図解粒度が `interview.json` 内の実際の質問文と一致しているか検証する
  - 帰納的思考: `kickoff.json` と `interview.json` の複数発話断片から共通する true_problem パターンを抽出し、summary の真の課題記述と一致するか確認する
  - アブダクション: ユーザーの断片的発話から「最もありそうな真の目的」を仮説形成し、`true_problem` がその最良の説明になっているか検証する
  - 垂直思考: 5軸テンプレ通りの深掘りで不足がないか、1軸ずつ深さを掘り下げて確認する
  - 要素分解: `intake.json` を5軸の各要素に分解し、各要素が独立して検証可能な粒度で書かれているか点検する
  - MECE: 5軸＋user_profile 6軸が重複なく漏れなく網羅されているか（同一内容が2軸に重複／どの軸にも属さない要望が孤立していないか）確認する
  - 2軸思考: 「熟練度×動機」等の2軸マトリクスで要望を再配置し、summary が偏った軸に依存していないか確認する
  - プロセス思考: kickoff→interview→purpose→summary の流れで情報が段階的に絞り込まれ、途中で目的がすり替わっていないか追跡する

- **Agent3（メタ・発想、9種）**
  - メタ思考: 「ヒアリングで聞くべきことを聞けているか」を一段上から問い、質問設計自体の妥当性を検証する
  - 抽象化思考: 個別要望（例: 「フォーム作りたい」）から本質的ニーズ（情報収集の自動化）を抽出し、対象スキル同定が具体機能名に引っ張られていないか確認する
  - ダブル・ループ思考: 質問手順（question-bank.md 由来の深度・順序設定）自体を疑い、今回のユーザー像に最適な設計だったか検証する
  - ブレインストーミング: true_problem に対する代替解釈を複数列挙し、summary が唯一の解釈に飛びついていないか確認する
  - 水平思考: 「対象スキルは既定パターンのどれか」という枠を外し、mode P（plugin 規模）のような非定型の対象同定漏れがないか探る
  - 逆説思考: 「聞かない方が良い情報」はないか（share_target=顧客時の機密性配慮・マスキング要否）を逆向きに検証する
  - 類推思考: 類似ヒアリング事例（過去 fixtures 等）と比較し、今回の summary が典型パターンから外れて見落としがないか照合する
  - if思考: 「もし user_profile の推定が誤っていたら」を仮定し、語彙・図解選択がその場合でも破綻しないか耐性を確認する
  - 素人思考: 非エンジニアが `intake.md`／`summary.md` を読んで専門用語なしで理解・同意できるかを、専門知識ゼロの視点で読み直す

- **Agent4（システム・戦略・問題解決、12種）**
  - システム思考: intake→Notion公開→next-action 推奨という一連の系全体で、一箇所の曖昧さが後工程（mode 判定／harness-creator 引き渡し）へどう波及するか俯瞰する
  - 因果関係分析: 「なぜこの要望が出たか」の原因（true_problem）と結果（要望）の因果を明示し、summary が結果だけを記録し原因を落としていないか確認する
  - 因果ループ: constraints（制約）が motivation（動機）を強化・抑制する循環を追い、5軸間の相互作用が記録から欠落していないか点検する
  - トレードオン思考: 「詳しく聞く」と「ユーザー負担を増やさない」の対立を両立させる質問設計か（過剰ヒアリングで離脱させていないか）検証する
  - プラスサム思考: ヒアリング精度向上とユーザー体験向上が同時に成り立つ改善（既存回答の言い換え確認等）を優先しているか確認する
  - 価値提案思考: 対象スキル同定（mode A-E/P）がユーザーの動機（効率化／品質等）にどう価値を返すかが `intake.md` に明記されているか確認する
  - 戦略的思考: `next-action.json` の mode 判定が share_target=顧客等の中長期運用まで見据えた選択になっているか検証する
  - why思考: 「なぜこの機能が欲しいのか」を3段階遡り、true_problem が表層要望(what)止まりになっていないか確認する
  - 改善思考: 現状の `summary.json`／`intake.json` の言い回し・構成を小さく直す余地（冗長な確認質問の削除等）を探す
  - 仮説思考: 「本当の要望は別にあるのではないか」という仮説を立て、`interview.json` の `pending_probes`／`needs_excavation` フラグと整合するか検証する
  - 論点思考: 「今回のヒアリングで本当に決着すべき論点は何か」を明確化し、summary がその論点に答えているか確認する
  - KJ法: 断片的な発話・回答をグルーピングし直し、5軸への割付が最も自然な分類になっているか（強引な当てはめがないか）検証する

- **Agent5（改善実行）**: Agent2/3/4 の結果を統合→4条件の未充足度で優先順位付け→改善作業を独立並列／依存直列に SubAgent 分割して実行→4条件を再検証する

### Layer6 オーケストレーション

Phase1 思考リセット俯瞰（Agent1、必須ゲート・省略不可）→ Phase2 3エージェント並列分析（Agent2／Agent3／Agent4、全完了までゲート）→ Phase3 改善実行（Agent5）。最大反復3回（4条件が未充足なら Phase1 へ戻り反復、3回で強制停止しユーザーへ未達条件を報告）。
