# /run-skill-feedback 用途プロンプト（30思考法エレガント検証つき・収集のみ）

> **前提の明記**: 本コマンドは**収集のみ**を行う。構造化フィードバックを Notion 改善要望 DB へ投入するところまでが責務であり、改善の実装・修正・着手は一切自動発火しない（人間ブリッジは `plugins/harness-creator/references/feedback-to-improvement-runbook.md`）。ブリッジ先で要望が actioned される際は `/plugin-dev-plan --mode update` 経由で task 仕様書＝`task-graph.json` へ還流し、次回 `/capability-build --handoff`（既定=task-graph route モード）が改善済み依存グラフで再駆動する。**本収集はその spec-improvement サイクルの起点**であり、収集品質が下流の task-graph 反映精度を左右する（ゆえに投入前チェックが重要）。以下の 30 思考法検証も「収集内容が schema 準拠かつ文脈ズレなく投入前チェックを通るか」だけを対象とする。

## ① スラッシュコマンド入力

まずこれを打つ。

```
/run-skill-feedback
```

※ 実態は `plugins/harness-creator/skills/run-skill-feedback`（`kind: run`, `user-invocable: true`）の **skill**。専用の command ファイルは存在せず、`/` 起動可能な skill として分類される。

### オプション

`/run-skill-feedback` は**引数・オプションを取らない**（対話起動のみ）。専用 command ファイルが無いため argument-hint も存在しない。起動後の対話で「対象同定 → 現状仕様提示 → 要望収集」を進める。②の投入前チェックで参照する唯一の CLI が `python3 scripts/notion-submit-improvement.py --dry-run`（対象プラグインのスキル一覧 DB 登録確認 = C4 判定）。

## ② 構造化プロンプト（説明）

以下は、収集した改善要望ドラフトを Notion へ投入する**直前**にエージェントへ渡す 7 層構造プロンプト。思考リセット → 30 思考法並列分析 → 4 条件セルフ検証 → 是正、を経てから投入する。

### Layer 1: 基本定義層（不変原則）

**用語集**
- 対象同定: 利用者はプラグイン名・スキル名を知らない前提で、目的から逆算して対象スキルを当てる工程
- 収集: improvement-request schema 準拠の構造化データを Notion 改善要望 DB へ新規ページとして投入する工程（改善実行ではない）
- 孤児レコード: 対象プラグインがスキル一覧 DB に未登録のまま投入された要望レコード
- 投入前チェック: Notion 投入直前に行う C1〜C4 の自己検証（本プロンプトの主眼）

**不変ルール**
- **CONST_001（リセット＝削除でなくクリア）**: 「たぶんこのプラグインだろう」という決めつけや下書き文面は物理削除せず、判断を保留してゼロから確認し直す
- **CONST_002（30 種全使用・省略禁止）**: 論理構造 9 ／ メタ発想 9 ／ システム戦略問題解決 12 の全 30 思考法を必ず適用する。1 つでも欠落したら投入前チェックは未完了
- **CONST_003（4 条件全充足）**: C1〜C4（下記）を全て PASS するまで Notion へは投入しない
- 本プロンプトは**収集のみ**を検証する。改善の実装・修正は範囲外（Agent5 は改善実行者ではなくセルフ検証者）

### Layer 2: ドメイン層（本質ロジック）

**schema 必須項目**（`doc/notion-schema/improvement-request.schema.json`）: 要望タイトル / 対象プラグイン（relation 必須）/ 対象スキル名 / 要望種別 / やってほしいこと / 背景・困っていること / 優先度 / 重要度

**検証 4 条件（読み替え版）**
- **C1 矛盾なし**: 要望内容が対象スキルの現状仕様（SKILL.md Purpose & Output Contract）や他の直近要望と矛盾しない。矛盾があれば明示する
- **C2 漏れなし**: schema 必須項目が全て埋まっている
- **C3 整合性あり**: 同定した対象プラグイン・スキルと要望本文の内容が一致し文脈ズレがない
- **C4 依存関係整合**: 対象プラグインがスキル一覧 DB に登録済み（`--dry-run` 確認済み）で孤児レコードにならない

**30 思考法カタログ（全 30 種・この用途への適用観点、省略禁止）**

論理分析系（Agent2 担当・5 種）
1. 批判的思考: 「本当にこのプラグインで合っているか」を利用者の一言だけで即断せず、発言のどの部分がどの機能に結びつくか根拠まで疑って遡る
2. 演繹思考: schema 必須項目という一般則から、収集済み個別回答が本当に各項目を満たしているか当てはめて検証する
3. 帰納的思考: 過去の改善要望や類似ヒアリングから「同定を誤りやすいパターン」を一般化し、今回の対話が同じ轍を踏んでいないか照合する
4. アブダクション: 利用者の断片的発言（例:「途中で止まった」）から最も妥当な対象プラグイン・要望内容の仮説を 1 つに収束させる
5. 垂直思考: 「なぜそう感じたか」を Why で繰り返し、表面的な不満表現から真の困りごと（背景欄）まで一直線に深掘りする

構造分解系（Agent2 担当・4 種）
6. 要素分解: 収集タスクを「対象同定／現状仕様提示／要望本文聴取／schema 必須項目充足／投入前確認」に分解し各要素を個別検査する
7. MECE: schema 必須項目チェックリストを漏れなく重複なく構成し、全項目が埋まっているか確認する
8. 2 軸思考: 優先度×重要度の組み合わせが利用者発言のトーンと整合するか（緊急感の割に「低」等の齟齬）を象限で点検する
9. プロセス思考: 目的ヒアリング→対象同定→現状仕様提示→要望聴取→存在確認→投入、の工程順序に gap や飛ばしがないか時系列で検査する

メタ抽象系（Agent3 担当・3 種）
10. メタ思考: 「収集フォーマットそのもの」が利用者の困りごとを表現しきれる枠組みか、schema 前提自体を疑う
11. 抽象化思考: 今回の個別要望を「他ユーザーが同種スキルで抱きがちな不満パターン」に汎化できるか考える
12. ダブル・ループ思考: 過去の類似要望・前回投入時の学習が、今回の同定・聴取に活きているか確認する

発想拡張系（Agent3 担当・6 種）
13. ブレインストーミング: 対象プラグインを 1 件に絞る前に、キーワードから連想できる候補スキルを 5 件以上発散的に列挙してから絞り込む
14. 水平思考: プラグイン名を知らない利用者向けに、技術用語でなく「作業内容・症状」からの逆引きという一見遠回りな経路で対象を当てる
15. 逆説思考: 「もしこの要望を収集しなかったら」を考え、収集しないと失われるもの（孤児化防止・文脈記録）を洗い出し必要性を再評価する
16. 類推思考: 類似の既存要望（Notion DB 上の過去レコード）と比較し、今回の要望が新規事象か重複的傾向かの差分理由を問う
17. if思考: 「もし候補が 1〜3 件に絞れなかったら」という極端ケースを想定し、全スキル一覧提示のフォールバックが機能するか破壊試験する
18. 素人思考: 利用者はプラグイン名もスキル名も知らない前提を徹底し、専門用語抜きで対話文面が理解できるか初見目線で検査する

システム系（Agent4 担当・3 種）
19. システム思考: 今回の投入が「未対応要望数」rollup を通じて他のスキル一覧 DB 運用・優先度判断全体にどう波及するかを俯瞰する
20. 因果関係分析: 「利用者が不満を感じた原因」→「どの機能仕様に起因するか」→「要望内容」という因果連鎖を明示する
21. 因果ループ: 対象同定を誤る→文脈ズレ要望が投入される→誤った優先度シグナルが積み上がる、という負のループを検出し断ち切る

戦略価値系（Agent4 担当・4 種）
22. トレードオン思考: 「対話を簡潔にする」と「schema 必須項目を漏れなく埋める」という一見相反する要求を両立できる聴取順序を探る
23. プラスサム思考: 利用者の摩擦最小化と開発チームの構造化データ取得を両方満たす聴取設計になっているか確認する
24. 価値提案思考: 「誰の何を解決する要望か」を要望タイトルと「やってほしいこと」欄の整合で問い直す
25. 戦略的思考: 数ヶ月後の改善サイクル（plugin-dev-planner 等）で読んでも文脈が分かる記述になっているか中長期視点で点検する

問題解決系（Agent4 担当・5 種）
26. why思考: 「なぜこの要望が必要か」を背景・困っていること欄が答えられているか、表層要望と真因のズレを追跡する
27. 改善思考: 前回同種投入で指摘された不備（孤児レコード・schema 未充足等）が今回再発していないか PDCA で確認する
28. 仮説思考: 「対象はこのプラグインだろう」という結論を先行させ、その仮説を現状仕様提示でユーザーに検証させる設計になっているか確認する
29. 論点思考: 「何が決まれば残りの聴取項目が決まるか」という本質論点（対象スキルの確定）を最優先で同定できているか点検する
30. KJ法: 収集済みの複数チェック項目の不備をボトムアップで島グループ化し、構造的な抜け漏れパターンを発見する

### Layer 3: インフラ層（外部依存）

- Agent Team 並列実行: Agent2 / Agent3 / Agent4 は独立 context-fork で並列起動し、互いの中間結果を参照しない
- Codex 委譲（任意）: 専門領域の深掘りが必要な finding は外部委譲可能（本 skill の必須経路ではない）
- 参照リソース: `plugins/harness-creator/skills/run-skill-feedback/SKILL.md`（局面カタログ・feedback_protocol 準拠の対話）／ `doc/notion-schema/improvement-request.schema.json`（必須項目正本）／ `doc/notion-schema/skill-list.schema.json#feedback_protocol`（発火条件・同定フロー正本）
- 外部ツール: `python3 scripts/notion-submit-improvement.py --dry-run`（対象プラグインのスキル一覧 DB 登録確認、C4 判定に使用）／ Read・Grep（SKILL.md 走査・現状仕様提示）

### Layer 4: 共通ポリシー層

- 30 思考法は全種使用必須。1 つでも欠落したら該当 Agent を再実行（省略・黙示スキップ禁止）
- 収集内容の是正は、どの思考法で検出した issue かを根拠として明示する（例:「why思考で検出: 背景欄が空欄」）
- C1〜C4 は全 finding で評価し、問題なしの条件も `issues: []` で明示（黙示 skip 禁止）
- エスカレーション: 対象プラグイン未登録／token 未解決／3 周回で C1〜C4 未達 → 投入を強制せず利用者へ差し戻し（force pass 禁止）

### Layer 5: エージェント層（実行主体定義）

**Agent1（リセット俯瞰）**
- 目的: 利用者の発言をバイアスなく受け止め、対象プラグイン名の決めつけを排して素の事実を観察する
- 背景: 先入観を持ち込むと目的逆算同定（`run-skill-feedback` の identification_step）が形骸化し孤児・文脈ズレの温床になる
- 達成ゴール: 「どんな作業をしていて何を感じたか」の生の発言と、絞り込み前の候補プラグイン群が事実／仮定に分離記録された状態
- 完了チェックリスト:
  - [ ] 既存の思い込みをクリアし記録から排除した（削除でなく保留＝CONST_001）
  - [ ] 利用者の発言を事実と仮定に分離した
  - [ ] 対象候補を 1 件に絞らず複数候補として保持し Agent2〜4 の検証に委ねた

**Agent2（論理構造分析・9 思考法）**
- 目的: schema 必須項目の網羅性と対象同定の論理整合を、論理分析系＋構造分解系の 9 思考法で検証する
- 背景: 演繹的な当てはめ検証を怠ると必須項目の形式的充足だけで内容矛盾を見逃す
- 達成ゴール: 上記 1〜9 の思考法観点で findings（`issues: []` 含む）が全て生成された状態

**Agent3（メタ発想・9 思考法）**
- 目的: 収集フォーマット自体の妥当性と対象同定の発散的検証を、メタ抽象系＋発想拡張系の 9 思考法で行う
- 背景: 利用者はプラグイン名を知らないため、素人思考・水平思考を欠くと専門用語前提の対話に陥り同定が破綻する
- 達成ゴール: 上記 10〜18 の思考法観点で findings が全て生成された状態

**Agent4（システム戦略問題解決・12 思考法）**
- 目的: 要望投入が周辺システム（rollup・優先度判断・改善サイクル）へ与える影響を、システム系＋戦略価値系＋問題解決系の 12 思考法で検証する
- 背景: 対象同定ミスは因果ループで誤った優先度シグナルを積み上げるため、根本原因の追跡が必須
- 達成ゴール: 上記 19〜30 の思考法観点で findings が全て生成された状態

**Agent5（収集内容の 4 条件セルフ検証＝投入前チェック。※改善実行ではない）**
- 目的: Notion 投入前に収集済みドラフトへ C1〜C4 を適用し、投入可否を判定する
- 背景: 投入後の要望は時系列ログとして重複除去されず永続するため、投入前の検証漏れは孤児レコード・文脈ズレレコードとして蓄積し優先度シグナルを汚染する
- 達成ゴール: C1〜C4 が全て PASS した状態、または FAIL 項目が特定され是正のため収集対話（局面カタログ）へ差し戻された状態
- 完了チェックリスト:
  - [ ] C1 矛盾なし: 要望内容が現状仕様・他の直近要望と矛盾しない（矛盾は明示）
  - [ ] C2 漏れなし: schema 必須項目が全て埋まっている
  - [ ] C3 整合性あり: 同定した対象と要望内容が一致し文脈ズレがない
  - [ ] C4 依存関係整合: 対象プラグインがスキル一覧 DB に登録済み（`--dry-run` 確認済み）
  - [ ] 改善実行（実装・修正）は行っていない。FAIL は是正対話へ差し戻すのみで、収集フローの範囲内に留めた

### Layer 6: オーケストレーション層

- **Phase1（リセット必須ゲート）**: Agent1 が対象同定前の素観察を行い、`run-skill-feedback` の局面カタログ「対象スキルの同定」を実行。事実／仮定の分離が完了するまで Phase2 へ進まない
- **Phase2（3 並列分析ゲート）**: Agent2 / Agent3 / Agent4 が Phase1 のドラフト（候補プラグイン群＋要望案）を入力に、担当 9／9／12 思考法で独立並列に findings を生成。30 件全て埋まるまで完了しない
- **Phase3（投入前セルフ検証・是正）**: Agent5 が findings を C1〜C4 に集約し判定。全 PASS なら実際の Notion 投入処理（`notion-submit-improvement.py`）へ進む。FAIL なら Phase1 の局面カタログ（対象同定 or 要望収集対話）へ差し戻す
- ハンドオフ: Phase1→Phase2（候補プラグイン群＋要望ドラフト）／ Phase2→Phase3（findings 30 件）／ Phase3→投入 or 差し戻し

### Layer 7: UI／提示層

**使い方**: まず ①（`/run-skill-feedback`）を実行し、スキル本来の対話（目的ヒアリング→対象同定→現状仕様提示→要望収集）を進める。要望ドラフトが揃った Notion 投入直前で、この ② のプロンプトをエージェントへ渡し、思考リセット（Agent1）→30 思考法並列分析（Agent2／Agent3／Agent4）→4 条件セルフ検証（Agent5）→是正、を回す。C1〜C4 が全て PASS してから初めて Notion 改善要望 DB へ投入する。本プロンプトのどの段階でも改善の実装・修正（改善実行）は行わない。
