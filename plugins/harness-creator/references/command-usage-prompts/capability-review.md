# /capability-review 用途プロンプト（30思考法エレガント検証・analyse-only）

## ① スラッシュコマンド入力

まずこれを打つ:

```
/capability-review <target-path> [scope_mode]
```

例:

```
/capability-review plugins/harness-creator/skills/run-build-skill
```

`scope_mode` は省略可 (省略時 `skill`。`plugin` / `repo` でレビュー幅を横断拡張)。

### オプション

`/capability-review` の全オプション（正本 = `commands/capability-review.md`）:

| オプション | 何を入れるか | 用途 |
|---|---|---|
| `target-path`（必須） | レビュー対象の絶対 or リポジトリ相対パス | 何を診断するか |
| `scope_mode`（任意） | `skill`（既定）/ `plugin` / `repo` | レビュー幅。`plugin`/`repo` で横断拡張 |

> `--dry-run` は本コマンドが内部で常時付与し Phase3（改善）を封じる（analyse-only）。ユーザーが手で渡すオプションではない。

## ② 構造化プロンプト（説明）

本コマンドは `run-elegant-review` を **analyse-only (`--dry-run`)** で起動する薄いラッパである。Phase 1 (思考リセット) → Phase 2 (30思考法並列分析) までを実行し 4 条件 verdict と findings を返すが、**Phase 3 (改善実行) は発火させない**。改善は findings 上の提案提示に留まり、実ファイルへの書き換えは一切行わない。

### Layer1 基本定義層（不変原則）

- 本コマンドは対象 capability (skill/agent/hook/command/composition) を多視点で検証する専用レビューであり、修正コマンドではない。目的: 診断と適用の責務分離。背景: 診断と修正を1コマンドに混ぜると誤ったverdictでの自動修正が回帰を招く。
- **改善は実行しない**: Phase3 (パッチ適用) を発火させず、findings整理・改善提案の提示までに留める。目的: read-onlyでの安全な現状診断。背景: `--dry-run` により write/auto-commit を物理的に禁止し、判断は人間へ委ねる。
- CONST_001 思考リセット=クリア(削除でなくクリア): 対象への先入観・既存rubric語彙を持ち込まず素の観察から始める。目的: バイアスのないレビュー。背景: 先入観混入は新規発見を抑制する。
- CONST_002 30種全思考法使用省略禁止: 1つでも欠落したらfalse-pass扱い。目的: 網羅性の機械的保証。背景: 部分適用は見落としの温床。
- CONST_003 4条件全充足判定: 矛盾なし/漏れなし/整合性あり/依存関係整合をそれぞれPASS/FAIL判定する。目的: 客観的verdict。背景: 曖昧な「良さそう」判断を排除する。

### Layer2 ドメイン層（用語集・30思考法・4条件）

用語集:
- capability: レビュー対象のskill/agent/hook/command/composition単位
- finding: 1つの思考法レンズで検出した観察+課題(思考法根拠を明示)
- verdict: 4条件のPASS/FAIL判定結果
- analyse-only: パッチ適用を伴わない診断限定モード(`--dry-run`)

30思考法(全30種・対象capabilityへの1行観点。省略禁止):

論理分析系
1. 批判的思考: 対象の前提・主張そのものの妥当性を疑う
2. 演繹思考: 規約/契約という一般原則から個別実装が正しく導けるか確認
3. 帰納的思考: 対象内の個別ケースから一貫パターンが成立するか確認
4. アブダクション: 観察された挙動から最も説明力のある背後仮説を推定する
5. 垂直思考: 既定の手順・階層構造を順序通り深掘りし抜け漏れを確認

構造分解系
6. 要素分解: 対象を最小構成要素まで分解し各要素の責務を確認
7. MECE: 要素分割が漏れなくダブりなくなっているか確認
8. 2軸思考: 2つの独立軸(例: 責務×依存)で対象をマッピングし歪みを確認
9. プロセス思考: 対象の処理フロー(入力→処理→出力)の各段階が繋がっているか確認

メタ抽象系
10. メタ思考: 対象の設計方針そのものを俯瞰し前提の妥当性を問う
11. 抽象化思考: 対象の具体実装から本質パターンを抽出し他への転用可能性を確認
12. ダブル・ループ思考: 対象の課題が表面対処で済むか根本方針の見直しが要るかを見極める

発想拡張系
13. ブレインストーミング: 対象の弱点に対する改善候補を数量制約なく発散させる(提示のみ・適用はしない)
14. 水平思考: 通常の枠を外れた別分野の解決策が対象に転用できないか探る
15. 逆説思考: 対象の前提を逆転させたら何が起きるかで盲点を炙り出す
16. 類推思考: 類似構造を持つ別capabilityとの比較で対象の欠落を発見する
17. if思考: 「もし〜だったら」の仮想条件で対象の耐性・脆弱性を確認する
18. 素人思考: 前提知識のない視点で対象の説明・命名がわかりやすいか確認する

システム系
19. システム思考: 対象を単体でなく周辺依存を含む系として捉え歪みを確認
20. 因果関係分析: 対象内の問題事象の直接原因を特定する
21. 因果ループ: 対象内のフィードバックループ(自己強化/自己抑制)の有無を確認

戦略価値系
22. トレードオン思考: 対象への改善案が別の価値を犠牲にしないか両立点を探る
23. プラスサム思考: 対象の改善が関係者全員にとって利得になる設計か確認
24. 価値提案思考: 対象が利用者に提供する価値そのものが明確かを確認
25. 戦略的思考: 対象の設計が中長期の運用方針と整合しているか確認

問題解決系
26. why思考: 表面課題に対し「なぜ」を繰り返し根本原因へ遡る
27. 改善思考: 現状とのギャップから具体的な改善候補を抽出する(提示のみ・適用はしない)
28. 仮説思考: 「おそらくこうだろう」という仮説を立て検証観点を設計する
29. 論点思考: 何が本当に議論すべき論点かを絞り込む
30. KJ法: 発散したfindingsを類似性でグルーピングし構造化する

4条件(検証観点):
- C1 矛盾なし: 対象内の記述・契約間に矛盾がないか
- C2 漏れなし: 対象が扱うべき責務・ケースに抜けがないか
- C3 整合性あり: 命名・構造・契約が一貫しているか
- C4 依存関係整合: 対象の依存方向・参照先が正しく機能するか

### Layer3 インフラ層

- Agent Team並列実行: Phase2は3エージェント(論理構造/メタ発想/システム戦略問題解決)が独立context-forkで並列起動、互いの中間結果を参照しない
- Codex委譲: 大規模対象や専門領域は任意で外部委譲可能(analyse-onlyでも観察range拡張目的で利用可)
- 入力: `target-path`(必須) / `scope_mode`(任意、既定skill)
- 出力先: `eval-log/<plugin>/<skill>/elegant-review/<run-id>/` 配下 (raw_observations / findings / verdict)

### Layer4 共通ポリシー層

- 30思考法全使用(必須・省略禁止)
- 改善提案は思考法根拠を明示すること(どのlensで検出したかをfindingsに残す)
- 4条件全てPASS/FAIL判定を出すこと(暗黙skipは false-pass扱いで禁止)
- **analyse-only制約**: Phase3(パッチ適用)は起動しない。改善はfindingsとしての提示に留め、実際のファイル変更は一切行わない
- エスカレーション: 4条件が最大3周回でも収束しない場合は`human_escalate`とし、`force_pass`は禁止

### Layer5 エージェント層

- Agent1 リセット俯瞰(Phase1相当): 対象を先入観なく素観察し事実/仮定を分離、read-only厳守
- Agent2 論理構造担当(9思考法並列独立): 批判的思考/演繹思考/帰納的思考/アブダクション/垂直思考/要素分解/MECE/2軸思考/プロセス思考
- Agent3 メタ発想担当(9思考法並列独立): メタ思考/抽象化思考/ダブル・ループ思考/ブレインストーミング/水平思考/逆説思考/類推思考/if思考/素人思考
- Agent4 システム戦略問題解決担当(12思考法並列独立): システム思考/因果関係分析/因果ループ/トレードオン思考/プラスサム思考/価値提案思考/戦略的思考/why思考/改善思考/仮説思考/論点思考/KJ法
- **Agent5 改善提案の提示(適用はしない)**: 通常フローの「改善実行」を読み替え、findings整理・4条件verdict・改善提案のドラフトのみを成果物とする。実ファイルへの変更・patch適用は行わない

### Layer6 オーケストレーション層

- Phase1 リセット必須ゲート: 対象への素観察完了が次工程への必須条件、bias未リセットならabort
- Phase2 3並列分析ゲート: Agent2/3/4が独立並列で30 findings + 4条件verdictを生成、担当思考法網羅(9/9/12)が完了条件
- **Phase3 提案提示まで(書き換えなし)**: 通常はここでパッチ適用するが、analyse-onlyではfindingsを改善提案としてまとめ提示するのみ。`changed_paths`は空、実ファイル改変なしで完了レポートを出す
- FAIL時導線: verdictがFAILの場合、改善実行を案内する。**in-place で直すなら `/skill-improve <target-path>`（plan/task-graph は非更新・速い）／plan 正本を保ち改善を task-graph へ還流するなら `/plugin-dev-plan --mode update --improvement-handoff <handoff>`（次回 `/capability-build --handoff` が既定=task-graph route モードで改善済み依存グラフを再駆動）**。対象が task-graph 駆動 build の成果物なら乖離を避けるため後者を推奨する

### Layer7 UserInput

まず①のスラッシュコマンド `/capability-review <target-path> [scope_mode]` を実行する。指定した対象capabilityに対し、思考リセット(Phase1)→30思考法並列レビュー(Phase2)→4条件(矛盾なし/漏れなし/整合性あり/依存関係整合)のverdictを出す。改善は提案(findings)に留め、実適用は行わない。実際にパッチを適用したい場合は、この結果を踏まえ別途 `/skill-improve <target-path>` を実行すること。
