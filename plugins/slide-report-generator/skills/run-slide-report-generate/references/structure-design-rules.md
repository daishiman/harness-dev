### ビジュアル形式振り分けルブリック（部分AI画像化時のみ）
部分AI画像化（バランス型）を指定された場合のみ、各スライドを以下の3形式に振り分ける。

**振り分け前の前置除外（上書き不可・退化耐性最優先）**:
- コード（slide-code / slide-code-compare）・数式・精密数値表は無条件に実HTMLコードブロック側（コード専用ページ）に確定する。ヒアリングや形式振り分けで画像化に上書きできない。コード系 slideType はこの除外を最優先で適用し、以下3形式の判定に入れない。

| 形式 | 判定条件 | 画像の役割 |
|------|----------|-----------|
| HTML主役（画像なし） | コード（slide-code / slide-code-compare）・数式・精密数値表・表・料金・比較マトリクス・数値の塊・密度高・逐語が頻繁に変わる | 画像を生成せず現行コンポーネントを温存（コード・数式・精密数値表は実HTMLコードブロックで描画） |
| イラスト主役（背景フル） | 章扉・コンセプト・キーメッセージ・概念図（被写体1点で語れる） | 背景フル＋overlayTextで意味を載せる |
| ハイブリッド（背景地＋HTML前面） | 対比・手順・構造（順序/個数/役割が意味を持つ） | 画像は地・モチーフ、前面にCSSカードを重ねる |

**原則**:
- 数値・逐語が変わりやすいほどHTML主役へ寄せる（退化耐性を優先）。迷ったらHTML主役。
- 人物なし・画像内テキスト非焼き込みを全形式で固定する。
- 意味テキストは overlayText（=逐語）が正本。画像へは焼き込まない。
- 生成器は事前確認済みの text-to-image バックエンドを meta の `source` に記録する。`codex` 単体を画像生成器として扱わない。
- per-slide ブロックは `backgroundSource`（`raster` / `svg` / `none`）を持つ。`svg` はSVG/CSSで背景を描きラスター画像を作らない指定で、`html-composite`（`raster` / `svg`）と `html-primary`（`none` / `svg`）で使う。`image-only` は `none`。
- `pattern` / `textPolicy` / `backgroundSource` の値域定義の正本は `${SRG_ROOT:-$CLAUDE_PLUGIN_ROOT}/references/style-genome-packaging.md` §4。ここでは再定義せず参照する。

## ビジネスルール
- **CONST_001 (1スライド1メッセージ)**: 1枚のスライドに複数の核心メッセージを詰め込まない。
  - 目的: 聴衆の認知負荷を下げ、各スライドの主張を明確にする。
  - 背景: Presentation Zen の1スライド1メッセージ原則。情報過多のスライドは伝達効率が落ちる。
- **CONST_002 (アイコン/図解必須)**: 全スライドにアイコンまたは図解を付与する。
  - 目的: テキストのみの単調なスライドを避け、視覚的理解を補助する。
  - 背景: slide:ology の視覚化原則。アイコンがない箇所は意味の手がかりが弱くなる。
- **CONST_003 (3項目以上は図解)**: 3項目以上の列挙は図解または表形式で表現する。
  - 目的: 長い箇条書きの可読性低下を防ぐ。
  - 背景: 文字列リスト禁止・図解優先の方針。並列情報は構造で見せた方が把握が速い。
- **CONST_004 (プロセスはフロー)**: 手順・ステップ説明にはフローチャートを使用する。
  - 目的: 順序関係を視覚的に明確にする。
  - 背景: 手順をリストで書くと前後関係や分岐が伝わりにくい。
- **CONST_005 (承認必須)**: ユーザー承認なしで次フェーズ（Phase 2.5）に進まない。
  - 目的: 構造段階での手戻り防止と過不足の早期検出。
  - 背景: 構造化データ先行の原則。HTML後の修正は構造修正より高コスト。
- **CONST_006 (質問スライド配置)**: 質問スライドは各セクション内の最後に配置し、セクション→背景情報→質問の順に並べる。背景を先に提示してから質問へ到達させる。
  - 目的: 聴衆が文脈を理解した状態で質問に向き合えるようにする。
  - 背景: 背景なしに質問だけ提示すると、聴衆が考える材料を持てず議論が浅くなる。
- **CONST_007 (コード/数式/精密数値表のHTML確定)**: コード（slide-code / slide-code-compare）・数式・精密数値表は無条件に実HTMLコードブロック側へ確定し、画像化に上書きしない。
  - 目的: 逐語が頻繁に変わる要素の正確性を保ち、画像化による退化を防ぐ。
  - 背景: ビジュアル形式振り分けの退化耐性最優先方針。画像内の文字は修正が効かず誤りが固定化する。
- **CONST_008 (読者中心の入口設計)**: ヒアリングの読者価値ブリーフ（対象範囲・共有課題/願望・視聴後の変化・専門の橋・深さの証拠・正式タイトル制約）を設計入力とする。タイトルスライド・冒頭のキーメッセージ・セクション扉は、想定聴衆の範囲内で共有される課題と「聴衆が得る変化（Before→After・根拠があれば数字）」を先に渡し、専門手段だけを主語にしない・内容理解に不要な属性スタックを置かない・発表者の資格紹介から始めない（入口ホリゾンタル）。本文スライドでは専門の深さ（確認済みの数字・手順・失敗・適用条件・限界）を保ち、各主要セクションに「自分に当てはまる兆候・判断の問い・選択肢・次の行動」のいずれかを置く（中身バーティカル＋自分ごと化）。
  - 目的: 聴衆が最初のスライドで「自分ごと」と感じて文脈へ入れるようにする。聴衆が知りたいのは発表者の知識ではなく自分の変化。
  - 背景: 入口まで専門特化するとタイトルの時点で聴衆を失う。「ターゲットを絞る」ことと「入口まで狭くする」ことは別。ただし社内定例・技術説明・監査/契約/仕様など正式名称や検索性が必要な場合は主タイトルを維持し、subtitle/keyMessage/冒頭スライドで聴衆価値を補う。素材にない数字・実績は作らない。読者中心設計の正本は `${SRG_ROOT:-$CLAUDE_PLUGIN_ROOT}/references/report-narrative-logic.md` §7（slide にも同原則を適用し、違反は deck-evaluator の D5 読者フックが検出する）。

## 5.7 設計仕様

テキストレイアウト指針・SVG設計メモ仕様・シリーズ構成品質ガイドライン・アイコン選定ロジック・アニメーションパターンの設計規範。5.4 実行方式のループ各周回で本節を判断軸として適用し、5.3 完了チェックリストで充足を確認する。

### テキストレイアウト指針

#### 改行位置
- 20文字を超える文章は意味の切れ目で`<br>`を挿入
- 読点「、」の後で区切りが自然な場合は改行
- 強調（`<strong>`）の前後で区切りを検討

#### フォントサイズ
- インラインスタイルでfont-sizeを指定しない
- CSS変数を使用（`var(--fs-body)`, `var(--fs-small)`等）
- 補足テキストは`.text-note`クラスを使用

#### 配置
- タイトル・メッセージ・セクション: 中央揃え
- リスト・テーブル: 左揃え
- フロー: 中央揃え

### SVG設計メモ仕様（必須）
**全SVG図解スライドに「SVG設計メモ」セクションを必ず記載する。**

詳細: `${SRG_ROOT:-$CLAUDE_PLUGIN_ROOT}/references/svg-design-spec.md`

#### 必須記載項目（省略不可）
1. **viewBox**: SVG座標空間（例: `"0 0 860 480"`）
2. **要素配置座標**: 全カード・ノードのx,y座標
3. **カードサイズ**: 幅×高さ×rx
4. **フォントサイズ**: CSS変数名+rem/px値
5. **最大文字数/行**: カード有効幅÷フォントサイズ×0.75で算出
6. **最大行数**: テキスト要素ごと
7. **テキスト改行位置**: 全テキストの改行位置+文字数（例: 「業務の流れ」(5文字)/「を整理する」(5文字)）
8. **padding/gap**: カード内余白・カード間余白
9. **接続線/矢印**: 始点・終点座標
10. **accent-color**: 各要素のアクセントカラー
11. **foreignObjectスタイル方針**: `class="fo-card"`（縦中央）または `class="fo-card fo-card--row"`（横並び）を使用。レイアウトプロパティのインラインstyle単独定義は禁止（GSAPのclearPropsで破壊される）

#### テキスト収まり計算ロジック
```
カード有効幅 = カード幅 - 左右padding - アイコン領域幅
最大文字数/行 = floor(有効幅 / フォントサイズpx) × 0.75
必要行数 = ceil(テキスト文字数 / 最大文字数)
```
- 必要行数 > 最大行数 → テキストをリライトするか、カードサイズを拡大
- **全テキスト要素について文字数検証を実施し、結果を記載する**

#### viewBox算出式
```
viewBox幅 = (カード幅 × N) + (gap × (N-1)) + (左右マージン × 2)
viewBox高さ = ヘッダー高さ + カード高さ + 下部テキスト高さ + 上下マージン
```

### シリーズ構成品質ガイドライン
**研修シリーズ等の複数デッキ構成時は以下を必ず設計する。**

#### メタファー・アナロジー設計
| 項目 | ルール |
|------|--------|
| 導入メタファー | 第1回で導入した比喩は全回で使用・発展させる |
| 追跡表 | structure.mdに「メタファー追跡表」を記載（比喩名、導入回、使用回、発展内容） |
| 一貫性 | 同一概念に異なる比喩を使わない |

**例**: 第1回で「新人社員がAIを使う」比喩を導入 → 第2回「新人が組み合わせを覚える」→ 第3回「新人が量産体制を作る」→ 第4回「新人が品質を高める」

#### 難易度の段階的設計
| 項目 | ルール |
|------|--------|

| WS複雑度 | ワークショップの複雑度が回を追って段階的に上がること |
| 依存関係表 | structure.mdに「前提知識の依存関係」を記載 |

#### ミッション・コンセプト整合

### アイコン選定ロジック
アイコン選定は `${SRG_ROOT:-$CLAUDE_PLUGIN_ROOT}/references/icons.md` を参照。

**選定方法**:
1. コンテンツからキーワードを抽出
2. icons.md でキーワードを検索
3. 最も関連性の高いアイコンを選択
4. 見つからない場合は類似キーワードで再検索

**アクセントカラー割り当て**:
- 主要情報: `accent-blue` (wave-blue)
- 警告・注意: `accent-pink` (sakura-pink)
- 成功・完了: `accent-aqua` (wave-aqua)
- 強調・重要: `accent-yellow` (autumn-yellow)
- 補助: `accent-violet` (spring-violet)
