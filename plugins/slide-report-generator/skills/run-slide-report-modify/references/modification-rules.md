# 既存成果物 部分修正規約（slide-report-modifier 手続き知識 SSOT）

> **正本**: このファイルは slide-report-modifier から抽出した手続き知識/規範の SSOT。run-slide-report-modify の SKILL.md と agent 本体（agents/slide-report-modifier.md）の双方がこれを参照する。規則の上位正本 (SR-ID) は spec-registry.md を辿る。

**責務**: 既存成果物（slide deck / report）の指定箇所部分修正のドメイン定義（用語集・評価基準・修正タイプ分類・制約カタログ CONST_001-012）と部分修正規範（パース/差分適用/非対象箇所保護・修正フローパターン・index.html ⇔ structure.md 同期維持の詳細ルール/コード例）の逐語正本。slide-report-modifier（薄化アダプタ）は役割・起動条件・I/O契約に専念し、詳細規範は本 reference を SSOT とする。

## ビジネスルール

各制約に CONST_NNN を付番し、目的（防ぐ事象）と背景（採用根拠）を併記する。

- **CONST_001 (structure.md 必須)**: 修正フローは structure.md を前提とし、存在しない場合は新規作成フロー（Phase 1-3）へ案内する。
  - 目的: 正本不在のまま HTML だけを直接編集し、状態が復元不能になる事態を防ぐ。
  - 背景: structure.md は本スキルの SSoT（単一の正本）であり、ここから index.html を再生成する駆動設計（v6.0.0）を採用しているため。

- **CONST_002 (index.html ⇔ structure.md 同期必須)**: HTML を修正したら必ず structure.md の該当セクションも更新する。
  - 目的: 両ファイルの乖離による状態不整合（次回修正時の復元不能・履歴不正確）を防ぐ。
  - 背景: 整合が崩れると structure.md から正しい状態を復元できず、構成変更時に意図しない結果になる（Layer 6 同期フローに詳述）。

- **CONST_003 (承認必須)**: 修正案はユーザー承認後にのみ実行する。承認前に index.html / structure.md を書き換えない。
  - 目的: ユーザー意図と異なる修正の既成事実化を防ぐ。
  - 背景: 修正は既存成果物の破壊リスクを伴うため、提示→承認→実行のゲートを設ける。

- **CONST_004 (履歴記録必須)**: 全ての修正を structure.md の修正履歴セクションに記録する。
  - 目的: 変更経緯を追跡可能にし、後続修正や差し戻し時の判断材料を残す。
  - 背景: 履歴が不正確だと修正の積み重ねで意図しない状態に至るため。

- **CONST_005 (原則維持)**: 1スライド1メッセージの原則を崩さない。
  - 目的: 修正に伴う情報過密化を防ぎ、デッキの可読性を保つ。
  - 背景: slide:ology の中核原則であり、structure-designer が確立した品質基準を Phase 4 でも維持するため。

- **CONST_006 (最小変更)**: 要求された修正のみを実施し、過剰な変更を避ける。
  - 目的: 要求外の改変による既存デッキの予期せぬ破壊を防ぐ。
  - 背景: 修正は破壊リスクが高く、変更面積が広いほど検証コストと回帰リスクが増すため。

- **CONST_007 (分離形式維持)**: 修正時も CSS は styles.css、JS は scripts.js へ分離を維持し、index.html へのインライン CSS/JS 追加を禁止する。
  - 目的: html-generator が確立した3ファイル分離構造を崩さず、保守性と再生成可能性を保つ。
  - 背景: インライン化はスタイル/挙動の散逸を招き、後続の検証・再生成を困難にするため。

- **CONST_008 (質問スライド配置)**: 質問スライド（.question-badge）は各セクションの最後（背景情報の後）に配置する。
  - 目的: 背景→質問の順序を守り、質問先行による理解阻害を防ぐ。
  - 背景: structure-designer の構成原則（背景→質問の順）を修正後も維持するため。

- **CONST_009 (質問フォント維持)**: `.question-badge ~ .main-message { font-size: var(--fs-subheading) }` を維持する。
  - 目的: 質問スライドのメッセージフォントサイズの一貫性を保つ。
  - 背景: 既存スタイル契約であり、修正時に上書き・欠落させると視覚的整合が崩れるため。

- **CONST_010 (印刷カード比率維持)**: @media print 修正時、カードの padding・font-size・gap・border-radius は画面 CSS と同比率を維持し、画面の50%以下に縮小しない。
  - 目的: 印刷（A4）時のカードの判読性・レイアウト崩れを防ぐ。
  - 背景: 印刷比率を過度に縮小すると印刷品質が劣化するため、画面との同比率基準を設けている。

- **CONST_011 (AI画像図解の明示指示必須)**: 画像生成・Codex 図解作成の明示指示がない場合は実行しない。通常は SVG/CSS/JS/HTML で改善する。
  - 目的: 意図しない画像化による責務逸脱・コスト発生を防ぐ。
  - 背景: 画像化は明示要求時のみ許可する運用（v8.0.3 / v8.0.7）に準拠するため。

- **CONST_012 (AI画像図解の責務分離)**: 通常は画像内テキストを禁止し、日本語ラベル・見出し・数値は HTML/SVG で重ね、structure.md に画像パス・alt・prompt/meta・差し替え理由を同期する。`kanagawa-comic-diagram` の漫画チック説明図では `image-only + baked-with-overlay` の時だけ短文・簡易表の焼き込みを許可し、`overlayText` を正本として残す。
  - 目的: 文字の判読性・編集可能性・多言語対応を保ちつつ、画像とテキストの正本を一意に保つ。
  - 背景: overlayText を正本とする責務分離（v8.0.3）であり、コード系・精密数値は実 HTML 側へ振り分ける横断原則（v8.0.7）に整合させるため。

## 修正フローパターン

修正タイプ別の部分修正手順（パース→対象特定→差分適用→非対象箇所保護→同期）。5.4 実行方式のループ各周回で本節を判断軸として適用し、5.3 完了チェックリストと同期確認チェックリストで充足を確認する。

### コンテンツ修正（軽微）
```
1. structure.md読み込み
2. 対象スライド特定
3. テキスト/数値を修正
4. HTML該当部分のみ更新
5. structure.md履歴更新
```

### タイプ変更（中程度）
```
1. structure.md読み込み
2. 対象スライド特定
3. 新タイプ用のHTML構造を生成
4. アニメーション定義を更新
5. 全体を再生成
6. structure.md履歴更新
```

### 構成変更（大規模）
```
1. structure.md読み込み
2. 構成変更内容を設計
3. structure-designer Taskで再設計
4. ユーザー承認取得
5. html-generator Taskで全体再生成
6. structure.md全面更新
```

## index.html ⇔ structure.md 整合性維持（重要）

**原則: index.html と structure.md は常に同期を維持すること（CONST_002）。**

両ファイルが整合していないと、以下の問題が発生する：
- 次回の修正時に structure.md から正しい状態を復元できない
- 修正履歴が不正確になる
- 構成変更時に意図しない結果になる

### 同期フロー
```
【HTMLを修正した場合】
index.html 修正 → structure.md 該当セクション更新 → 修正履歴追記

【構成案を変更した場合】
structure.md 修正 → ユーザー承認 → index.html 再生成
```

### structure.md に反映すべき項目
| HTMLの変更内容 | structure.md の更新箇所 |
|---------------|---------------------|
| テキスト変更 | 該当スライドの「コンテンツ詳細」 |
| スライドタイプ変更 | 該当スライドの「タイプ」 |
| アイコン変更 | 該当スライドの「アイコン」 |
| CSSスタイル調整 | 該当スライドの「スタイル調整メモ」 |
| スライド追加/削除 | 「スライド一覧」全体 |
| 16:9レイアウト調整 | 該当スライドの「レイアウト調整メモ」 |

> **同期確認チェックリスト（完了ゲート＝停止条件）は agent 本体（agents/slide-report-modifier.md）の Layer 6 が保持する**（4項目: スライド数一致・タイトル/メッセージ一致・タイプ一致・履歴記録）。上記の同期フローと反映項目表を適用した結果を、当該ゲートで検証する。
