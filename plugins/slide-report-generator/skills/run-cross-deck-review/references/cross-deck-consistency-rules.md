# シリーズ横断整合性 検証規約（cross-deck-reviewer 手続き知識 SSOT）

> **正本**: このファイルは cross-deck-reviewer から抽出した手続き知識/規範の SSOT。run-cross-deck-review の SKILL.md と agent 本体（agents/cross-deck-reviewer.md）の双方がこれを参照する。規則の上位正本 (SR-ID) は spec-registry.md を辿る。

**責務**: 複数デッキをシリーズ横断で検証する際のドメイン定義（用語集・評価基準・ビジネスルール CONST_001-005）と評価軸（検証項目 C1〜C15・4条件判定マトリクス）・3レンズ分析（Agent A/B/C・単一 fork context 内多角分析）プロンプトテンプレート・修正の優先度分類（P0/P1/P2）の逐語正本。cross-deck-reviewer（薄化アダプタ）は役割・起動条件・I/O契約に専念し、詳細規範は本 reference を SSOT とする。cross-deck-reviewer は read-only 検出・分類専任であり、修正の適用（ファイル書き換え）は run-slide-report-modify（slide-modifier）へ委譲する。

## 評価基準（ドメイン固有の判定基準）
| 基準 | 条件 |
|------|------|
| 機械チェック完了 | `cross-deck-consistency.js` が終了コードと差分一覧を返し、C1〜C2・C11〜C13・C15 の機械検出結果が揃う（C3は Agent A が判定） |
| 3レンズ分析完了 | Agent A/B/C の3レンズが「問題リスト（重要度 高/中/低）＋修正提案」を返し、C3〜C15 が全件カバー（単一 context 内で完結） |
| 4条件判定 | 各条件に PASS/WARN/FAIL が付与され、根拠となるC番号が明示されている |
| 修正分類完了 | 全 FAIL/WARN が P0/P1/P2 に分類され、対象デッキ・対象ファイル・修正内容が付与されている（適用は run-slide-report-modify へ委譲） |
| シリーズ成立条件 | P3.5 通過済みデッキが2つ以上（デッキ2未満ならP5スキップ） |

## ビジネスルール
- **CONST_001 (機械チェック先行)**: 構造的整合性（C1-C2・C11-C13・C15）は `cross-deck-consistency.js` の実行結果を一次根拠とし、C3 は Agent A が structure.md を目視比較して判定する。
  - 目的: 共通仕様差分・rem残存・外部URL混入を、チェッカーが実際に実装する検出方式で再現可能に検出する。
  - 背景: 複数デッキの共通仕様差分は人間目視では取りこぼしやすく、再現性ある検出にはスクリプトが必要。
- **CONST_002 (3レンズ多角分析の固定)**: 横断検証は必ず Agent A（論理・構造）/ B（メタ・発想）/ C（システム・戦略）の3レンズに分割し、単一 fork context 内で3レンズを適用する（再 fork＝SubAgent 起動はしない）。観点を1レンズへ集約しない。
  - 目的: 30種思考法を分散させ、特定観点の検証漏れを防ぐ。
  - 背景: 単一視点では論理・コンテンツ・技術のいずれかが手薄になるため、観点を構造的に分担する。cross-deck-reviewer は Read / Bash のみを持つため、3レンズは同一 context 内の分析フレームとして適用する。
- **CONST_003 (P1のユーザー承認必須)**: P1（要判断）の修正提案は下流での適用前にユーザー確認を必須とする。P0（即時修正候補）と P1 の区別は残すが、cross-deck-reviewer は適用（ファイル書き換え）を一切行わず、分類済み提案を run-slide-report-modify へ委譲する。
  - 目的: コンテンツ追加・スライド追加など編集判断を伴う変更を勝手に適用しない。
  - 背景: 比喩追加やスライド追加は意図に依存するため、機械的差分修正（P0候補）と区別する必要がある。
- **CONST_004 (read_only原則・修正の委譲)**: cross-deck-reviewer は検出・分類専任で、いかなるファイル（structure.md / ソースmd / index.html / styles.css / scripts.js）も直接書き換えない。P0/P1/P2 いずれの修正も、適用は後続の run-slide-report-modify（slide-modifier）へ委譲する。
  - 目的: 横断検証が誤って生成物を破壊するリスクを避け、修正経路を run-slide-report-modify に一本化する。
  - 背景: P5は検証中心の工程であり、デッキ本体・structure.md・ソースmd の修正は専任エージェント（run-slide-report-modify）の責務だから。
- **CONST_005 (絵文字ゼロ)**: 修正・提案・レポート出力に絵文字を使わない。アイコンが要る場合はFont Awesomeの `fa-*` 名で表現する。
  - 目的: スライド品質規約（絵文字禁止・fa-*統一）をシリーズ全体で維持する。
  - 背景: スキル全体で絵文字禁止・Font Awesome統一がルールとして定められている。

## 評価軸: 検証項目（C1〜C15）

> C1〜C15 は 4条件判定（§評価軸: 判定マトリクス）と完了チェックリスト（agent 5.3）が参照する評価軸の定義である。機械的チェック（`cross-deck-consistency.js`）は C1-C2・C11-C13・C15 を一次根拠とし、Agent A/B/C の3レンズ分析（§Agent A/B/C 3レンズ分析テンプレート・単一 context 内多角分析）は C3-C5 / C6-C10 / C11-C15 を分担検証する。統合判定では両者を突き合わせ、各判定に根拠C番号を明示する。

### カテゴリ1: 構造的整合性（機械的チェック + Agent A）
| # | 検証項目 | 基準（合否判定可能な客観条件） | 検出方法 |
|---|---------|------|----------|
| C1 | shared-spec完全一致 | 全structure.mdの共通仕様セクション（A4印刷/コードブロック/GSAP/フォント）が機械一致（SVG設計/スライドタイプ定義は Agent A 目視） | `cross-deck-consistency.js --check shared-spec` |
| C2 | rem残存なし | 全 styles.css で数値付き rem 単位が 0 件（unit-system.md §3 に従い vw へ移行済み） | `cross-deck-consistency.js --check rem-units` |
| C3 | スライドタイプ定義一致 | CSSクラス対応表がシリーズ全体で統一 | Agent A: 各 structure.md のスライドタイプ定義/CSSクラス対応表を横断比較 |
| C4 | セクション構成の段階的積み上げ | 第N回の前提が第N-1回までに提示済み | Agent A: 各回のスライド内容を読み依存関係を検証 |
| C5 | 用語の統一 | 同一概念に異なる用語を使っていない | Agent A: 全デッキのテキストを横断検索 |

### カテゴリ2: コンテンツの一貫性（Agent B）
| # | 検証項目 | 基準（合否判定可能な客観条件） | 検出方法 |
|---|---------|------|----------|
| C6 | メタファーの継続性 | 第1回で導入した比喩が後続回でも使われている | ソースmd/structure.mdの比喩表現を追跡 |
| C7 | コンセプト文の一貫性 | 全体概要のコンセプトが各回スライドで正確に引用されている | ソースmd vs スライドテキストの照合 |
| C8 | デモの連続性 | 共通デモ（例: 営業日報）が各回で進化している | デモスライドの内容を回別に比較 |
| C9 | 難易度の段階性 | 第N回のWS/プロンプト複雑度が第N-1回以下に逆行せず、未提示の前提知識を要求しない | WS内容・プロンプト複雑度を回別に比較 |
| C10 | 受講者体験フロー | 各回のタイムラインに前提知識ギャップ（未提示概念の使用）と重複説明がない | 3時間×4回のタイムライン全体を俯瞰 |

| C12 | GSAPパターンの一貫性 | アニメーション設定が全デッキで統一 | scripts.js間のGSAP設定比較 |
| C13 | 印刷品質の統一 | A4印刷時のレイアウトが全デッキで同じ品質 | @media print CSSの比較 |

## 評価軸: 判定マトリクス（4条件判定基準）

### 矛盾なし (Contradiction-Free)
| 判定 | 基準 |
|------|------|
| PASS | C1-C5が全てクリア |
| WARN | C4またはC5に軽微な不一致 |
| FAIL | C1-C3のいずれかに差分がある |

### 漏れなし (Completeness)
| 判定 | 基準 |
|------|------|
| PASS | C6-C10が全てクリア |
| WARN | C6-C8のいずれかに部分的な欠落 |
| FAIL | C9（難易度）またはC10（体験フロー）に重大な欠落 |

### 整合性あり (Consistency)
| 判定 | 基準 |
|------|------|
| PASS | C11-C14が全てクリア |
| WARN | C11-C12に軽微な差異 |
| FAIL | C13（印刷）またはC14（a11y）に不一致 |

### 依存関係整合 (Dependency Alignment)
| 判定 | 基準 |
|------|------|
| PASS | C4が全てクリアかつC15に問題なし |
| WARN | C4に前提の暗黙的な依存がある |
| FAIL | C15に外部参照混入がある |

## Agent A/B/C 3レンズ分析テンプレート（単一 context 内多角分析）

cross-deck-reviewer が単一 fork context 内で Agent A/B/C の3レンズとして順に適用するプロンプトテンプレート（CONST_002 で3レンズ分割を固定・再 fork＝SubAgent 起動はしない）。各レンズは Read / Bash / Grep で対象を横断分析する分析フレームであり、別 context の worker を起動するものではない。

### Agent A: 論理・構造分析
```
以下のスライドシリーズを論理・構造の観点で横断検証してください。

検証対象: {series-dir} 内の全structure.md + ソースmd
検証項目: C3（スライドタイプ定義一致）、C4（段階的積み上げ）、C5（用語統一）

分析手法:
1. 演繹法: 全体概要の方針→各回の実装の整合性
2. 帰納法: 各回のパターンから全体の法則性を抽出
3. MECE: 4回分の内容が漏れなく重複なく構成されているか
4. 因果関係: 第N回の学びが第N+1回で活かされる構造か

出力形式:
- 問題リスト（重要度: 高/中/低）
- 各問題の修正提案
```

### Agent B: メタ・発想分析
```
以下のスライドシリーズをコンテンツ品質の観点で横断検証してください。

検証対象: {series-dir} 内の全structure.md + ソースmd + 全体概要md
検証項目: C6-C10

分析手法:
1. メタ認知: 受講者がどう受け取るかを俯瞰
2. アナロジー: 比喩・メタファーの一貫性と効果
3. リフレーミング: 各回の位置付けが全体像の中で明確か
4. ストーリーテリング: 4回通した学習体験にストーリーがあるか

出力形式:
- 問題リスト（重要度: 高/中/低）
- コンテンツ改善提案
```

### Agent C: システム・戦略分析
```
以下のスライドシリーズを技術的品質の観点で横断検証してください。

検証対象: {series-dir} 内の全structure.md + styles.css + scripts.js + index.html
検証項目: C11-C15

分析手法:
1. システム思考: 全デッキが一つのシステムとして機能するか
2. ボトルネック分析: 品質のボトルネックとなる箇所
3. 制約理論: 16:9/A4印刷/フォント制約の整合性
4. リスク分析: 外部参照・依存の健全性

出力形式:
- 問題リスト（重要度: 高/中/低）
- 技術的修正リスト（P0候補: 機械的差分 / P1候補: 手動判断要。適用は run-slide-report-modify へ委譲）
```

## 修正の優先度分類
> cross-deck-reviewer は read-only ゆえ、いずれの優先度でも自らファイルを書き換えず、分類済み提案を run-slide-report-modify へ委譲する。「対応」列は下流での適用方針を示す。

| 優先度 | 分類 | 対応（適用は run-slide-report-modify） |
|--------|------|------|
| P0: 即時修正 | shared-spec差分、外部URL混入、CSS変数/GSAP/印刷の機械的差分 | run-slide-report-modify で structure.md/ソースmd を修正（本 skill は検出・分類・提案のみ） |
| P1: 要判断 | コンテンツ改善（比喩追加、スライド追加等） | ユーザー承認後に run-slide-report-modify で適用（修正提案を提示） |
| P2: 将来改善 | 体験フローの最適化、難易度曲線の調整 | バックログとして記録 |
