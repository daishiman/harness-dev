# レイアウト・ビジュアルガイドライン

**責務**: レイアウト法則・分割パターン・図形・矢印・余白・統一感

---

## 1. 読み順の原則

### 基本ルール

スライドは**左上から右下**の順に読まれる。

```
┌─────────────────────────────────────┐
│  1 ───────────→ 2                   │
│  │                │                 │
│  ↓                ↓                 │
│  3 ───────────→ 4                   │
└─────────────────────────────────────┘
```

### 重要な情報の配置

| 位置 | 優先度 | 配置する内容 |
|------|--------|-------------|
| 左上 | 最高 | 最重要情報・タイトル |
| 右上 | 高 | 結論・数字 |
| 左下 | 中 | 補足情報・根拠 |
| 右下 | 低 | 注釈・出典 |

## 2. 分割パターン

### 2分割（比較・対比）

```
┌─────────────┬─────────────┐
│             │             │
│    左側     │    右側     │
│   (Before)  │   (After)   │
│             │             │
└─────────────┴─────────────┘
```

**用途**: 比較、Before/After、対立する概念

### 4分割（複数要素）

```
┌─────────────┬─────────────┐
│      1      │      2      │
├─────────────┼─────────────┤
│      3      │      4      │
└─────────────┴─────────────┘
```

**用途**: 4つの要素、2x2マトリックス、4象限

### 上下分割

```
┌─────────────────────────────┐
│           上部              │
│      （主要コンテンツ）     │
├─────────────────────────────┤
│           下部              │
│       （補足・注釈）        │
└─────────────────────────────┘
```

**用途**: メイン情報と補足の分離

## 3. 矢印の使い分け

### 矢印タイプ

| 矢印 | 名称 | 用途 | 形状例 |
|------|------|------|--------|
| 直線矢印 | ストレート | 因果関係・単純な流れ | → |
| 鍵線矢印 | エルボー | 要素間の接続 | └→ ┌→ |
| 三角矢印 | ブロック | 全体の流れ・強調 | ▶ ▷ |
| 双方向矢印 | ダブル | 相互関係 | ↔ |
| 曲線矢印 | カーブ | 循環・回転 | ↻ |

### 使い分けガイド

```
【因果関係】
原因 ────→ 結果

【プロセス】
STEP1 ──▶ STEP2 ──▶ STEP3

【要素間の接続】
┌── 要素A
│
└→ 要素B

【全体の流れ（三角矢印）】
背景 ▷ 課題 ▷ 解決策 ▷ 効果
```

## 4. 図形の使い分け

### 図形と表現対象

| 図形 | 表現対象 | 例 |
|------|---------|-----|
| 四角形 | 具体的な要素 | 組織、システム、製品、ステップ |
| 楕円 | 抽象的な概念 | ビジョン、価値観、目標、開始/終了 |
| 角丸四角 | 中間的な要素 | プロセス、ステップ、ボタン |
| 三角形 | 階層構造 | ピラミッド、優先度、方向性 |
| ひし形 | 判断・分岐 | 条件分岐、Yes/No |
| 六角形 | 連携・結合 | ハブ、中心要素 |

### 使い分けルール

```
【具体 → 四角形】
┌─────────────┐
│  営業部門   │
└─────────────┘

【抽象 → 楕円】
    ╭─────────╮
    │ ビジョン │
    ╰─────────╯

【プロセス → 角丸四角】
╭─────────────╮
│  STEP 1    │
╰─────────────╯

【判断 → ひし形】
    ◇
   /  \
  Yes  No
```

## 5. 余白ルール

### 図形の余白最小化

```
【NG】余白が大きすぎる
┌───────────────────────────┐
│                           │
│      テキスト             │
│                           │
└───────────────────────────┘

【OK】余白を最小化
┌───────────────┐
│  テキスト     │
└───────────────┘
```

### 余白の目安

| 要素 | 内側余白 | 外側余白 |
|------|---------|---------|
| 図形内テキスト | 8-16px | - |
| 図形間 | - | 16-24px |
| スライド端 | - | 32-48px |

## 6. 統一感の作り方

### 色の統一

| 用途 | カラー | 統一ルール |
|------|--------|-----------|
| メインカラー | `--wave-blue` | 重要な要素・タイトル |
| アクセント1 | `--sakura-pink` | 注意喚起・課題 |
| アクセント2 | `--wave-aqua` | 解決策・ポジティブ |
| アクセント3 | `--autumn-yellow` | 強調・数字 |
| 背景 | `--bg-dim` | カード・ボックス |

### 順番の統一

スライド間で同じカテゴリの要素は同じ順番で表示：

```
【スライドA】
1. コスト削減
2. 生産性向上
3. 品質改善

【スライドB】（同じ順番を維持）
1. コスト削減の詳細
2. 生産性向上の詳細
3. 品質改善の詳細
```

### ページ番号

すべてのスライドにページ番号を記入（表紙を除く）：

```
┌─────────────────────────────┐
│                             │
│        コンテンツ           │
│                             │
├─────────────────────────────┤
│                        2/10 │
└─────────────────────────────┘
```

## 10. カードレイアウト最適化

### タイトル改行防止

カード内のタイトル（h4）は**一行で表示**されるべき。

```css
.list-item h4, .flow-step h4 { white-space: nowrap; }
```

### カード幅とフォントサイズのバランス

| 項目 | 推奨値 | 注意点 |
|------|--------|--------|
| カード最小幅 | 280px (list), 180px (flow) | タイトルが一行で収まる幅 |
| カード最大幅 | 380px (list), 220px (flow) | 横並び時のバランス |
| タイトルフォント | 1.2-1.4rem × font-scale | 大きすぎると改行発生 |

## 11. 計算式ベースレイアウト

### 核心原則

カード・図解のサイズは**文字数×文字サイズ×余白**の計算式で決定する。

```
カード必要幅 = (最長タイトル文字数 × 平均文字幅係数 × フォントサイズ) + (左右パディング × 2)
```

### 計算パラメータ

| パラメータ | 値 | 説明 |
|-----------|-----|------|
| 全角文字幅係数 | 0.9 | フォントサイズの90% |
| 半角文字幅係数 | 0.5 | フォントサイズの50% |
| 左右パディング | 1.2rem | カード内の余白 |
| font-scale | 1.3 | CSS変数で定義 |

### 画面表示用計算例（14文字タイトル、3カラム）

```
コンテナ幅: 1400px
カード幅: (1400 - gap×2) / 3 ≈ 450px
テキスト領域: 450 - padding×2 ≈ 412px
フォントサイズ: 412 / (14 × 0.9) ≈ 33px → 1.5rem × scale

→ min-width: 380px, max-width: 450px
→ font-size: calc(1.5rem * var(--font-scale))
```

### 印刷用計算例（A4横向き）

```
【リストスライド（3カラム）】
A4横: 265mm利用可能, gap 3mm×2 = 6mm
カード幅: (265 - 6) / 3 ≈ 86mm
→ min-width: 80mm, max-width: 86mm, font-size: 12pt

【フロースライド（4カラム+矢印）】
矢印幅: 6mm×3 = 18mm, gap 2mm×6 = 12mm
カード幅: (265 - 18 - 12) / 4 ≈ 59mm
→ min-width: 55mm, max-width: 60mm, font-size: 11pt
```

### 逆算アプローチ

```javascript
function calculateFontSize(charCount, availableWidth, padding) {
  const textAreaWidth = availableWidth - (padding * 2);
  const requiredFontPx = textAreaWidth / (charCount * 0.9);
  return Math.max(requiredFontPx / 16 / 1.3, 1.1);
}
```

### 適用ルール

1. **同一スライド内統一**: 最長タイトルを基準に全カードを統一
2. **white-space: nowrap**: タイトルは改行禁止
3. **min-width/max-width**: 計算結果に基づく範囲指定
4. **印刷CSS同期**: 画面用と印刷用で同等のバランスを維持

## 12. 意図的改行による可読性向上

### 問題: 自動改行の読みにくさ

カード内の説明文がカード幅に合わせて自動改行されると、**意味の途中で切れて読みにくくなる**。

```
【NG: 自動改行】
講義ではなく、実際にAIを使いながらプ
ロンプト作成を体験

「プロンプト」が途中で切れて読みにくい
```

### 解決: 意図的な`<br>`タグ

意味のまとまりで**明示的に改行**することで可読性を大幅に向上。

```html
<!-- OK: 意図的改行 -->
<span>講義ではなく、<br>実際にAIを使いながら<br>プロンプト作成を体験</span>
```

```
【OK: 意図的改行】
講義ではなく、
実際にAIを使いながら
プロンプト作成を体験

意味のまとまりで切れており読みやすい
```

### 改行位置の原則

| 優先順位 | 改行位置 | 例 |
|---------|---------|-----|
| 1 | 句読点の後 | 「〜ではなく、」の後 |
| 2 | 助詞「を」「に」「で」の後 | 「AIを使いながら」の後 |
| 3 | 動詞の連用形の後 | 「作成を」の後 |
| 4 | 15-20文字ごと | 長い文の場合 |

### 適用対象

- `.list-item span`: リストカードの説明文
- `.flow-step span`: フローステップの説明文
- `.compare-item li`: 比較リストの項目
- `.grid-card p`: グリッドカードの説明文

### CSS設定

```css
/* 説明文は white-space: normal を維持 */
.list-item span,
.flow-step span {
  white-space: normal;
  line-height: 1.5;  /* 改行時の行間を確保 */
}
```

### スライドタイプ別の適用例

#### リストスライド（.list-item span）

```html
<!-- 3つの特徴カード -->
<span>講義ではなく、<br>実際にAIを使いながら<br>プロンプト作成を体験</span>
<span>一つ一つ作るのではなく、<br>量産できる仕組みを構築</span>
<span>汎用的な例ではなく、<br>各自の実務で使える<br>プロンプトを作成</span>
```

#### フロースライド（.flow-step span）

```html
<!-- 研修の流れ 4ステップ -->
<span>AI基礎と<br>初めてのプロンプト</span>
<span>プロンプト設計<br>パターンを学ぶ</span>
<span>量産の仕組みを<br>構築</span>
<span>実務適用と<br>振り返り</span>
```

#### 図解スライド（.diagram-text li）

```html
<!-- 研修で身につく4つの基礎力 -->
<li>AIと効果的に<br>対話する方法</li>
<li>目的に合わせた<br>プロンプト構造</li>
<li>出力を見て<br>改善するサイクル</li>
<li>自分の業務を<br>言語化する力</li>
```

### AutoLineBreaker（フォールバック用・デフォルトOFF）

v4.7.0で追加された`AutoLineBreaker`クラスは**デフォルトOFF**。
機械的なパターンマッチングでは文脈を正しく判断できないため、**LLMによる意図的改行を優先**する設計。

**推奨ワークフロー:**
1. **LLMがHTML生成時に`<br>`を挿入**（html-generator.md参照）
2. AutoLineBreakerはフォールバック用として待機

**有効化する場合（特殊ケースのみ）:**
```javascript
// DOMContentLoaded内でコメントを外す
new AutoLineBreaker({ maxChars: 35, minChars: 15 }).init();
```

**機械的改行の限界:**
- 文脈・意味を考慮できない
- 固有名詞の途中で切れる可能性
- 専門用語が不自然に分割される

**詳細**: `agents/html-generator.md`「意図的改行の挿入（LLM責務）」セクション参照
