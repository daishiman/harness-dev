# report content-regime — 書式規律（read-through）

> 責務: output_mode=report の本文書式規律（content-regime）を定義する。slide の「1メッセージ / chip 強制 / 長文禁止」を report では**緩和**し、通読される読み物として文章多めを許容する。slide 規律との差分を明示するのが本書の主眼。
> 正本の役割分担: slide 側の書式規律は移植元 `references/slide-text-guidelines.md` / `writing-rules.md` / `bp-classification.md` が持つ。本書は report モード固有の緩和と維持ラインを定める。骨格は [report-types.md](report-types.md)、ビジュアルは [report-visual-strategy.md](report-visual-strategy.md)。

---

## 0. 前提: なぜ緩和するのか

slide は「投影され、一瞬で読み取られる」ことを前提に、1枚1メッセージ・chip 化・長文禁止という厳しい規律を敷く。report は「手元で腰を据えて読まれる」ことを前提とする。したがって slide の規律をそのまま適用すると、report は「見出しと chip の羅列」になり、読み物としての価値（文脈・論拠・ニュアンス）を失う。**report では文章多めが正**である。

ただし、緩和するのは**コンテンツ意図層**（1メッセージ性・文量・chip 強制）だけであり、**意匠/技術層**（配色・フォント・最小サイズ・印刷）は slide と共有 SSOT のまま維持する（build-contract §D）。

## 2. report の本文書式（推奨）

### 2.1 段落
- **1段落=1論点**。段落の先頭に結論文（トピックセンテンス）を置き、続く文で根拠・補足を述べる。
- 段落は自然な長さでよい（slide の 20 字ルールは適用しない）。ただし 1 段落が長くなりすぎたら論点で分割する。
- 抽象論だけで終わらせず、具体（数値・事例・手順）で裏づける。

### 2.2 markdown 要素
| 用途 | 記法 | 備考 |
|------|------|------|
| 強調 | `**…**` | 要点・キーワードの強調に限る（乱用しない） |
| 並列 | `- ` 箇条書き / `1. ` 番号リスト | 手順は番号、並列は箇条書き |
| 精密な数値・料金・対照 | markdown 表 | 逐語が変わる値は本文の表で持つ（画像に焼かない） |
| コード | インラインコード / ```コードブロック``` | tech-doc の手順で多用 |
| 参照 | `[ラベル](URL)` | 外部仕様・出典へのリンク |
| 注記・警告 | section.callouts（note/warning/tip/caution） | 落とし穴・補足は callouts で目立たせる |

### 2.3 長さ方針（meta.length）
| length | 目安 | 段落密度 |
|--------|------|----------|
| `brief` | 要点先行の短報 | 各節 1〜2 段落。結論中心 |
| `standard` | 標準 | 各節 2〜4 段落。結論＋根拠 |
| `deep` | 精読向け | 各節 3 段落以上。背景・根拠・含意まで |

## 3. 維持ライン（緩和してはいけないもの）

- **最小フォント 1.4rem**: 本文の可読性下限。report でも割らない。
- **Kanagawa 配色・フォント・印刷 CSS**: 意匠 SSOT を共有し report 独自に発明しない。
- **退化耐性（CONST_007 相当）**: コード・数式・精密数値・料金は画像へ焼き込まず、本文（表・コードブロック）で正確に持つ。生成画像に逐語を載せない。
- **1項目1ビジュアル**: 図解は要る節に 1 点。図解過多で本文が痩せない。
- **構造同期**: 本文は承認済み構造（report-structure.json）の忠実な射影。勝手に節を増減しない。

## 4. アンチパターン（report で避ける）

| アンチパターン | なぜ悪いか | 正しくは |
|----------------|-----------|----------|
| 見出しと chip だけで本文がない | 読み物として成立しない（空節） | 各節に結論文＋根拠段落を書く |
| slide をそのまま縦に並べる | 1メッセージ圧縮が通読を阻害 | セクション＋段落へ再構成する |
| 全節に図解を付ける | 図解過多で可読性低下 | 要る節に 1 点、他は文章 |
| 数値・料金を画像に焼く | 更新時に誤りが固定化 | markdown 表で本文に持つ |
| 長文を `<br>` で無理に折る | 読み物では不自然 | 段落として自然に書く |
| 専門手段だけ・不要な属性限定だけのタイトル | 入口が専門側に狭く、読者が「自分ごと」と感じられない | 想定読者の共有課題と変化を先に渡す。数字は根拠がある場合のみ（§7） |
| 書き手の資格紹介から書き出す | 読者が知りたいのは書き手の知識でなく自分の変化 | 第1段落は読者の状況・痛みから始める（§7） |

## 5. チェックリスト

| 項目 | 基準 |
|------|------|
| 読み物として成立しているか | 各節に空でない段落、見出しだけの空節ゼロ |
| 1段落1論点か | 段落先頭に結論文、論点混在なし |
| 緩和を活かしているか | chip 強制で痩せず、文章多めで書けている |
| 維持ラインを守っているか | 最小1.4rem・Kanagawa 意匠・退化耐性・1項目1ビジュアル |
| 逐語要素を本文で持つか | コード・数値・料金が画像でなく本文（表/コード）にある |
| 構造同期しているか | 本文が report-structure.json の射影で過不足ゼロ |

## 6. 1.2.0 追補 — read-through 執筆規律

### 6.1 多様性 < 適合性（block は内容に従わせる）

構造化ブロックは「種類を増やす」ためでなく「内容に合わせる」ために使う。

- **適合**: 3項以上の対照→表、手順→番号リスト、用語対→定義リスト、出典→脚注、次アクション→タスクリスト。内容が要求した器だけを使う。
- **禁止（水増し）**: 多様性を演出するための不要な table/stat-tile。適合しない構造化はノイズで減点。
- **床（最低ライン）**: 全ブロックが `paragraph` だけの節は羅列退化。ここだけを叩く（多様性それ自体は加点しない）。評価規律の詳細は [report-narrative-logic.md](report-narrative-logic.md) §6.4。

### 6.2 inline highlight は2チャネル（色覚非依存）

`==要点==` は色マーカー単独でなく**色 + 非色（font-weight:700 + underline）を併存**させる。色覚特性で色が判別できない読者にも要点が届く。render-report.js の `mark.report-hl` が両チャネルを持つ（新規配色は足さず意匠 SSOT を流用）。

### 6.3 highlight 密度予算（per-section + doc-level）

強調は「絞るほど効く」。二段の上限を守る。

| レベル | 上限 | 根拠 |
|---|---|---|
| 段落 | `==…==` は **1段落1箇所** | §2.2 / narrative-logic §3 |
| 節 | `key-point` は **1節0〜1個** | narrative-logic §3 |
| **文書** | `==…==` の**総数 24 箇所まで** | C25 `doc_highlight_budget=24`（超過で warn） |

per-section cap を各節で守っても、節数が多いと文書全体で強調が希釈する。doc-level 予算で「文書としての要点」を絞る。

### 6.4 新 block 型の content-regime（いつどれを使うか）

1.2.0 で `definition-list` / `footnote` / `task-list` を追加。使いどころ:

| type | いつ使うか | 主フィールド | 主な reportType |
|---|---|---|---|
| `definition-list` | **用語↔定義の対**が3組以上並ぶ（本文に「A は… B は…」と流さない） | `terms[{term,definition}]` | tech-doc（用語集）/ learning（概念定義） |
| `footnote` | **主張の出典・根拠**を本文から分離し可検証にする | `footnotes[{id?,marker?,text,citation?}]` | internal-analysis（根拠）/ client-proposal（実績出典） |
| `task-list` | **意思決定・次アクション**を担当/完了付きで第一級表現 | `tasks[{text,done?,owner?}]` | internal-analysis（次アクション）/ client-proposal（導入 TODO） |

> 順序が本質なら `ordered-list`、チェック管理なら `task-list`。列挙が本質なら `bullet-list`、用語定義なら `definition-list`。器は内容の性質で選ぶ（§6.1）。

**footnote のインライン係り先**: 特定の主張に脚注を係らせるときは、脚注 entry に `id` を付し、本文 text 中に `[^id]` を書く。render-report.js が文書順で連番採番し、本文の `[^id]` を上付き番号リンク（→ 脚注）に、脚注実体を係り先アンカー（本文へ戻る ↩ 付き）にする双方向リンクを生成する。`id` を付けない footnote は本文非係りの巻末注（従来どおり）。`[^id]` に対応する `id` 付き footnote が同一文書内に無いと C25 が `footnote-ref` dangling warn を出す（id typo 検出）。

## 7. 1.3.0 追補 — 入口の執筆規律（タイトル・リード・自分ごと化）

正本は [report-narrative-logic.md](report-narrative-logic.md) §7（入口ホリゾンタル・中身バーティカル）。執筆時は次を守る。

- **タイトル・冒頭要約は読者の変化を先に渡す**: 「◯◯（専門手段）を活用した△△」だけでなく「（Before の痛み）を（After の状態）にした方法」へ翻訳する。数字は入力素材・出典に根拠があるときだけ使う。内容理解に不要な読者属性の AND 限定（年代×地域×職種 …）をタイトルに積まない。
- **書き出しは読者の状況から**: 第1段落は読者が今いる場所（痛み・願望）から始める。書き手の資格・経歴の自己紹介から始めない（信頼材料は根拠・出典の位置へ後置する）。
- **専門用語は入口で翻訳**: 導入部では専門トピックを読者の目的語へ言い換える（「AIを学ぶ」→「仕事を早く終わらせる」）。本論に入ってから専門の解像度を上げる。
- **自分へ移す橋を置く**: 各主要 part / 節に「自分に当てはまる兆候」「判断する問い」「選択肢」「次に試す行動」のいずれかを置き、読者が内容を自分の状況へ移せるようにする。
- **深さは薄めない**: 入口を広げても、本論の確認済みの数字・手順・失敗・再現条件・限界（§6.4 の block が器）は削らない。「広い入口 × 浅い中身」は誰でも書ける一般論への退化。
- **精密タイトルの例外**: tech-doc、社内定型報告、監査・契約・仕様文書など、正式名称・検索語・適用範囲が主タイトルに必要な場合は保持する。subtitle / keyMessage / summary の先頭で読者の課題と変化を補い、検索性と自分ごと化を両立する。
