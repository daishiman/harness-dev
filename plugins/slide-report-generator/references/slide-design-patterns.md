# スライドデザインパターン集

**責務**: 「デザインセンスが良い」スライドを実現するための具体的なビジュアルパターンと判断基準。Apple Keynote水準のデザインTips

---

## 1. Apple Keynoteに学ぶ5つのデザイン原則

### 1.1 Less is More（削ぎ落とし）

| 原則 | 具体 |
|------|------|
| 1スライド1メッセージ | 伝えたいことを1文で言えるか？ |
| 余白は味方 | スライドの50%以上が余白でもOK |
| 要素数上限 | 1スライド最大6要素（タイトル含む） |
| テキスト上限 | 80文字以下（タイトル除く） |

**実践**: 「このスライドから1つ要素を削れないか？」を必ず問う。

### 1.2 Visual Weight（視覚的重み）

全要素に「重み」がある。重い要素は視線を集める。

| 重い | 軽い |
|------|------|
| 大きいフォント | 小さいフォント |
| 太字 | 細字 |
| ビビッドカラー | ミュートカラー |
| 画像・アイコン | テキスト |
| 影付き要素 | フラット要素 |
| 密集エリア | 余白エリア |

**実践**: 最も「重い」要素が最も伝えたいメッセージか確認する。

### 1.3 Progressive Disclosure（段階開示）

1度に全情報を見せない。聴衆の理解に合わせて順に開示する。

```javascript
// 良い例: stagger で順に表示
gsap.from('.item', {
  opacity: 0, y: 15,
  stagger: 0.05,
  ease: 'power2.out'
});

// 悪い例: 全要素同時表示
// → 情報過多、何を見ればいいかわからない
```

### 1.4 Breathing Room（呼吸する余白）

テキストや要素の周囲に「呼吸する」空間を確保する。

| 要素 | 最低余白 |
|------|---------|
| タイトル下 | --space-6 (32px) |
| カード内padding | --space-4 (16px) |
| ヒーロー数値の周囲 | --space-8 (64px) |
| スライド端からコンテンツ | 3%（.slider__item padding） |

### 1.5 Intentional Asymmetry（意図的な非対称）

完全対称は退屈。意図的に非対称を作ると洗練される。

| 対称（退屈） | 非対称（洗練） |
|-------------|--------------|
| テキスト中央、全要素中央 | 左寄せテキスト + 右にアクセント |
| 4等分グリッド | 1:2比率のレイアウト |
| 全カード同色 | 1枚だけアクセント色 |

## 2. スライドパターンカタログ

### 2.1 ヒーロー数値パターン

**用途**: 印象的な数値を強調したいとき

```
┌─────────────────────────────────────┐
│                                     │
│           ┌─────────┐               │
│           │  4.5x   │  ← L1: --fs-display, --accent-blue-vivid, --fw-bold
│           └─────────┘               │
│        生産性向上率                   │  ← L2: --fs-subheading, --fg
│                                     │
│  AI導入企業 n=500 の平均値            │  ← L3: --fs-small, --fg-dim
│                                     │
└─────────────────────────────────────┘
```

**デザインTips**:
- 数値の周囲に --space-8 以上の余白
- 数値に box-shadow: var(--glow-blue) を微量適用でさりげない光彩
- 単位（x, %, 円）は数値より1段階小さいフォント

### 2.2 Before → After パターン

**用途**: 変化・改善を視覚的に対比

```
┌─────────────────────────────────────┐
│  Before → After で見る改善効果        │  ← タイトル
│                                     │
│  ┌──────────┐    ┌──────────┐       │
│  │  Before  │ →  │  After   │       │
│  │          │    │          │       │
│  │ --bg-dim │    │ glass-   │       │
│  │ グレー調  │    │ card     │       │
│  │ pink注意 │    │ aqua成功 │       │
│  └──────────┘    └──────────┘       │
│                                     │
└─────────────────────────────────────┘
```

**デザインTips**:
- Before=ミュートトーン（--bg-dim + --accent-pink-vivid）
- After=ブライトトーン（.glass-card + --accent-aqua-vivid）
- 矢印は --fg-muted で控えめに
- After パネルに box-shadow: var(--shadow-medium) で浮遊感

### 2.3 3ステッププロセスパターン

**用途**: 手順・フローを3ステップで表現

```
┌─────────────────────────────────────┐
│          導入プロセス                 │
│                                     │
│  ┌────┐      ┌────┐      ┌────┐    │
│  │ 01 │ ──→  │ 02 │ ──→  │ 03 │    │
│  │    │      │    │      │    │    │
│  │分析│      │設計│      │実装│    │
│  └────┘      └────┘      └────┘    │
│  説明文       説明文       説明文    │
│                                     │
└─────────────────────────────────────┘
```

**デザインTips**:
- ステップ番号はビビッドカラー + --fw-bold
- コネクタ矢印は --fg-muted で細く
- stagger 0.08s で左→右に順次表示
- 現在ステップがあれば border: 2px solid var(--accent-blue-vivid)

### 2.4 引用・証言パターン

**用途**: 顧客の声、専門家のコメント

```
┌─────────────────────────────────────┐
│                                     │
│      "                              │  ← 大きな引用符: --accent-*-vivid, --fs-display
│                                     │
│     この製品を導入してから、          │  ← --fs-body-lg, --fw-medium, --fg
│     チームの生産性が劇的に            │
│     向上しました。                   │
│                                     │
│              "                      │
│                                     │
│     — 山田太郎, CTO, Example社       │  ← --fs-small, --fg-dim
│                                     │
└─────────────────────────────────────┘
```

**デザインTips**:
- 引用符は装飾的に大きく（--fs-display相当）
- 引用文は --fs-body-lg で読みやすく
- 発言者情報は --fg-dim で控えめに
- 背景に .glass-card を敷くとリッチ感UP

### 2.5 データストーリーテリングパターン

**用途**: グラフ + インサイトを組み合わせ

```
┌─────────────────────────────────────┐
│  市場トレンド分析                     │
│                                     │
│  ┌──────────────────┐  ┌─────────┐  │
│  │                  │  │ KEY     │  │
│  │   グラフ          │  │ INSIGHT │  │
│  │   (chart)        │  │         │  │
│  │                  │  │ 前年比   │  │
│  │                  │  │ +23%    │  │
│  └──────────────────┘  └─────────┘  │
│                                     │
└─────────────────────────────────────┘
```

**デザインTips**:
- レイアウト比率 2:1（グラフ:インサイト）
- インサイトパネルにビビッドカラーの数値
- グラフの注目データポイントだけビビッドカラー、他はミュート
- 「何を読み取るべきか」をインサイトパネルで明示

### 2.6 アイコン+テキストグリッドパターン

**用途**: 特徴・機能・メリットの列挙

```
┌─────────────────────────────────────┐
│          3つの強み                   │
│                                     │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐│
│  │  [icon] │ │  [icon] │ │  [icon] ││
│  │  blue   │ │  aqua   │ │  violet ││
│  │         │ │         │ │         ││
│  │ 見出し  │ │ 見出し  │ │ 見出し  ││
│  │ 説明文  │ │ 説明文  │ │ 説明文  ││
│  └─────────┘ └─────────┘ └─────────┘│
│                                     │
└─────────────────────────────────────┘
```

**デザインTips**:
- アイコンはFontAwesome 6 Free、--fs-icon-lg サイズ
- 各カードのアイコン色を変えて分類を視覚化（同色なら統一感）
- カードは .glass-card + hover:lift で立体感
- stagger 0.05s で左→右に表示

### 2.7 タイムラインパターン

**用途**: 時系列の変化・マイルストーン

```
┌─────────────────────────────────────┐
│          プロジェクト計画             │
│                                     │
│  ●────●────●────●────●              │
│  Q1   Q2   Q3   Q4   完了           │
│  分析  設計  開発  テスト  リリース    │
│                                     │
│  [Q2を強調: --accent-blue-vivid]     │
│                                     │
└─────────────────────────────────────┘
```

**デザインTips**:
- タイムライン線は --fg-muted で控えめに
- ノードは --bg-card 背景、アクティブノードだけビビッドカラー
- stagger で左→右に順次表示
- 各ノードにホバーで詳細ツールチップ

### 2.8 問題→解決フレームパターン

**用途**: 課題提起から解決策提示

```
┌─────────────────────────────────────┐
│                                     │
│  ┌─────────────────────────────┐    │
│  │  こんな課題はありませんか？    │    │  ← --accent-pink-vivid アクセント
│  │                             │    │
│  │  ・課題1                     │    │
│  │  ・課題2                     │    │
│  └─────────────────────────────┘    │
│               ↓                     │
│  ┌─────────────────────────────┐    │
│  │  解決策                      │    │  ← --accent-aqua-vivid アクセント
│  │                             │    │
│  │  ソリューション説明           │    │
│  └─────────────────────────────┘    │
│                                     │
└─────────────────────────────────────┘
```

**デザインTips**:
- 課題パネル: --accent-pink-vivid ボーダー or 背景ティント
- 解決パネル: --accent-aqua-vivid ボーダー or .glass-card
- 矢印は --fg-muted で控えめに
- enterアニメーション: 課題→矢印→解決の順で stagger 表示

## 3. デザインセンスを高める10のTips

### 3.1 「空白」を恐れない

空白は「何もない」のではなく「意図的な呼吸」。Appleのスライドは50-70%が空白。

### 3.2 要素を揃える

数ピクセルのずれが「雑」な印象を生む。CSS Gridを使えばピクセル単位で揃う。

### 3.3 一貫性を保つ

同じ種類の要素は同じスタイル。border-radiusが12pxと16pxが混在するだけで品質が落ちる。

### 3.4 コントラストを大胆に

差をつけるなら2倍以上。微妙な差は「間違い」に見える。

### 3.5 色は引き算

使う色を減らすほど洗練される。1スライド2色のアクセントが上限。

### 3.6 動きに意味を持たせる

アニメーションは「装飾」ではなく「情報の順序」を伝える手段。意味のない動きは削除する。

### 3.7 テキストは減らせるだけ減らす

「読ませる」のではなく「見せる」。テキストが多いなら図解に変換できないか検討する。

### 3.8 視覚的重心を意識する

重い要素（大きい・暗い・カラフル）が集中すると重心が偏る。重心はスライド中央〜やや上が理想。

### 3.9 奇数の法則

要素を並べるときは奇数（3、5）が安定する。偶数（2、4）は対称になりやすく退屈になりがち。ただし比較スライドは2が最適。

### 3.10 最後に1要素削る

完成したと思ったら、もう1つ要素を削れないか試す。削っても意味が通じるなら、それは不要な要素。

## 4. パターン選択フローチャート

```
何を伝えたい？
├── 1つの数値 → ヒーロー数値パターン
├── 変化・改善 → Before/Afterパターン
├── 手順・流れ → 3ステッププロセス / タイムライン
├── 声・意見 → 引用・証言パターン
├── データ → データストーリーテリング
├── 特徴3-4個 → アイコン+テキストグリッド
├── 課題→解決 → 問題→解決フレーム
└── 迷う → キーメッセージ（テキスト1行+余白）
```

## 5. デザインパターン適用チェックリスト

### 必須チェック

- [ ] 1スライドの要素数が6個以下か
- [ ] テキスト量が80文字以下か（タイトル除く）
- [ ] 余白率が40%以上か
- [ ] stagger で情報の表示順序が制御されているか
- [ ] 最も「重い」要素が最も伝えたいメッセージか

### 品質チェック

- [ ] パターンカタログから適切なパターンが選択されているか
- [ ] Apple Keynote 5原則のうち3つ以上が適用されているか
- [ ] 完成後に「もう1つ削れないか」テストしたか
- [ ] アニメーションの全てに情報伝達の意味があるか
- [ ] 視覚的重心がスライド中央〜やや上にあるか

## 失敗パターン提示型スライド（SR-12, 2026-05-09 追加）

### 目的

「やってはいけないこと」を明示する専用スライドタイプ。手順だけでなく**典型的な失敗例**を可視化することで、聴衆の落とし穴回避を支援する（過去フィードバック: 04-04 F-12 由来）。

### 適用シーン

- 現場担当向け研修（推奨ペルソナ → composition-patterns §10.2）
- ツール導入後のアンチパターン共有
- レビュー観点の事前共有

### 推奨レイアウト: `desc-compare`（左=失敗 / 右=正解）

```html
<div class="slider__item slide-misfire" data-type="diagram-misfire">
  <div class="slider__content">
    <h2>よくある失敗と正解</h2>
    <div class="desc-compare">
      <div class="desc-compare__panel desc-compare__panel--bad">
        <div class="dc-badge"><i class="fa-solid fa-triangle-exclamation"></i> 失敗パターン</div>
        <h3 class="dc-title">プロンプトに目的を書かない</h3>
        <pre class="misfire-sample">「コードを書いて」</pre>
        <p class="dc-desc">何を達成したいか・制約・期待形式が欠落 → AI が外す</p>
      </div>
      <div class="desc-compare__panel desc-compare__panel--good">
        <div class="dc-badge dc-badge--ok"><i class="fa-solid fa-check"></i> 正解パターン</div>
        <h3 class="dc-title">目的・入力・期待出力を明示</h3>
        <pre class="misfire-sample">「Python で CSV を読んで月次集計を出すスクリプト。pandas 使用」</pre>
        <p class="dc-desc">AI が判断不要、再現性が高い</p>
      </div>
    </div>
  </div>
</div>
```

### CSS 必須要件

```css
.desc-compare { display: flex; gap: 4%; align-items: stretch; }
.desc-compare__panel { width: 48%; padding: 2vw; border-radius: 1vw; }
.desc-compare__panel--bad {
  border: 2px solid var(--accent-pink-vivid);
  background: color-mix(in srgb, var(--accent-pink-vivid) 6%, var(--bg-card));
}
.desc-compare__panel--good {
  border: 2px solid var(--accent-aqua-vivid);
  background: color-mix(in srgb, var(--accent-aqua-vivid) 6%, var(--bg-card));
}
.dc-badge { font-size: 1.4rem; color: var(--accent-pink-vivid); margin-bottom: 1vw; }
.dc-badge--ok { color: var(--accent-aqua-vivid); }
.dc-title { font-size: 1.85rem; margin-bottom: 0.8vw; }
.dc-desc { font-size: 1.45rem; line-height: 1.6; }
.misfire-sample {
  font-family: 'SF Mono', 'Fira Code', monospace;
  font-size: 1.6rem;
  padding: 1vw 1.4vw;
  border-radius: 0.6vw;
  background: var(--bg-dim);
  margin: 0.8vw 0;
}
```

### バリエーション

| バリエーション | 用途 | 構成 |
|--------------|------|------|
| 単体（`misfire-single`） | 1 つの大きな失敗を深掘り | 失敗例 + 原因 + 改善 + 教訓 |
| 並列（`desc-compare`） | 失敗 vs 正解 | 左右 2 パネル |
| 列挙（`misfire-list`） | 4-6 つの失敗を一覧 | 2x2 / 2x3 グリッド、各セル `失敗+×印+短い改善ヒント` |

### 配置ルール

- **手段（means）章の中盤**に 1 枚配置（手順説明の直後）
- 結論章の前に置かない（ネガティブで終わると印象悪化）
- 1 デッキあたり 1-2 枚まで（詰め込むと「危険・大変」の印象が強くなる）

### 配色

- 失敗側: `--accent-pink-vivid`（SR-2-07 「課題・Before=ピンク」と整合）
- 正解側: `--accent-aqua-vivid`（同「解決・After=アクア」）
- アイコン: `fa-triangle-exclamation` / `fa-check`（絵文字禁止 — feedback_slides_no_emoji_use_fontawesome 準拠）

### スキーマ参照

- `schema.slideType` に `diagram-misfire` を追加することを将来検討（現状は `slide-compare` または `content_generic` にフォールバック）
- 過渡期は `slide-compare` で `axis: "失敗 vs 正解"` を指定して運用
