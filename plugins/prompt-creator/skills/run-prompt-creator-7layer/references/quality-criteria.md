# 品質チェック基準

review-prompt.md（品質レビューエージェント）が参照する品質基準定義。
validate-prompt.py（Script）の構造検証を補完する意味的品質チェック。

## Script vs LLM の検証範囲

| 検証項目 | validate-prompt.py (Script) | review-prompt.md (LLM) |
|---------|---------------------------|------------------------|
| Layer 1-7の存在 | **担当** | - |
| 変数テンプレート空チェック | **担当** | - |
| セパレータ規約 | **担当** | - |
| Prompt作成シート全項目反映 | - | **担当** |
| エージェント設計妥当性 | - | **担当** |
| 依存関係整合性 | - | **担当** |
| 実用性評価 | - | **担当** |
| 簡潔性評価 | - | **担当** |
| 原子性評価 | - | **担当** |

## 1. 網羅性基準

### 1.1 項目反映チェック（マッピングテーブル照合）

| Prompt作成シート項目 | 7層構造の配置先 | 必須度 |
| -------------------- | --------------- | ------ |
| プロンプト名 | Layer 1: メタ情報 > プロジェクトID | 必須 |
| 目的 | Layer 1: 最上位目的 | 必須 |
| 背景 | Layer 1: 背景コンテキスト | 必須 |
| 完了条件 | Layer 1: 成功基準 | 必須 |
| 想定利用者 | Layer 1: プロジェクト概要 / Layer 7: 初回質問設計 | 必須 |
| ドメインの専門用語 | Layer 2: 用語集 | 推奨 |
| ドメインの制約・ルール | Layer 2: ビジネスルール | 推奨 |
| 解決すべき課題 | Layer 2: ビジネスルール / Layer 4: 品質基準 | 必須 |
| 出力で使うツール | Layer 3: ツール定義 | 条件付き |
| 注意点・制約条件 | Layer 4: 共通ポリシー | 必須 |
| プロンプトに対する課題 | Layer 4: 品質基準（設計課題として参照） | 推奨 |
| ゴール材料（達成状態・完了条件） | Layer 5: ゴール定義 / 完了チェックリスト | 必須 |
| 出力フォーマット | Layer 5: インターフェース（引き渡し形式） | 必須 |
| ハンドオフ（直列/並列の接続） | Layer 6: オーケストレーション | 必須 |
| 想定入力例 | Layer 7: ユーザーインタラクション | 推奨 |
| 必要な情報 | Layer 7: 初回質問の設計材料 | 推奨 |
| evaluation_priorities | Layer 4: 品質基準（動的評価基準の生成材料） | 推奨 |
| derivation_log | Layer 1: メタ情報（設計判断の追跡記録） | 推奨 |

### 1.2 反映の質的判定

| 質問 | 判定 | アクション |
|------|------|-----------|
| 必須項目が全て反映されているか？ | はい→PASS | そのまま出力 |
| 未反映項目があるが、プロンプトの目的達成に影響しないか？ | はい→REVISE（軽微） | 修正箇所を指示 |
| 未反映項目がプロンプトの目的達成に影響するか？ | はい→REVISE（要修正） | 修正箇所を指示、再レビュー |
| 核心的な項目（目的・完了条件・ゴール定義）が欠落しているか？ | はい→FAIL | Phase 4からやり直し |

## 2. 整合性基準

### 2.1 エージェント間データフロー

- 全エージェントの入力に提供元が明記されている
- 全エージェントの出力に受領先が明記されている
- 提供元→受領先のチェーンが途切れていない
- 循環依存がない（A→B→C→Aのようなループ）

### 2.2 Layer間整合性

| 照合ペア | チェック内容 |
|---------|-------------|
| Layer 1 ↔ Layer 6 | 成功基準と完了判定が一致 |
| Layer 2 ↔ Layer 5 | ビジネスルールがエージェントに反映 |
| Layer 3 ↔ Layer 5 | ツールがエージェントのツール利用に反映 |
| Layer 4 ↔ Layer 5 | ポリシーがエージェントのポリシーに反映 |
| Layer 5 ↔ Layer 6 | 全エージェントがオーケストレーションで参照 |

## 3. 実用性基準

### 3.1 変数テンプレート

- `{{}}`（空の変数）がない → validate-prompt.pyで検出済み
- 変数名が自己説明的（`{{顧客名}}`は良い、`{{x}}`は不可）
- 同一変数名が一貫して使用されている

### 3.2 Layer 7 初回質問

- 質問がユーザーにとって回答可能
- 質問が過度に技術的でない
- 回答例が具体的で参考になる
- 質問数が必要情報を収集できる適正範囲（多すぎず少なすぎず）

### 3.3 エージェントのゴール定義

- 達成ゴールが「成果状態」で書かれている（手順の列挙ではない）
- 目的・背景から達成ゴールが論理的に導出されている
- 完了チェックリストの各項目が第三者に検証可能な基準
- 固定手順（思考プロセスのステップ列挙）が書かれていない

## 4. 簡潔性基準

### 4.1 テキスト品質

- 冗長な修飾語がない
- 同じ内容が複数箇所で重複していない
- 説明文がプロンプト本文に混入していない
- 1つの概念に1つの表現（一意性）

### 4.2 フォーマット品質

- `#`コメントのみ使用（`====`、`----`なし → validate-prompt.pyで検出済み）
- インデントが一貫している
- フォーマット固有のルール（YAML: 2スペース、JSON: 日本語キー等）に準拠

## 5. 要素原子性基準

> LLMは要素が短く原子的であるほど理解精度が高い。
> 1つの要素（文章）が長いと精度が低下する傾向がある。

### 5.1 フィールド値の粒度

| チェック項目 | 質的基準 | NG例 | OK例 |
|-------------|---------|------|------|
| 1値=1概念 | その値を読んで「何をすればいいか」が一意に決まるか | `"顧客管理と売上分析を行うシステム"` | 2つのフィールドに分割 |
| 値の分解可能性 | これ以上分解すると意味が失われるか | 3行にわたる散文的説明 | 短文 or リスト分解 |
| 構造化優先 | 散文より構造（key-value/リスト/テーブル）で表現できるか | 段落形式の長文 | 箇条書き or サブキー |

### 5.2 長文分解パターン

| 状況 | 分解方法 |
|------|---------|
| 複合的な目的 | サブキー（主目的/副目的）に分割 |
| 複数条件 | 配列（リスト）に分割 |
| 複合的なゴール | 完了チェックリストの個別項目に分割 |
| 背景情報 | 「課題」「原因」「影響」のサブキーに分割 |

### 5.3 原子性の質的判定

| 質問 | PASS条件 |
|------|---------|
| 各値は「これ以上分解すると意味が失われる」最小単位か？ | はい |
| 各値を読んで実行者が「次に何をすべきか」迷わないか？ | はい |
| 散文ブロックを構造化データに変換できる余地がないか？ | はい |

## 6. 意味的深度基準（質ベース判定）

> 構造的に正しいだけでは不十分。各Layerが**実行可能な粒度**まで情報を展開しているか。
> 「数が揃っているか」ではなく「質的に十分か」を基準とする。

### 6.1 Layer 5 ゴール定義の深度

> ゴールシーク型では「手順の深さ」ではなく「ゴールと完了条件の検証可能性」を見る。
> 固定手順が書かれていないこと自体が合格要件。
> L5 サブ構造（節構成・禁止/必須事項）の正本は `seven-layer-format.md`「Layer 5 契約」
> (l5-contract v2.0.0)。本節と §3.3 の判定質問はその契約に従属する。

| 質的基準 | 判定質問 | 合格条件 |
|---------|---------|---------|
| **ゴールの導出可能性** | 目的・背景から達成ゴールが論理的に導けるか？ | 目的→ゴールの因果が追える |
| **成果状態での記述** | 達成ゴールが「成果状態」で書かれ、手順の列挙になっていないか？ | 固定手順（ステップN）が不在 |
| **チェックリストの検証可能性** | 各項目を読んで「達成したか」を第三者が客観判定できるか？ | YES/NO で合否判定できる |
| **知識ベースの適用性** | 参考文献の手法が「ゴール達成にどう使うか」明確か？ | 手法とゴールの対応が具体的 |
| **ハンドオフ整合性** | 出力の受領先と次エージェントの入力提供元が接続しているか？ | チェーンが途切れていない |

### 6.2 Layer 2 評価基準の具体性

| 質的基準 | 判定質問 | 合格条件 |
|---------|---------|---------|
| **レベル識別可能性** | 各レベルの境界が明確で、どのレベルに該当するか迷わないか？ | 具体的な条件で識別可能 |
| **用語の関係性** | 用語が孤立せず、関連概念とのつながりが理解できるか？ | 概念マップとして機能する |

### 6.3 Layer 6 遷移ロジック

| 質的基準 | 判定質問 | 合格条件 |
|---------|---------|---------|
| **完了条件の検証可能性** | 「このフェーズは完了した」と客観的に判断できるか？ | 曖昧さなく完了を判定できる |
| **引き渡しの明確性** | 次フェーズが受け取るデータの形式・内容が明確か？ | 次フェーズが「何を受け取るか」事前に分かる |
| **確認ポイントの妥当性** | ユーザー確認が「手戻りリスクが高い箇所」に設定されているか？ | 確認なしで進むと問題が起きる箇所にある |

### 6.4 Layer 7 回答例の多様性

| 質的基準 | 判定質問 | 合格条件 |
|---------|---------|---------|
| **パターンカバレッジ** | 想定される入力パターンを網羅しているか？ | 典型・境界・例外ケースをカバー |
| **ドメイン転用可能性** | 例を見て「自分のケースに当てはめる」ヒントが得られるか？ | 異なる文脈への応用が想像できる |
| **難易度グラデーション** | 初心者〜上級者まで参考になる例があるか？ | 段階的に複雑さが増す例がある |

### 6.5 出力テンプレートのメタデータ

| 質的基準 | 判定質問 | 合格条件 |
|---------|---------|---------|
| **根拠の追跡可能性** | 出力を見て「なぜこの結果になったか」遡れるか？ | 判断プロセスが記録されている |
| **手法の再現可能性** | 同じ入力・同じ手法で同じ結果が得られるか？ | 使用手法が明記されている |
| **適用範囲の明示** | この結果が「どこまで有効か」限界が分かるか？ | 前提条件・制約が記載されている |

### 6.6 深度の質的判定

> 数量カウントではなく、以下の質問に「はい」と答えられるかで判定する。

| 判定質問 | PASS条件 |
|---------|---------|
| 達成ゴールが成果状態で書かれ、固定手順を含んでいないか？ | はい |
| 完了チェックリストの各項目が第三者でも検証可能か？ | はい |
| 回答例が想定入力パターンを十分にカバーしているか？ | はい |
| 出力を見て「なぜこうなったか」「どこまで有効か」分かるか？ | はい |

**1つでも「いいえ」があれば該当箇所を修正 → 再判定**

## 7. 動的評価基準フレームワーク

> ヒアリング時に収集した `evaluation_priorities` に基づき、Pass 1-4の各チェック項目を動的に強化する。
> これにより、ユーザーが重視する品質観点が優先的にレビューされる。
> **正本 (SSOT)**: `evaluation_priorities` の語彙（日本語 5 値）と上限（最大 2）の機械正本は
> `run-prompt-elicit/schemas/hearing-result.schema.json` の `evaluation_priorities` enum（2026-07-02 移行済み）。
> 本節 §7.2 は同 enum 値の**意味マッピング（優先度→Pass 強化）のみ**を定義し、値を再定義しない
> （enum 変更時は §7.2 のマッピング行を追従更新する）。workflow-guide.md Round 2.5 /
> run-prompt-elicit R1-interview / interview-user agent の質問文言・個数は同 enum に従属する。

### 7.1 動的基準の生成プロセス

```
入力: hearing-result.json の evaluation_priorities + domain情報
  ↓
生成: 優先度に応じたPass別の追加チェック項目
  ↓
適用: Pass 1-4の各判定質問に動的項目を追加
```

### 7.2 優先度→Pass強化マッピング

| evaluation_priority | 強化対象Pass | 追加チェック項目 |
|---------------------|-------------|-----------------|
| 正確性・精度 | Pass 3 (深度) | 判断基準に数値・閾値が含まれるか？曖昧語（「適切に」「十分に」等）が排除されているか？ |
| 創造性・柔軟性 | Pass 4 (実用性) | 回答例に多様なアプローチが示されているか？制約が創造性を不必要に制限していないか？ |
| ユーザー親和性 | Pass 4 (実用性) | 専門用語に全て説明が付いているか？初回質問が3問以内で始められるか？ |
| ドメイン専門性 | Pass 1 (網羅性) + Pass 3 (深度) | 業界固有用語がLayer 2に網羅されているか？評価基準がドメイン慣行に準拠しているか？ |
| 実行速度・効率 | Pass 2 (整合性) | 冗長なステップがないか？入力項目が最小限に絞られているか？ |

### 7.3 動的基準の適用ルール

| ルール | 説明 |
|--------|------|
| 追加のみ | 動的基準は既存チェック項目に追加する（既存項目を削除・緩和しない） |
| 最大2項目 | evaluation_prioritiesは最大2つ → 追加チェックも最大2観点分 |
| Pass横断可 | 1つの優先度が複数Passに影響してよい（例: ドメイン専門性→Pass 1 + Pass 3） |
| 判定基準統一 | 追加項目も既存項目と同じ「はい/いいえ」判定で評価する |

### 7.4 導出確認との連携

動的評価基準は `derivation_log` と合わせて透明性を確保する：

- ユーザーが選択した優先度 → どのPassが強化されたか → レビュー結果に明示
- レビュー結果に「動的基準による追加チェック」セクションを設け、通常チェックと区別する

## 8. 冪等更新基準（既存プロンプト改善時）

> 既存プロンプトをゴールシークで改善・更新するときの重複回避基準。
> 正本は `references/idempotent-update-policy.md`。本節は判定要約のみ。

### 8.1 上書き vs 新規の判定

| 質問 | 判定 |
|------|------|
| 提案改善が既存要素と同一対象/同一意図/同一改善ポイントを指すか？ | はい→最類似要素を**上書き統合** |
| 既存に類似要素が一切ないか？ | はい→**新規追加** |
| 改善後、同一意図の要素が 2 つ以上残っているか？ | はい→FAIL（統合して重複ゼロにする） |

### 8.2 冪等更新の質的判定

| 判定質問 | PASS条件 |
|---------|---------|
| 改善前に既存内容を原子要素へ分解・分析したか？ | はい（分析前に追加/上書きを決めていない） |
| 上書きは情報を失わない統合か（劣る表現のみ破棄）？ | はい |
| 同一意図の要素が重複せず一意に決まるか？ | はい |
| ゴールシーク反復を分離 context（SubAgent/チーム）で実行したか？ | はい（親へは最終差分のみ） |

**1つでも「いいえ」があれば該当箇所を修正 → 再判定**
