# 7層構造プロンプトフォーマット

プロンプト生成時に使用する7層構造の論理テンプレート（正規形）。
構造定義は YAML で示すが、**最終成果物の既定出力フォーマットは Markdown** とし、
`convert-format.py` で YAML 正規形 → Markdown / JSON / XML へ変換する。

## アーキテクチャ原則

本7層構造は**Clean Architecture**と**Domain-Driven Design**に基づく。

### 依存性ルール（Clean Architecture）
- 内側の層は外側の層を参照しない（L1は他のどの層にも依存しない）
- 外側の層は内側の層に依存する（L7→L6→L5→L4→L3→L2→L1）
- Layer番号が小さいほど安定・不変（変更頻度が低い）

### DDD適用マッピング

| Layer | Clean Architecture | DDD概念 | AI処理順序 |
|-------|-------------------|---------|-----------|
| 1 | Enterprise Business Rules | Value Objects（不変定義） | 最初に読む（コアコンテキスト） |
| 2 | Entities / Domain Model | ユビキタス言語・集約ルート | 2番目（用語を理解） |
| 3 | Frameworks & Drivers | 腐敗防止層（ACL） | 3番目（利用可能ツール把握） |
| 4 | Application Business Rules | 横断的関心事 | 4番目（制約を把握） |
| 5 | Use Cases | 境界づけられたコンテキスト | 5番目（実行単位を理解） |
| 6 | Controllers | ドメインサービス/Saga | 6番目（実行計画を策定） |
| 7 | Interface Adapters | アプリケーションサービス | 最後（ユーザー対話設計） |

### AI処理最適化指針

- **要素原子性**: 1フィールド=1概念、1値=1短文（目安50文字以内）。長文は精度低下の原因
- **長文分割**: 複合的な内容はリスト・テーブル・サブキーに分解して短く保つ
- **構造化データ優先**: prose（散文）よりkey-value、テーブル、enumを使用
- **ID参照**: 名前参照ではなくID参照で曖昧性を排除（例: `@agent_1`）
- **自己完結セクション**: 各Layerは独立して理解可能に設計
- **処理順序 = 読み順**: Layer 1→7の順で読めば依存関係が自然に解決
- **明示的制約**: 暗黙のルールは排除、全制約をLayer 4に集約
- **一意表現**: 1概念1表現、同じ意味を別の言い方で書かない

### ゴールシーク原則（手順非記述）

- **固定手順を書かない**: Layer 5 のエージェントには「手順（ステップ列挙）」を持たせない。
  目的・背景・達成ゴール・完了チェックリストだけを宣言する（背景: 状況に応じて
  実行時に最適な手順を組めるようにするため。固定手順は動的適応を阻害する）。
- **チェックリスト駆動**: 完了チェックリストがゴール到達の唯一の判定基準であり、
  ゴールシークループの停止条件（postcondition）になる。
- **動的手順生成**: 各エージェントは「ゴール・目的・背景・完了チェックリスト」を参照し、
  未充足項目を解消する手順をその都度自ら設計・実行・自己評価し、全項目充足まで反復する。
- **中間成果物アンカー（Anchor Step）**: 各周回末に `original_goal`（初期ゴール。全周回で不変）/
  `current_goal_snapshot`（今周回の到達点 1 文）/ `delta_from_original`（初期ゴールとの差分）/
  `merged_directive_for_next`（差分を圧縮する次周回指示）/ `drift_signal`
  （initial/aligned/compressing/stagnant/widening/oscillating の 6 値）を append-only に記録し、
  次周回の手順立案（ループ Step 2）の必須入力にする。固定手順を持たないループが
  集約化ドリフト（具体的要望が一般解へ流れる）を起こした瞬間に検知・押し戻すための機構。
- **ハンドオフ連鎖**: 各エージェントの出力は次エージェント（直列）または並列エージェントが
  そのまま入力として実行できる形式・粒度で受け渡す。
- **正本追従**: 本節と各ループ定義（Layer 5 実行方式.ループ / Layer 6 ゴールシークループ）は
  harness-creator `run-build-skill/references/goal-seek-paradigm.md`
  （since 2026-05-24 / **6 ステップ・Step 5=Anchor Step 版** / 追従日 2026-07-02）に追従する。
  生成物は配布先で単独動作するため自己完結記述を維持し、正本更新時は本ファイルを追従更新して
  本行の追従日を書き換える（drift 検出用）。反復上限の既定値は
  `run-elegant-review/references/convergence-policy.json` の `loop_bounds` を出所とする（下記 Layer 4 注記）。

## Layer 5 契約（L5 サブ構造の正本定義 / l5-contract v2.0.0, 2026-07-02）

Layer 5 のサブ構造（節番号・見出し名・各節の意味・禁止/必須事項）は**本節が唯一の正本**。
YAML 形（本ファイルのテンプレート）と Markdown 形（`seven-layer-markdown-template.md`）は本契約の
提示形式であり、サブ構造を独自に再定義しない。
旧構成（`5.2 推論手順（再現可能）` / `5.3 自己検証 checklist（self_evaluation_checklist）`）は
本契約 v2.0.0 で**廃止**（固定手順列挙禁止に抵触するため）。

### サブ構造（両提示形式の対応）

| ブロック | 意味 | YAML 形 key | Markdown 形 節番号・見出し |
|---|---|---|---|
| 実行主体 | 担当 agent / context-fork 要否の宣言 | `エージェント.名前` / `プロフィール` | `### 5.1 担当 agent` |
| ゴール定義 | 目的・背景・達成ゴール。到達すべき**状態**の宣言（手順ではない） | `ゴール定義`（`目的`/`背景`/`達成ゴール`） | `### 5.2 ゴール定義` |
| 完了チェックリスト | ゴール到達の判定基準＝ループ停止条件 | `完了チェックリスト` | `### 5.3 完了チェックリスト (ゴール到達の停止条件)` |
| 実行方式 | 固定手順を持たない動的手順生成ループの宣言 | `実行方式`（`方針`/`ループ`/`逸脱時`） | `### 5.4 実行方式` |

YAML 形（複数エージェント形）はこの 4 ブロックに加え、知識ベース・インターフェース（ハンドオフ契約）・
依存関係・ツール利用・ポリシーを持つ。Markdown 形（1 prompt = 1 責務 = 1 agent）はハンドオフを Layer 6 に置く。

### 禁止事項

- L5 に固定手順（ステップ列挙）を書かない。「推論手順」「思考プロセス」「手順 / Steps」見出し配下の
  連番手順列挙は形式を問わず全て違反。
- チェックリスト項目に手順を埋め込まない（「Edit で X を書く」型は違反。手順はループ Step 2 で都度生成する）。
- 合格条件を項目数・ステップ数などの数量レンジで定義しない（数量は構造上の下限「非空」のみ許容。
  品質は検証可能性＝質ベースで判定する）。

### 必須事項

- 達成ゴールは成果状態（「〜が〜の状態になっている」完了形・観測可能）で記述する。
- 完了チェックリストの全項目が第三者に YES/NO で判定可能である（検証可能チェックリスト）。
- 実行方式はゴールシークループ（6 ステップ・Step 5=Anchor。下記 Layer 5 テンプレート参照）を宣言し、
  上限は Layer 4 最大反復回数を参照する。

## 7層の役割

| Layer | 名称 | 役割 |
|-------|------|------|
| 1 | 基本定義層 | プロジェクトの不変定義（目的・成功基準・スコープ） |
| 2 | ドメイン定義層 | 用語集・ビジネスルール |
| 3 | インフラストラクチャ定義層 | ツール・外部システム接続 |
| 4 | 共通ポリシー層 | セキュリティ・品質基準・エスカレーション |
| 5 | エージェント定義層 | ゴール駆動の自律実行単位（目的・ゴール・チェックリスト・実行方式・I/O） |
| 6 | オーケストレーション層 | ゴールシーク制御（選択・ハンドオフ・反復・完了判定） |
| 7 | ユーザーインタラクション層 | 初回質問・回答例 |

    - 依存関係
    - ツール利用
    - ポリシー

  # 具体的なエージェント定義

        背景: |
          {{その目的が必要になった背景・前提}}
        達成ゴール: |
          {{目的+背景を踏まえ、何が出来上がれば「到達」とみなすか。

      # ── 完了チェックリスト（ゴール到達の判定基準＝停止条件） ──
      完了チェックリスト:

        # 各項目は「満たすべき状態」を書き、手順では書かない。
        - 項目: "{{検証可能な達成条件1}}"
          判定: "{{第三者が客観的に YES/NO を判定できる基準}}"
        - 項目: "{{検証可能な達成条件N}}"
          判定: "{{第三者が客観的に YES/NO を判定できる基準}}"
        - 項目: "事実確認: 推測を事実として述べていないか"
          判定: "不確実な情報に限定詞が使われている"

      # ── 実行方式（固定手順の代わりに動的生成ループを宣言） ──
      実行方式:

## フォーマット変換ガイド

最終成果物の既定フォーマットは **Markdown**。YAML 正規形を `convert-format.py` で変換する。

### Markdown変換時（既定）
- Layer → `##` 見出し（## Layer 1: 基本定義層）
- ネストされた項目 → `###` 以下の見出し or テーブル
- リスト → `-` 箇条書き
- 完了チェックリスト → `- [ ]` チェックボックス形式
- テンプレート → コードブロック

### JSON変換時
- Layer → トップレベルキー（`"layer1_基本定義": {...}`）
- 配列 → JSON配列
- テンプレート → 文字列値

### XML変換時
- Layer → `<layer1>` タグ
- 属性 → `name="基本定義層"`
- テンプレート → CDATA
