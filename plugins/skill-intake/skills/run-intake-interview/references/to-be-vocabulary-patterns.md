# To-be Vocabulary Patterns

`abstract-answer-patterns.md` と対になる **as-is/to-be 分離** の語彙正本 (goal-spec C7)。
ヒアリング担当の責務は相手の課題・問題・流れ・仕組みの **as-is (現状の事実)** を忠実記録することに限定され、
**to-be (改善提案・理想手順・最適化・一般化)** の設計は後段 build の責務である。

`validate-procedure-completeness.py` (C02) の contamination check は、handoff 対象の as-is フィールド
(`procedure.steps[]` / `procedure.overview` の各テキスト値、および `five_axes.rows[name="真の課題"].content`)
に to-be 語彙が混入していないかを、下記の 3 セクションを機械パースして走査する。raw 会話ログ中の
to-be 発話は検査対象外であり、C01 が handoff 対象 as-is フィールドへ保存しないことで混入なしを維持する。

判定は第二線 (機械的下界検出) である。第一線 = C01 の保存抑止 (R3-as-is-fidelity)、
第三線 = OUT2 独立レビュー。語彙集外の to-be 表現の検出は第一線・第三線が担う。

## 強 to-be シグナル (それ自体が提案・当為・理想・一般化を表す。単独出現で混入とみなす)

- べきである / べき / すべき / するべき
- した方がいい / したほうがいい / する方がいい / した方が良い
- 望ましい / 理想は / 理想的には / 本来は / 本来であれば
- より良い方法 / もっと良い / もっと効率的 / 改善案 / ベストプラクティス
- 一般的には / 一般論では / 普通は / 通常は / 典型的には

## 弱 to-be シグナル (名詞的用法では as-is。当為表現との共起でのみ to-be とみなす)

- 最適化 / 効率化 / 自動化 / 改善 / 強化 / 合理化

## 当為表現 (弱シグナルの近傍にあれば提案的用法と判定する共起語)

- すべき / するべき / べき / した方 / したほう / する方 / たほうがいい
- 望ましい / 理想 / 本来 / 目指す / 目指し

## 判別規則 (C02 実装契約)

1. 各テキスト値に **強 to-be シグナル** が部分一致で含まれれば `contamination.detected=true` とし、
   その語を `matched_terms` に、フィールドパスを `fields` に記録する。
2. **弱 to-be シグナル** は、同一テキスト値内で近傍 (前後いずれか) に **当為表現** が共起する場合のみ
   `detected=true` とする。当為表現を伴わない名詞的用法 (例: 「在庫最適化ツールを毎日使う」) は
   as-is の業務語彙であり `detected=false`。この場合、誤検知を透明化するため `fields` に
   `<path> (warn: 名詞的用法)` を warn 記録するが `detected` は false のままとする。
3. 名詞的用法/提案的用法の判別基準の正本は本ファイルであり、C02 は本ファイルを機械パースする
   (フォールバック既定値は C02 内 DEFAULT_* に持つ)。

## 例

- 「請求書を freee で毎月作成している」→ as-is (混入なし)
- 「請求書作成を **自動化すべき**」→ to-be 混入 (弱シグナル『自動化』+ 当為『すべき』)
- 「在庫最適化ツールを日次で回している」→ as-is (弱シグナル『最適化』が名詞的用法・当為なし → warn のみ)
- 「**本来は** Slack に一本化したい」→ to-be 混入 (強シグナル『本来は』)
- 「**一般的には** CSV で受け渡すことが多い」→ to-be 混入 (強シグナル『一般的には』= 平均回帰)
