## Layer 1: 基本定義層 (不変原則)

### 1.1 不変ルール
- verdict にかかわらず `fidelity-report.md` を必ず書き出す (silent-fail 禁止)。
- score は小数 2 桁固定 (`f"{score:.2f}"`)、文字数は Unicode codepoint 数。
- セクション順は canonical の `id` 昇順固定、タイムスタンプを MD に含めない (冪等性)。

### 1.2 倫理ガード
- R2 のスコアを再計算しない (信頼ベース)。

## Layer 2: ドメイン層 (本質ロジック)

### 2.1 責務 (Single Responsibility)
- 担当: Markdown レンダリング、exit code 確定、verdict 別運用メッセージ。
- 非担当: score 再計算 (R2 結果を信頼)、canonical 再派生 (R1 責務)、Notion 通信。

### 2.2 ドメインルール

| 用語 | 定義 |
|---|---|
| `delta` | canonical と context の差分。`missing_slots / excess_slots / warnings` から構成。 |
| `section table` | section ごとに 1 行を持つ Markdown 表 (id / score / missing / excess / verdict 列)。 |
| `runbook line` | verdict 別の「次の手」案内。CI ログ最末尾に出力。 |

ビジネスルール:
- verdict=pass でも全 section を表に出す (回帰検知用ベースライン)。
- Markdown 出力は冪等 (同一 JSON で生成すると bit-identical)。

### 2.3 入力契約

| field | type | required | 説明 |
|---|---|---|---|
| fidelity-report.json (positional) | path | yes | R2 出力 |

### 2.4 出力契約
- 出力: `<同階層>/fidelity-report.md` (UTF-8, LF)
- 必須セクション: Header / Sections 表 / Top Issues / Next Steps (+ blockers 存在時のみ Blockers)

## Layer 3: インフラ層 (外部依存)

### 3.1 参照リソース

| id | path | when_to_read |
|---|---|---|
| rules | references/fidelity-check-rules.md | verdict 閾値・runbook 文面の根拠 |
| report (input) | fidelity-report.json | R2 が直前に出力 |

### 3.2 外部ツール / API
- Python 3 (json, sys, pathlib)
- 外部 HTTP / Notion API 禁止

## Layer 4: 共通ポリシー層

### 4.1 失敗時挙動
- exit: 0=pass / 1=warn / 2=fail / 64=usage エラー (context/snapshot 不在・閾値順不正)。実体 `validate-notion-fidelity.py` に exit 3 経路は無い。
- context / snapshot 不在・閾値順不正 → exit 64 (verdict 未算出のため MD 未生成のまま停止)。入力 JSON parse 失敗は未捕捉例外で異常終了。
- verdict=warn を 3 回連続観測 → runbook line に R1 (canonical 再生成) 推奨を追記。

### 4.2 観測 / ロギング
- stdout: 1 行サマリ (`[fidelity-guard] verdict=<v> overall=<score>`)。
- stderr: warn/fail の運用メッセージのみ。

### 4.3 セキュリティ
- 入力 JSON 内の PII フィールドは MD に転記しない (key のみ列挙)。

## Layer 5: エージェント層 (ゴール駆動の実行主体)

### 5.1 担当 agent
- `@delta-reporter` (整形器、スコア再計算禁止、context-fork 不要)
- 実体: `scripts/validate-notion-fidelity.py` 内 `_render_markdown()` (独立プロセス不要)

### 5.2 ゴール定義
- 目的: R2 の fidelity-report.json を冪等な Markdown レポートに整形し、verdict 別の運用 runbook と exit code を提示する。
- 背景: JSON のみでは人間レビューと CI ログの可読性が低く、verdict が pass でも回帰検知ベースラインが必要。silent-fail 防止のため verdict にかかわらず MD を必ず書き出す機構が要る。
- 達成ゴール: fidelity-report.md (UTF-8 / LF) が必須セクションを揃えて存在し、同一 JSON 入力で bit-identical 再生成され、exit code が verdict と 1:1 で返る状態。

### 5.3 完了チェックリスト (ゴール到達の停止条件)
- [ ] E1: verdict にかかわらず `fidelity-report.md` が存在 (silent-fail なし)
- [ ] E2: Sections 表の行数 == `len(report.sections)` (canonical id 昇順)
- [ ] E3: verdict と exit code が 1:1 (`pass=0 / warn=1 / fail=2`、usage エラー=64)
- [ ] E4: 同一 JSON で連続 2 回生成し MD が bit-identical (タイムスタンプ非含、determinism)
- [ ] E5: `report.blockers != []` のとき MD に `## Blockers` 節が存在
- [ ] E6: 全 score 表記が `\d\.\d{2}` 形式 (小数 2 桁固定)
- [ ] E7: 必須節 (Header / Sections / Top Issues / Next Steps) が揃っている
- [ ] E8: スコア再計算を行っていない (R2 結果を信頼)、PII 値が MD に転記されていない

### 5.4 実行方式
- 固定手順を持たない。未充足チェック項目を特定→解消手順を都度立案 (ヘッダ / 表 / Top Issues / Next Steps レンダリング / atomic write など)→実行→チェックリストで自己評価→全項目充足まで反復 (上限: Layer 4 最大反復回数)。
- 逸脱時: context / snapshot 不在は Layer 4.1 に従い exit 64 で停止 (verdict 未算出のため MD 未生成)。

## Layer 6: オーケストレーション層

### 6.1 上位 skill との接続
- 呼び出し元: `assign-notion-fidelity-evaluator` SKILL.md Step 2 直後 (R2 の `validate-notion-fidelity.py` 内部で自動実行)
- 後続: verdict=fail → `skill-intake-handoff` に `fidelity-report.md` の絶対パスを渡す

### 6.2 並列性
- 同一 JSON 宛ての並列起動禁止 (出力 MD パス衝突)。

### Markdown スケルトン

```markdown
# Fidelity Report

- **Verdict**: `<pass|warn|fail>`
- **Overall Score**: `0.87`
- **Blockers**: `0`
- **Context**: `<path>`
- **Canonical sha256**: `<short>`

## Blockers
<!-- only if blockers != [] -->

## Sections
| id | score | char | field | viz | missing | excess |
|---|---|---|---|---|---|---|
| section_0_executive_summary | 0.92 | 1.00 | 0.83 | 1.00 | - | overview_table |

## Top Issues
1. **section_3_value_proposition** (score=0.62): char 不足 (180 < 200)。提案: 背景説明を 1 段落追加。

## Next Steps
<!-- verdict 別 runbook line -->
```

## Layer 7: UI / 提示層

### 7.1 ユーザー提示形式 (CLI 規約)
- **exit 0 (pass)**: stdout に `[fidelity-guard] verdict=pass overall=<score>` の 1 行。
- **exit 1 (warn)**: stderr に warn 詳細と「次の手」案内 (日本語 / 1 行 80 文字以内 / 絵文字なし)。warn の意味、Notion 公開継続、3 回連続観測時の R1 再実行推奨、低スコア section の参照節案内を含める。
- **exit 2 (fail)**: stderr に `[fidelity-guard] verdict=fail — render_notion_page.py was halted. See <md path>.` と blocker section id 列挙。

### 7.2 言語
- 本文: 日本語 (verdict / id / JSON key は英語のまま)
