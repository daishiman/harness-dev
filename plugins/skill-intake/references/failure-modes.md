# ヒアリング失敗パターン辞書

ヒアリング中に発生しがちな失敗を 10 種に分類し、検出シグナルと対処法を定義する。SubAgent はこれらを検出したら自動で軌道修正する。

`anti-patterns.md` が結果物の構造的失敗を扱うのに対し、本ファイルは**フロー中の動作的失敗**を扱う。

## 一覧

| # | パターン | 一言 |
|---|----------|------|
| 1 | 表層要望受領 | 最初の言葉を真に受ける |
| 2 | 同意ループ | ユーザーが面倒で全部 YES |
| 3 | 抽象度暴走 | 「効率化したい」で止まる |
| 4 | 完了主義 | 全項目埋めようとして疲弊 |
| 5 | マジックナンバー欠如 | 数値が出ない |
| 6 | 仮説の押し付け | こちらが答えを誘導 |
| 7 | 詰問化 | 5 Whys が圧迫面接化 |
| 8 | 文脈無視 | プロファイル不明のまま専門用語 |
| 9 | 共有相手忘却 | 5 軸のうち share_target が空 |
| 10 | 価値の言語化失敗 | 何が浮くか不明のまま完了 |

---

## 1. 表層要望受領

| 項目 | 内容 |
|------|------|
| 検出シグナル | `purpose.stated` に入れたまま `excavated` が空 |
| 例 | 「フォーム作りたい」だけで Phase 0 を抜けようとする |
| 対処 | `skill-intake-assumption-challenger` を必ず起動。Magic Wand + 5 Whys |

## 2. 同意ループ

| 項目 | 内容 |
|------|------|
| 検出シグナル | Reverse Brief 5 回中 0 回も訂正されない |
| 原因 | ユーザーが疲れている／面倒に感じている |
| 対処 | わざとずらして問う（「Y は週次でしたよね？」と日次→週次に変えて反応を見る） |

## 3. 抽象度暴走

| 項目 | 内容 |
|------|------|
| 検出シグナル | jtbd.so_i_can に「効率化」「便利に」「自動化」のみ |
| 対処 | 「効率化って、具体的には何分が浮きますか？」と数値化を要求 |

## 4. 完了主義

| 項目 | 内容 |
|------|------|
| 検出シグナル | フェーズ反復が上限超過、ユーザー疲労兆候 |
| 対処 | `recommended_next.mode = "verify-only"` に切替。ただし Notion 指定ありの場合は未公開・未一致のまま `run-skill-create` へ渡さず停止する |

## 5. マジックナンバー欠如

| 項目 | 内容 |
|------|------|
| 検出シグナル | 痛み・価値の発話に「だいたい」「いつも」「結構」のみ |
| 対処 | Pain Story で「直近の 1 件」を再現してもらい数値で確定 |

## 6. 仮説の押し付け

| 項目 | 内容 |
|------|------|
| 検出シグナル | 質問が選択肢提示型（「A ですか？B ですか？」）ばかり |
| 対処 | open question に切替（「どんな状況で？」） |

## 7. 詰問化

| 項目 | 内容 |
|------|------|
| 検出シグナル | 「なぜ？」が 3 連続 |
| 対処 | 「もう少し教えてもらえますか？」「具体的にどんな場面？」に変換 |

## 8. 文脈無視

| 項目 | 内容 |
|------|------|
| 検出シグナル | user_profile.technical_level が未確定なまま標準語彙で進行 |
| 対処 | `skill-intake-user-profiler` を最初の 3 発話で必ず起動 |

## 9. 共有相手忘却

| 項目 | 内容 |
|------|------|
| 検出シグナル | five_axes.share_target.answer が空 |
| 対処 | Stakeholder Map を 1 問だけ強制。「これ、最終的に見る人は？」 |

## 10. 価値の言語化失敗

| 項目 | 内容 |
|------|------|
| 検出シグナル | true_problem.depth が standard 止まり |
| 対処 | 「これが解決したら、その時間で何をしますか？」と次行動を聞く |

## 検出から対処までの自動フロー

```
[interviewer 出力]
  ↓
[quality_gate.py]
  ├─ failure-modes パターンマッチ
  └─ 該当あり → 修正指示を返却
  ↓
[interviewer 再実行]
  ├─ 反復 1〜3 回まで
  └─ 上限超過 → mode=verify-only で打ち切り
```

## 反復上限の理由

ユーザー疲労を避けるため、各 SubAgent 内 3 回・フェーズ全体 2 回を上限とする。それでも完了しない場合は停止して不足項目を提示する。Notion 指定ありの場合、部分完了で `run-skill-create` に渡してはならない。`notion-log.json.status=="published"` と `notion_target.page_id` 一致が確認できるまでは next-action 推奨も確定せず、一致後も harness-creator 実行はユーザーの別アクションに委ねる。

## サンプル: 表層要望受領の検出と修正

```
ユーザー: 「フォーム作りたい」
intake.purpose.stated = "フォーム作りたい"
intake.purpose.excavated = ""    ← 空

quality_gate: FAIL（パターン 1 検出）
→ skill-intake-assumption-challenger 強制起動
→ 「フォームができた次の朝、何が違っていますか？」
ユーザー: 「申込集計が手作業 90 分→0 分になってる」
→ excavated = "申込集計の手作業 90 分削減"
quality_gate: PASS
```
