# 価値判定基準

スキル本体に価値はない。価値は **依頼者の時間が浮いた瞬間** に生まれる。ヒアリング結果には**測定可能な価値 KPI** を必ず 1 つ以上含めなければならない。

## Outcome / Output の区別

| 区分 | 定義 | 例 |
|------|------|----|
| Output | スキルが生成するもの | フォーム 1 件、議事録 1 本、スライド 1 枚 |
| Outcome | ユーザーの行動・状態が変わること | 告知準備が 90 分→10 分に短縮、SNS 拡散に時間が回せる |

**Outcome のみが価値**。Output は副産物。

## KPI 3 カテゴリ

| カテゴリ | 単位 | 例 |
|----------|------|----|
| 時間短縮 | 分／時間 | 90 分→10 分 |
| 品質向上 | エラー率／満足度 | ミス率 10%→1% |
| 露出・反応 | 件数／率 | 申込数 +30 件、開封率 +20% |

最低 **1 KPI 必須**、推奨 **2 KPI**（時間 + 品質 or 露出）。

## 時間短縮 KPI の設計

### 計測式

```
削減時間 = 旧手段の所要時間 − 新スキル使用後の所要時間
発生頻度 = 月あたり実施回数
月次削減 = 削減時間 × 発生頻度
年次削減 = 月次削減 × 12
```

### サンプル: google-forms-generator

| 項目 | 値 |
|------|----|
| 旧所要時間 | 90 分／回 |
| 新所要時間 | 10 分／回 |
| 削減時間 | 80 分／回 |
| 月実施回数 | 4 回 |
| 月次削減 | 320 分 ≒ 5 時間 20 分 |
| 年次削減 | 64 時間 |

### 質問テンプレ

1. 「いまこれに何分／何時間かかってますか？」
2. 「月に何回くらい？」
3. 「自動化したら何分まで縮められそう？」
4. 「浮いた時間で何をしますか？」← Outcome

## 品質向上 KPI の設計

| 指標 | 計測方法 |
|------|----------|
| ミス率 | エラー件数 ÷ 全件数 |
| やり直し率 | 再実施件数 ÷ 全件数 |
| 満足度 | 5 段階アンケート |
| クレーム数 | 件数 |

### 質問テンプレ

1. 「いまどれくらいミスが出てますか？」
2. 「ミスが起きるとどう対処してますか？」
3. 「自動化でゼロにできそうですか？難しければ何 % まで？」

## 露出・反応 KPI の設計

| 指標 | 計測方法 |
|------|----------|
| 申込数 | フォーム回答件数 |
| 開封率 | メール開封／配信数 |
| クリック率 | URL クリック／配信数 |
| フォロワー増加 | 月次差分 |

## 「次に何をする？」の重要性

価値の最終確認質問:

> 「<KPI> が達成されたら、次に何をしますか？」

回答が空／「特に何も」→ **真の課題が深掘れていない**。良い回答: 「浮いた 90 分で SNS 投稿の文章を磨く」「次はリマインドメールを自動化する」

## アンチパターン

| アンチパターン | 検出 | 対処 |
|----------------|------|------|
| 数値なき価値 | KPI 全フィールドが文字列のみ | 数値質問を強制 |
| Output 至上主義 | 価値の説明が「フォームができる」のみ | 「できた後に何が変わる？」を問う |
| 抽象な効率化 | 「便利になる」だけ | 「便利の中身は分？回数？金額？」 |

## サマリ提示テンプレ（summarizer 用）

```
このスキルが完成すると…

【浮く時間】
  - 月 320 分（年 64 時間）

【上がる品質】
  - 申込集計のミス: 月 3 件 → 0 件想定

【その先で】
  - 浮いた時間で SNS 拡散の文章を磨く
  - 次は申込者リマインドの自動化に進む
```

## 6 軸目: 上流スキル連携軸（採否任意・連携あれば必須）

複数のスキルを資産連携（参照・流用・拡張）する場合、上流スキル更新時の追従検知が intake 段階で議論されないと生成スキルに陳腐化リスクが残る。

### 適用判定

- 上流スキル資産（references／assets／scripts）を参照する場合は **必須**
- 完全独立スキルは省略可

### 質問テンプレ

1. 「上流スキルの資産を参照する場合、(a) 都度同期 (b) 生成時凍結 (c) ハッシュ比較で差分検知 のどれを採用しますか？」
2. 「陳腐化検知機構は必要ですか？必要ならタイミングと対応は？」
3. 「上流スキルが破壊的変更（API ／スキーマ）した場合の検知・通知方法は？」

### 完了判定

- `upstream_sync.policy` が `per-call` / `frozen` / `hash-diff` のいずれかで明示
- `hash-diff` の場合は比較対象（ファイルパス・hash アルゴリズム）が明示
- `frozen` の場合は snapshot 取得日付が記録される

### サンプル

| 項目 | 値 |
|------|----|
| 上流スキル | presentation-slide-generator |
| 参照資産 | references/style-genome-base.md |
| 追従方針 | hash-diff（毎回比較・差分時に警告） |
| 検知 script | scripts/compare_upstream_hash.py |

## 計測スクリプト

`scripts/measure_value_realized.py` が intake.json を読み、KPI 数を数え、数値が含まれているか、Outcome 質問への回答有無を機械検証する。1 つでも欠ける場合は完了判定 FAIL。

## 連続低下の扱い

`value_realized_score` が直前2回連続で低下した場合、`skill-intake-self-updater` は question-bank への追記を halt する（`status: halted_score_decline`）。これは自己進化ループが質を下げる方向へ暴走することを防ぐ安全装置である。

### 連携している暴走防止条件 (skill-intake-self-updater)

1. `value_realized_score` の直前2回連続低下 → `status: halted_score_decline`
2. `references/question-bank.md` 行数上限 3000 行超過 → `status: halted_capacity`
3. halt 時は `output/<hint>/question-bank.snapshot.md` を保持し、`python3 ${CLAUDE_PLUGIN_ROOT:-plugins/skill-intake}/scripts/update_question_bank.py --rollback <hint>` で復元可能

`measure_value_realized.py --history <self-update.json>` で `previous_scores` と `declining` フラグを返し、self-updater がこの判定に使う。
