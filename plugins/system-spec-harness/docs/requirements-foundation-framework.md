# 要件定義の土台 — 上位概念抽出フレームワーク (要件 C9)

> **なぜこれが必要か**: 現設計は「カテゴリ×プラットフォームの技術マトリクス」(具体・下位概念) から収集を始めるが、その手前に**上位概念 (本質的目的・背景・ゴール・目標)** の抽出が無いと、マトリクスをいくら網羅しても「本当にやりたいこと」から乖離する (spec drift)。
> **原則**: 上位概念を**最初に・しっかり**拾い集め、要件定義書の憲法として固定し、以降の全技術決定をそこへ**トレース (anchor)** する。上位概念がブレなければ、仕様が整った後もブレない。

## 2 層モデル (上位概念 → 具体)

```
┌─ 上位概念層 (requirements_foundation・最初に抽出・要件定義書の憲法) ──────┐
│  本質的目的 / 背景 / ゴール / 目標 / 成功基準 / ステークホルダー          │
│  スコープ / 制約 / 具体的にやりたいこと                                   │
└──────────────────────────┬───────────────────────────────────────────────┘
                           │ serves_goals (トレーサビリティ = anchor)
┌──────────────────────────▼───────────────────────────────────────────────┐
│  下位概念層 (収集マトリクス・既存)                                        │
│  カテゴリ(DB/認証/UI-UX/…) × プラットフォーム(web/mobile/…) の技術要素     │
└──────────────────────────────────────────────────────────────────────────┘
```

## 上位概念の要素 (拾い集める対象・深掘りヒアリングで抽出)

| # | 要素 | 抽出する問い (深掘り技法) | ブレ防止の役割 |
|---|---|---|---|
| U1 | **本質的目的 (essential_purpose)** | 「なぜこのシステムを作るのか」を 5 Whys で表層要望の奥の真の動機まで掘る。「何を作るか」でなく「なぜ」 | 全ての判断の最終根拠。ここがブレると全てブレる |
| U2 | **背景 (background)** | 現状のどんな課題・きっかけ・文脈から生まれたか。なぜ「今」必要か | 目的の妥当性を支える文脈 |
| U3 | **ゴール (goal)** | 達成したい最終状態 (定性)。「完成したらどうなっていたいか」 | 到達点。技術選択の判断軸 |
| U4 | **目標 (objectives)** | ゴールを分解した測定可能な中間目標 (定量・期限付き) | 進捗と優先順位の基準 |
| U5 | **成功基準 (success_criteria)** | どうなれば「成功」と二値判定できるか | 受入の客観基準 (Goodhart 回避) |
| U6 | **ステークホルダー (stakeholders)** | 誰の何の課題を解決するか (JTBD: どんな状況で何を成し遂げたいか) | 価値の受け手。優先順位の源 |
| U7 | **スコープ (scope: in/out)** | 何を含み何を含まないか。対象外の理由 | 際限ない拡張の抑止 |
| U8 | **制約 (constraints)** | 予算 / 期限 / 技術 / 組織 / 法規の制約 | 実現可能性の枠 |
| U9 | **具体的にやりたいこと (concrete_intents)** | 上位概念に紐づく、具体寄りの「細かくやりたいこと」の列挙 (各 intent は U1-U6 のどれに資するか) | 抽象と具体の橋渡し。マトリクス項目の発生源 |

> **抽出の順序と深さ**: U1 (本質的目的) を最優先で深掘りし、U2-U8 で肉付け、U9 で具体へ降ろす。skill-intake の purpose-excavator (5 Whys / JTBD) の設計流儀を着想として借用する (機構は再利用しない)。上位概念が曖昧なまま技術マトリクスへ進まない (未確定の上位概念は再質問し放置しない = C3 往復ヒアリングと同じ規律)。

## アンカリング機構 (spec drift を機械で検出)

1. **抽出**: C01 の新責務 **R0-foundation** が U1-U9 を深掘りヒアリングで抽出し、`spec-state.json` の `requirements_foundation` へ確定する (マトリクス収集 R1-init の**手前**)。
2. **トレース**: カテゴリ×プラットフォームの各セル確定時に `serves_goals: [<goal_id>, ...]` を記録し、その技術決定がどの上位概念 (U3 ゴール / U4 目標) に資するかを明示する。
3. **検証 (anti-drift)**: validate が (a) `requirements_foundation` の U1-U6 非空、(b) 各確定セルが ≥1 の goal へトレース、(c) 逆に「どのゴールにも資さない技術収集」= drift 候補を surface する。
4. **要件定義書**: C03 compile が `system-spec/00-requirements-definition.md` を**最初の章**として生成し、上位概念 (U1-U9) を明示する。以降の各技術章 frontmatter に `serves_goals` を持たせ、要件定義書=憲法として全章を貫通させる。

## 要件 C9 (新規・上位概念抽出→要件定義 anchor)

> **C9**: システム構築ヒアリングは、カテゴリ×プラットフォームの技術収集に先立ち、ユーザーから本質的目的・背景・ゴール・目標・成功基準・具体的やりたいこと (上位概念 U1-U9) を深掘りヒアリングで抽出して `requirements_foundation` へ確定し、生成仕様書の最初の章 (要件定義書) として明示する。各技術決定は `serves_goals` で上位概念へトレースし、どのゴールにも資さない収集を drift として検出する仕組みが C01 (R0-foundation) / C03 (要件定義章) / validate として定義されている。

## 既存要件との関係
- C1 (カテゴリ一覧+収集状態): C9 の下位。カテゴリは C9 の U9 (具体的やりたいこと) から発生し U3/U4 (ゴール/目標) へトレースする。
- C3 (往復ヒアリングで停止しない): U1-U9 の未確定にも同じ resume 規律を適用 (上位概念を完了扱いしない)。
- C6 (章立て出力): 00-requirements-definition.md を先頭章に追加。
- C7 (網羅マトリクス検証): serves_goals トレーサビリティ検証を追加。
