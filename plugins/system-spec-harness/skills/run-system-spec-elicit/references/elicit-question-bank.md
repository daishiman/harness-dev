# elicit-question-bank — カテゴリ×platform ヒアリング質問テンプレ集

R2-interview / R3-reask が未収集セルへ質問を設計するときの補助資料。カテゴリの正本は C04 taxonomy。ここは質問文の雛形であり、収集項目そのものの SSOT ではない。

## 進め方の原則

- 未収集セルを対象に「質問→回答→仕様反映」を1 turn として回す。
- **platform 横断の一括判断を優先**: 「対象プラットフォームはどれですか?」で非対象を一括 `対象外` にすると turn 数を圧縮できる (approval_log に一括承認を記録し、各セルへ `approval_ref`)。
- 対象 platform だけ各カテゴリの要件を確定 (`確定` + qa_ref)。
- 1 invocation は最大 5 turn。超過は未完了保存 (resume)。

## platform スコープ確認 (最初に聞く)

- 「本システムが対象とするプラットフォームは? (Web / モバイル / タブレット / デスクトップ Windows / Linux / macOS)」
- 「対象外のプラットフォームとその理由は?」→ 一括承認 (approval) で非対象列を `対象外` に。

## カテゴリ別 質問テンプレ (対象 platform ごとに確定)

| カテゴリ (id) | 代表質問 |
|---|---|
| データベース (database) | データモデル/永続化/トランザクション境界/スキーマ移行方針は? |
| 認証(ログイン) (auth) | 認証方式(OIDC/独自)/認可(RBAC 等)/セッション/SSO/MFA は? |
| UI-UX (ui-ux) | 画面設計/操作フロー/アクセシビリティ(WCAG)/レスポンシブ要件は? |
| セキュリティ (security) | 脅威モデル/暗号化/入力検証/監査ログ/OWASP Top10 対応は? |
| インフラ (infrastructure) | 実行環境/デプロイ/スケーリング/監視/CI-CD は? |
| バックエンド (backend) | サーバ構成/API 契約/ビジネスロジック/非同期処理は? |
| フロントエンド (frontend) | クライアント構成/状態管理/レンダリング/ビルドは? |
| 保守運用管理 (maintenance-ops) | 運用手順/障害対応/バックアップ/バージョン管理/ドキュメントは? |

## 再質問 (R3-reask)

- 未確定セルへ「前回未回答の点」を絞って再質問する。既に確定/対象外のセルは再質問しない。
- 5 loop 到達時: `next_question` に次の未収集セルの質問を保存して停止 (resume)。

## 再オープン (R4-reopen)

- 確定済みセルの前提が崩れたとき「なぜ再検討が必要か(根拠)」を確認し、reopen (要 reason) してから追加質問へ。
