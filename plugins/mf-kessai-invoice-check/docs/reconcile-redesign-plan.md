## 1. 確定仕様（請求期待の判定ルール）

入力（基準）= **請求確認シート**（プロパティ: 取引先/商品/確認内容/年月(select)/契約開始日/契約終了月/AI確認(checkbox)/チェック済み(checkbox)）。

対象月 `YYMM`（例 2026-06 → `2606`）について:

1. **年月フィルタ**: 年月 select == 対象月の行のみを基準集合とする。
2. **契約境界で集約**: 取引先名だけでなく **取引先 × 商品 × エンドクライアント × 金額** の契約境界で照合（同名異契約＝同じ取引先が業務委託費と利用料を別行で持つケースの取り違えを防ぐ。既存 `mfk_reconcile.find_mf_match` / `_expected_categories` を使用）。
3. **支払サイクル別の請求期待**:
   - **年払い**: 商品 `100億ThinkTank利用料` / `チイキズカン利用料（1年目）`、または確認内容に `期間：…〜…`・`*12` 明記。→ その年月＝更新月に **年額一括** を期待。年払い期間中の各月は月次請求が無いのが正常（シートにもその月の行は無い）。
   - **月額**: 商品 `チイキズカン業務委託費` / `利用料（2年目以降）`、または「月額」明記。→ 当月の **月次請求** を期待。
   - **従量**: 確認内容に `/件`。→ 金額不定のため **自動✓不可（要確認）**。
   - **契約終了月 == 対象月**: 請求は発行されない → **対象外**。
4. **MF掛け払い当月実績を参照専用GET**（`/billings/qualified` + `/transactions`）→ 取引先で索引化。
5. **presence＋金額で照合**: 契約境界の全明細がMFに有れば ✓、欠落なら **発行漏れ**。重複明細はMFで1請求にまとまる前提（presence-based）。数量差は `要確認` へ降格し ✓ にしない。
6. **AI確認 = 機械が付与** / **チェック済み = 人間が付与**（人間列は機械不可侵）。

### 年払い境界の技術判断（重要）
- 「契約開始日 + 365日」の自前計算ではなく、**確認内容の `期間：2025/7/31〜2026/7/30` を一次ソース**にする方が正確。
- ただし **判定ロジック（既存 `billing_lifecycle` の `契約開始日 + 12ヶ月 elapsed`）は差し替えない**。代わりに **抽出層 `sheet_to_master` に「期間」→ 契約開始日 / 年間月数 の構造化抽出を追加**し、契約開始日が空欄の360行も救う（`REVIEW_DATA_INCOMPLETE` を縮減）。explicit-column を最優先する既存SSOTと整合させる。

## 2. 採用アーキテクチャ：責務分離ハイブリッド

3体の独立分析（論理／メタ／システム）が収束した解。**新しいモデルを建てず、既存 reconcile 資産を流用**し、記録先だけ整える。

| レイヤ | 役割 | 実体 |
|---|---|---|
| **請求確認シート**（SoR入力面 + 一覧表示） | 経理の入力面。各行に **AI確認✓（機械が片方向ミラー）** と **判定 select（5値・色付き）** を非破壊で書き戻し。人間列(チェック済み/備考)は不可侵・空欄のみ埋め誤値非投入。 | Notion DB `55555555…` |
| **裏方台帳**（不変アーカイブ・判定SoR） | 判定の不変履歴・期待金額・MF金額・証跡・**逆方向orphan(要マスタ登録)** を保持。当月のみ更新・過去月構造的に不可侵・人間対応済み凍結。**これが「翌月も先月確認済みが残る」アーカイブの実体**。 | 月次チェックDB2（既存 `notion_reconcile_sink` の非破壊性を流用） |

設計原則:
- **片方向ミラー**: 判定SoR=裏方台帳。シートのAI確認/判定は台帳から決定論的に再計算した投影。stale は再実行で自己修復（冪等）。
- **三値以上を潰さない**: 「未照合(空)／AIの確認OK(緑)／対象外(灰)／要確認(黄)／発行漏れ(赤)」を **単一 select** で表現（checkbox 1個に畳まない）。経理は「赤だけ対応」の保存ビュー1枚で回せる。
- **双方向維持**: 逆方向orphan（MF実績あり・シート未登録）はシートに行が無いため台帳側に留め、「要マスタ登録」ビューで提示。
- **集計列を足さない**: 全体トータル等は作らず Notion ビューのフィルタ件数で代替（過去のサマリ列破綻の教訓）。

### 判定5値 ← 既存16 verdict のマッピング（実装時に `verdict-mapping.json` の warning_class と対応付け）
| シート表示(5値) | 色 | 含む内部verdict(例) |
|---|---|---|
| 未照合 | （空） | 照合未実行（判定列が空） |
| AIの確認OK | 🟢 | MATCH 系 |
| 対象外 | ⚪ | SUPPRESS_ENDED / 年払い期間中 / 契約終了月 |
| 要確認 | 🟡 | REVIEW_*（金額差/数量差/従量/データ不足） |
| 発行漏れ | 🔴 | GAP |

## 2.5 スコープ境界 — 掛け払い単体で完結（CSV/OAuthは不要）

本ツールの照合範囲は **請求確認シート ↔ MF掛け払い** で完結する。CSV取込・MFクラウド請求書OAuth連携は**本スコープに含めない**（2026-06-26 ユーザー判断）。

P0 検証で「掛け払いに一度も請求が無い」契約が出る（2606で12社）。これらは機械的には「掛け払い発行漏れ」だが、業務的には次のいずれか — ①真の発行漏れ ②当月新規契約（請求が月末/翌月発行のタイミング差）③別の請求方法。**この区別は経理の業務知識で行う領域**であり、ツールは「掛け払いに実績なし」を正直に提示し、人間が裏方台帳の `人間対応済み` / `請求要否` で一度処理すれば翌月以降は凍結され再通知しない（責務分離ハイブリッドの人間判断レイヤがそのまま担う）。製品横断の機械照合（CSV）は、将来その必要性が明確になった時点で別途検討する（既定では作らない）。

## 4. 未解決の論点（実装時に詰める / 必要ならユーザー確認）
- **【2026-06-26 決定済】クロス製品照合(CSV/OAuth)は本スコープ外 → §2.5**。掛け払い単体で完結し、掛け払いに無い契約は人間判断レイヤで処理する。

- 桁typo（例 `50,0000円`）の正規化方針：金額パーサが typo候補を分離する既存挙動を踏襲（誤✓化しない）。
