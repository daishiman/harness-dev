# xl-skills → harness 移管計画

この文書は、`~/dev/dev/xlocal/xl-skills` の内容をこの harness に反映する前の Runbook である。現時点では **実行しない**。実行前レビューで、この文書の手順・対象・除外・検証ゲートを確定する。

---

## 1. ゴール

- harness を plugin/skill の正本にする。
- `xl-skills` の Git 管理された最新成果物を harness に取り込む。
- harness 配下の取り込み対象領域から Git 管理外の古い情報・生成物・cache を排除する。
- XLOCAL 固有値を `tenants/xlocal/` に分離し、core は企業非依存にする。
- 将来、`xl-skills` 側から harness 側へ symlink 共有できる構造にする。

## 2. 非ゴール

- この手順整理段階では、実ファイル削除・コピー・symlink 作成をしない。
- Keychain / 1Password / `.env` / Google SA json / API token はコピーしない。
- 企業別 fork を作らない。
- XLOCAL 固有の実値を core plugin に残さない。

---

| 種別 | パス | 扱い |

## 4. 取り込み対象の分類

### 4-1. core として取り込む

| 分類 | 例 | 条件 |

| 汎用 plugin | `harness-creator`, `prompt-creator`, `skill-intake`, `skill-governance-*`, `plugin-dev-planner` | 企業固有値なし、または genericize 可能 |

| tests | plugin / script tests | 実 tenant 値なしで動く、または fixture 化できる |
| installers | install scripts / manifests | 正本が harness である前提に更新 |

### 4-2. tenant として移す

| 分類 | 移管先 | 注意 |
|---|---|---|
| XLOCAL Notion DB ID | `tenants/xlocal/notion-config.json` | 実値ファイルは gitignore。example だけ commit |

| 秘密情報 | `.env`, token, service account json, keychain dump |

| Git 内部 | `.git/`, `.git/worktrees/` |

## 4.5 現状診断(harness 側の実態)— なぜ「追加」でなく入れ直しか

`xl-skills` と harness を実測で突き合わせた結果、harness は **git HEAD と作業ツリーが食い違う半移行状態**にある。§5-6 の棚卸し・削除判断はこの事実を起点にする。

| 層 | 実態 | `git ls-files` に出るか |
|---|---|---|
| git HEAD(追跡) | 旧 meta-skill-creator 構造(`skills/` `docs/` `CLAUDE.md` 旧 README) | ○(harness の追跡集合はこれだけ) |
| 作業ツリー(すべて未追跡) | `xl-skills` の**古い部分コピー**(`plugins/` `scripts/` `doc/` `eval-log/` `installers/` `CONVENTIONS.md` `Makefile` …) | ✗(§5 の untracked 一覧に大量に出る) |
| 供給元 `xl-skills` | 現行・最新(plugin 17本) | ―(移管元) |

**作業ツリーが stale(古い)である実測根拠**:

- plugin **7本欠落**: `company-master` / `harness-creator`(harness は旧 `skill-creator` のまま未改名)/ `mf-kessai-invoice-check` / `notion-gmail-send` / `plugin-dev-planner` / `slide-report-generator` / `ubm-goal-setting`
- `skill-intake` が **0.1.1**(`xl-skills` は 0.1.3)
- `scripts/lint-config-version-sync.py` / `scripts/lint-legacy-plugin-name.py` が**未取込**(§7 検証ゲートで使うのに現状 harness に無い)

**§5-6 への含意**:

- §5 の問い「untracked がユーザ作業か古い生成物か」の**答えは大半が『xl-skills の古い部分コピー』**。したがって §6 Step 4 は「部分コピーの上に足し算」ではなく、対象 import path を**一度掃いてから** §6 Step 2 の archive を展開する(= クリーンスレート)。欠落 7本・版ずれ・重複を一掃し、genericize の起点を一意にする。
- 例外として `doc/マルチ企業展開/`(本計画)は `xl-skills` に存在しない未追跡物なので、clean 対象から除外し退避する。§6 Step 3 の `git clean -nd` は `plugins scripts tests installers .claude-plugin ...` に path 限定されており `doc/` を含まないため、この scope を維持すれば本計画は保護される。

**再現コマンド**:

```bash
# harness の追跡集合(= 旧構造だけが返る)
git -C ~/dev/dev/個人開発/harness ls-files | awk -F/ '{print $1}' | sort -u

# plugin 欠落(左のみ = harness に無い)
comm -13 <(ls ~/dev/dev/個人開発/harness/plugins | sort) \
         <(ls ~/dev/dev/xlocal/xl-skills/plugins | sort)
```

## 5. 実行前棚卸し手順

実行時は、必ず次の順で棚卸しする。

```bash
# 供給元の Git 管理ファイル一覧
cd ~/dev/dev/xlocal/xl-skills
git status --short
git ls-files > /tmp/xl-skills.git-files.txt

# 供給元の Git 管理外ファイル一覧
git ls-files --others --exclude-standard > /tmp/xl-skills.untracked.txt
git ls-files --ignored --others --exclude-standard > /tmp/xl-skills.ignored.txt

# harness 側の現状
cd ~/dev/dev/個人開発/harness
git status --short
git ls-files > /tmp/harness.git-files.txt
git ls-files --others --exclude-standard > /tmp/harness.untracked.txt
git ls-files --ignored --others --exclude-standard > /tmp/harness.ignored.txt
```

この段階では削除しない。`/tmp/*.txt` を見て、以下を判断する。

- harness 側の untracked がユーザ作業か、古い生成物か。
- `plugins/`, `.claude-plugin/`, `scripts/`, `tests/`, `installers/` のどこを取り込み対象にするか。
- `doc/`, `analysis/`, `eval-log/` を移管対象に含めるか。原則は docs と運用上必要な reference のみに絞る。

## 6. 反映順

### Step 1: harness 側の作業保護

目的は、削除前に「残すべき未追跡情報」を判定できる状態にすること。

```bash
cd ~/dev/dev/個人開発/harness
git status --short
git diff --stat
```

必要なら `git diff` と untracked 一覧を保存する。実行時に未コミットのユーザ作業がある場合は、移管対象外として保護する。

### Step 2: 供給元から Git 管理ファイルだけを export

`rsync ~/dev/dev/xlocal/xl-skills/ ./` のような丸ごとコピーは禁止する。代わりに `git archive` を使う。

```bash
cd ~/dev/dev/xlocal/xl-skills
git archive --format=tar HEAD \
  plugins scripts tests installers .claude-plugin .claude/commands .claude/agents .github references \
  > /tmp/xl-skills.git-managed.tar
```

必要に応じて対象 path はレビューで増減する。

### Step 3: harness 側の Git 管理外情報を dry-run

```bash
cd ~/dev/dev/個人開発/harness
git clean -nd plugins scripts tests installers .claude-plugin .claude/commands .claude/agents .github references
git clean -ndX plugins scripts tests installers .claude-plugin .claude/commands .claude/agents .github references
```

`-n` の結果をレビューし、削除してよいものだけを確定する。実削除はこの文書の次段階で承認してから行う。

### Step 4: core を反映

承認後、Step 2 の archive を harness に展開する。展開後、企業固有値を core に残したまま完了扱いにしない。

### Step 5: XLOCAL 実値を tenant へ退避

core に存在する実 DB ID、会社プロフィール、Keychain service 固定名、実ドメインを `tenants/xlocal/` に移す。秘密情報はファイル化しない。

### Step 6: genericize

core 側は placeholder + fail-closed にする。

- Notion DB ID は `tenant.json` / `notion-config.json` 由来で解決する。
- Keychain は `keychain_prefix` から `service = "<purpose>.<prefix>"` を導出する。
- 未設定時に `xlocal` / `xl-skills` へ戻らない。

### Step 7: 正本方向を固定

移管完了後の編集方向を固定する。

```text
harness/plugins/*  = 正本
xl-skills/plugins/* = 後続で symlink または read-only mirror
```

symlink 化は別 PR/別手順に分離する。

## 7. 検証ゲート

| ゲート | コマンド例 | 期待 |
|---|---|---|
| Git 管理外混入なし | `git status --short` | 想定外の cache / generated file がない |
| core 固有値なし | `python3 scripts/lint-tenant-isolation.py` | XLOCAL 固有値 0 |
| plugin 構成整合 | `python3 scripts/validate-plugin-completeness.py` | marketplace / bundle / plugin metadata が一致 |
| config version 整合 | `python3 scripts/lint-config-version-sync.py` | config lock と実ファイルが一致 |
| symlink drift なし | `python3 scripts/build-claude-symlinks.py --check` | generated link が一致 |
| XLOCAL tenant 動作 | `python3 scripts/tenant-doctor.py xlocal` | 既存同等 |
| 2 社目分離 | `python3 scripts/tenant-init.py demo-corp && python3 scripts/tenant-doctor.py demo-corp` | 未設定は fail-closed。XLOCAL へ fallback しない |

検証コマンド自体が未実装の場合は、Phase 4 で作る。未実装のまま「移管完了」としない。
