# 構築手順(Phase 0-7)

大元(master)+ テナント派生を段階的に構築する手順。各 Phase は独立にレビュー・マージ可能な単位。用語・全体像は [`README.md`](README.md)、移管前の棚卸しは [`移管計画.md`](移管計画.md)、`tenant.json` の詳細は [`テナント仕様.md`](テナント仕様.md) を参照。

> **前提**: master = この harness リポジトリ。`xl-skills` は最新の汎用 plugin の**供給元**であり、かつ XLOCAL テナントの**実値の抽出元**でもある。Phase 1 で Git 管理された汎用部を取り込み、Phase 3 で XLOCAL 実値を `tenants/xlocal/` へ退避する。

## 進め方の原則

- **今は実行しない**: この文書は実行計画。削除・コピー・symlink は別途承認後に実施する。
- **Git 管理物だけを入力にする**: `xl-skills` は `git archive` / `git ls-files` で読む。cache / worktree / local log / 秘密情報は持ち込まない。
- **取り込み対象の Git 管理外情報は先に棚卸しする**: harness 側の untracked / ignored は `git clean -n` で dry-run し、必要なものを退避してから削除対象を確定する。
- **fail-closed を壊さない**: 既存 config ローダは未設定時に `sys.exit(2)` で停止する。placeholder 化は「空値 + fail-closed」を維持し、黙って動く既定値を残さない。
- **1 Phase = 1 PR** を目安にする。Phase 2 と Phase 3 は「core から抜く」「tenant へ入れる」の表裏なので、同一 PR にして**退避の往復で欠落が出ないこと**を差分で担保してよい。
- **XLOCAL を壊さない**: 全 Phase 完了後、`tenant-build xlocal` + 各 doctor が緑になること(Phase 6)を最終ゲートにする。
- **正本方向を固定する**: 移管完了後は harness 側 `plugins/` が正本。`xl-skills` は後続で symlink または mirror にする。

## Phase 0 — 原則確定・棚卸し設計

**目的**: テナント軸と移管方法を文章で固定し、以降の判定基準を確定する。

1. [`移管計画.md`](移管計画.md) をレビューし、取り込み対象 path / 除外 path / 削除候補を確定する。
2. `CONVENTIONS.md` に**テナント軸**を追記する。三層モデル(層A/B/C)に対し、「層A(配布対象 plugin 本体)は**企業非依存**であること」「企業固有値は `tenants/<company>/` にのみ置くこと」「`tenants/` は層B(プロジェクト固有運用)の下位に属し配布対象外」を明記。
3. `tenants/_template/tenant.json` のスキーマを確定する(→ [`テナント仕様.md`](テナント仕様.md))。`credentials` は purpose map 方式を基本にし、規約外 service だけ完全指定を許す。
4. 中立命名方針を確定: core(`plugins/`, `scripts/`, `marketplace.json`, `README.md`)から `xl-`/`XLOCAL`/`.xl-skills`/実ドメインを排除する対象を洗い出す。

**完了確認**:

- `tenants/_template/tenant.json` のフィールド一覧が [`テナント仕様.md`](テナント仕様.md) と一致する。
- `xl-skills` と harness の Git 管理外棚卸し手順が [`移管計画.md`](移管計画.md) に固定されている。

## Phase 2 — master の genericize(XLOCAL 焼き込み除去)【中核】

**目的**: core から企業固有値を完全に抜き、placeholder のみにする。Phase 3 と対で行う。

### 2-1. Keychain resolver の一元化
4 箇所に vendored された `notion_config.py`(`plugins/company-master/scripts/` / `plugins/skill-intake/scripts/` / `plugins/harness-creator/scripts/` ほか)と `mf-kessai-invoice-check/lib/mfk_keychain.py`、各 `keychain_get_secret.py` の `.xl-skills`/`"xl-skills"` 定数を、**`keychain_prefix` 由来の単一 resolver** に集約する。

- 解決規則: `service = f"{purpose}.{keychain_prefix}"`, `account = keychain_prefix`。
- `keychain_prefix` の供給順: env `HARNESS_KEYCHAIN_PREFIX`(or 既存 `MFK_KEYCHAIN_SERVICE` 等の個別 env)→ 有効テナントの `tenant.json` → **無ければ fail-closed**(黙って `xl-skills` に落ちない)。

### 2-2. DB ID 焼き込みの退避
以下の実 DB ID を placeholder(空文字 + fail-closed)にし、実値は Phase 3 で `tenants/xlocal/` へ:

| ファイル | 退避する値 |
|---|---|
| `plugins/company-master/notion-config.fixed.json` | company-master DB ID |
| `plugins/skill-intake/notion-config.fixed.json` | hearing-sheet DB ID |
| `plugins/mf-kessai-invoice-check/mf-kessai-config.default.json` | `database_id` + 照合 3 DB(`sheet_db_id`/`reconcile_db1_id`/`reconcile_db2_id`)+ `keychain_service` |

### 2-3. 会社プロフィール・ドメイン固有値の退避
- `plugins/contract-generator/references/party_a.default.json`(株式会社XLOCAL 一式)→ placeholder。
- `impersonate=…@shonai.inc` / `google-sa.xl-skills`(`notion-gmail-send` skeleton, repo-root `.notion-config.json`)→ tenant へ。

### 2-4. ガバナンス lint の新設
`scripts/lint-tenant-isolation.py`(新規)を作り、**core(`plugins/`, `scripts/`, `*.json` の配布対象)に企業固有文字列が残っていないこと**を fail-closed 検査する:
- 検査対象パターン: `XLOCAL` / `xlocal`(大小)/ `.xl-skills` / 実ドメイン / 実 DB ID(UUID 形)。
- 除外(allowlist): `tenants/` 配下、`eval-log/`(凍結層)、`doc/`。
- 例外は allowlist file に理由と期限を書く。コード内 `# noqa` 的な局所抑止は原則禁止。
- CI(`.github/workflows/`)と `.pre-commit-config.yaml` に追加。

### 2-5. version 束縛
placeholder 化で焼き込み config が変わるため、対象 plugin の `plugin.json` + `marketplace.json` の version を bump し `make config-version-lock` を再生成(`scripts/lint-config-version-sync.py` が sha256 一致を強制)。

**完了確認**:
```bash
python3 scripts/lint-tenant-isolation.py          # core に企業固有値なし(新設)
python3 scripts/lint-config-version-sync.py        # 焼き込み config の版束縛が緑
grep -rIn --exclude-dir=tenants --exclude-dir=eval-log --exclude-dir=doc \
  -e 'XLOCAL' -e '\.xl-skills' plugins scripts | grep -v placeholder   # ヒット 0 を目視
```

## Phase 3 — テナント層の新設

**目的**: `tenants/` を作り、XLOCAL 実値を tenant #1 として移設する。

1. `tenants/_template/` を作成(`tenant.json` + `notion-config.example.json` + `party_a.example.json` + `ref-company-rules/`。すべて placeholder)。
2. `tenants/xlocal/` を作成し、Phase 2 で抜いた XLOCAL 実値をここへ移設:
   - `notion-config.json`: 全業務 plugin の実 DB ID + parent_page + `keychain_prefix: "xlocal"`。
   - `party_a.json`: 株式会社XLOCAL の会社プロフィール。
   - `tenant.json`: slug=`xlocal`, `keychain_prefix=xlocal`, `enabled_bundles`(XLOCAL が使う plugin 集合)。
3. **秘密情報は移設しない**(Keychain のみ)。`tenants/*/` のうち秘密を含みうるパスは `.gitignore` で保護し、`.example` テンプレのみコミット。
4. 有効化機構を実装: repo-root `.notion-config.json` を `tenants/<active>/notion-config.json` への symlink にする(既存 4 層解決の②で拾われる)。`HARNESS_TENANT=<company>` env でも切替可能に。
5. 実値ファイルと example ファイルを分ける:
   - commit する: `tenant.json`, `notion-config.example.json`, `party_a.example.json`, `ref-company-rules/*.md`
   - commit しない: `notion-config.json`, `party_a.json`, service account json, `.env`

**完了確認**:
```bash
ls tenants/_template tenants/xlocal                # 雛形と tenant#1 が存在
python3 scripts/lint-tenant-isolation.py           # core は依然クリーン
```

## Phase 4 — オンボーディング・オーケストレータ

**目的**: 企業導入を「散在した手動手順」から「3 コマンド」に束ねる。

- `scripts/tenant-init.py <company>`
  - `tenants/_template/` から `tenants/<company>/` を scaffold。`slug`/`keychain_prefix` を対話 or 引数で埋める。
- `scripts/tenant-build.py <company>`
  - `tenant.json` を読み、`.notion-config.json`(有効テナント)をレンダ。
  - Keychain 登録コマンドを提示/実行(`security add-generic-password ... -s <purpose>.<prefix> -a <prefix>`)。既存 `scripts/build-notion-config.py` を **harness 系 3 DB → 全業務 plugin DB** に拡張して再利用。
  - `enabled_bundles` から有効 plugin 集合を確定。
- `scripts/tenant-doctor.py <company>`
  - 有効 plugin の doctor(`company_master.py doctor` / `mfk_doctor.py` / `contract-generator/lib/setup_doctor.py` / `notion-gmail-send/lib/setup_doctor.py`)を統合実行し OK/WARN/FAIL を集約。

**完了確認**:
```bash
python3 scripts/tenant-init.py demo-corp            # tenants/demo-corp/ が生成
python3 scripts/tenant-build.py demo-corp           # .notion-config.json 生成 + Keychain コマンド提示
python3 scripts/tenant-doctor.py demo-corp          # 未設定項目が FAIL として出る(fail-closed 確認)
```

## Phase 6 — 検証・XLOCAL リグレッション

**目的**: 「既存 XLOCAL が壊れていない」かつ「2 社目が分離されている」を実証する。

### 6-1. XLOCAL リグレッション
```bash
python3 scripts/tenant-build.py xlocal
python3 scripts/tenant-doctor.py xlocal    # 全 doctor が従来同等(OK / 想定内 SKIP)
```
- 受け入れ基準: XLOCAL の各 doctor が genericize 前と**同じ結果**を出す(DB 到達・列一致・トークン解決)。

### 6-2. 2 社目の分離(config-only の成立証明)
```bash
python3 scripts/tenant-init.py demo-corp
# demo-corp の tenant.json に別の keychain_prefix / ダミー DB ID を設定
python3 scripts/tenant-build.py demo-corp
```
- 受け入れ基準:
  1. **Keychain 非衝突**: `demo-corp` は `notion-api-key.demo-corp` を読み、`notion-api-key.xlocal` に触れない。
  2. **DB 非混入**: `demo-corp` の書き込み先が XLOCAL DB を指さない。
  3. **fail-closed**: 未設定の秘密/DB があると `demo-corp` の doctor が FAIL で停止し、XLOCAL の既定値に**フォールバックしない**。
  4. **core クリーン**: `scripts/lint-tenant-isolation.py` が緑(企業固有値が core に一切ない)。

## Phase 7 — xl-skills への symlink 共有【後続】

**目的**: 移管後、`xl-skills` 側から harness 正本を参照できるようにする。これは初回移管とは分離し、Phase 1-6 が安定してから行う。

1. symlink 対象を一覧化する:
   - `xl-skills/plugins/<plugin>` → `harness/plugins/<plugin>`
   - `xl-skills/.claude/skills/<skill>` → harness の install/build 出力
   - `xl-skills/.claude-plugin/*` → harness の marketplace/bundle 定義
2. `xl-skills` 側の既存実体を退避し、symlink に置換する。
3. harness 側で編集した変更が `xl-skills` 側に見えることを確認する。
4. 逆方向の編集を禁止する運用を README / hook / doctor で明示する。

**完了確認**:
```bash
test -L ~/dev/dev/xlocal/xl-skills/plugins/harness-creator
python3 scripts/build-claude-symlinks.py --check
```

**受け入れ基準**:
- harness を消すと `xl-skills` 側は実体を失う。つまり正本方向が明確である。
- `xl-skills` 側に独自変更が発生しても、harness に自動逆流しない。
- symlink 対象外の tenant 実値・秘密情報は各環境に残す。
