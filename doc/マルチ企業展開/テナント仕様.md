# テナント仕様 — `tenant.json` スキーマ / overlay 契約 / 注入点マップ

1 企業 = `tenants/<company>/` の overlay 一式。本書はその**契約(何を置けば core がどう解決するか)**を定義する。全体像は [`README.md`](README.md)、移管 Runbook は [`移管計画.md`](移管計画.md)、導入手順は [`構築手順.md`](構築手順.md) を参照。

## 1. `tenants/<company>/` の構成

```
tenants/<company>/
├─ tenant.json          # テナント記述子(本書の中心)
├─ notion-config.example.json   # Notion overlay template(commit)
├─ notion-config.json           # Notion DB ID・parent_page(企業実値, gitignore)
├─ party_a.example.json         # 会社プロフィール template(commit)
├─ party_a.json                 # 会社プロフィール=契約書の甲(企業実値, gitignore)
├─ google-config.example.json   # contract-generator Google 系 template(commit)
├─ google-config.json           # 台帳/出力フォルダ/slack_channel(企業実値, gitignore)
└─ ref-company-rules/   # L0 company 品質/セキュリティ基準(注入用)
```

> **秘密情報(トークン・SA鍵)はrepoに一切置かない**。企業基準で承認したsecret backendだけを使う(§3)。macOSはKeychain、Windowsは暗号化backend実装までBLOCKED、Linux平文XDGは条件付きBLOCKED。`notion-config.json` / `party_a.json` は実値を含むため `.gitignore` 対象とし、`*.example.json` テンプレのみコミットする。

## 2. `tenant.json` スキーマ

`tenant.json` は「この企業は誰で、どの Keychain 名前空間を使い、どの plugin 集合を有効にし、どの overlay ファイルを参照するか」を宣言する**唯一の入口**。core の各ローダは、有効テナントの `tenant.json` からこれらを導出する。

本節の code block は**契約(placeholder)**であり、実インスタンス値を書かない。実値は各 `tenants/<slug>/tenant.json` を参照する(実例 — テナント #1 xlocal: `slug=xlocal` / `display_name=株式会社XLOCAL` / `keychain_prefix=xlocal`)。

```jsonc
{
  "schema_version": 1,

  // --- 同定 ---
  "slug": "<company-slug>",            // テナント識別子(ディレクトリ名と一致・英小文字/ハイフン)
  "display_name": "<会社正式名称>",     // 人間可読名(ログ・レポート表示用)

  // --- Keychain 名前空間(二社の秘密情報の物理衝突を防ぐ核) ---
  "keychain_prefix": "<company-slug>", // service = "<purpose>.<keychain_prefix>", account = keychain_prefix

  // --- core 版整合(方式B 必須・方式A 任意) ---
  "core_compat": "<harness semver release tag>",  // commit SHA は release manifest.source_commit に分離

  // --- 有効 plugin 集合(= 配布目録の選択) ---
  "enabled_bundles": ["skills-full"],      // .claude-plugin/bundles.json に実在する bundle 名(tenant-build が実在を fail-closed 検証)

  // --- overlay ファイル参照(tenants/<slug>/ からの相対パス) ---
  "overlays": {
    "notion_config": "notion-config.json",   // → 有効テナントセレクタ(§6)。context-pointer であり repo 実体を mutate しない
    "party_a":       "party_a.json",         // → contract-generator の甲プロフィール(env PARTY_A_JSON_PATH。旧 XL_PARTY_A_JSON_PATH は期限付 alias、§4-3)
    "google_config": "google-config.json",   // → contract-generator の台帳/出力フォルダ/slack_channel(§5。同じく受け皿 slot)
    "ref_company_rules": "ref-company-rules/" // → L0 company 基準の注入元
  },

  // --- 秘密情報の宣言(下記 §3 で設計) ---
  "credentials": {
    "notion-api-key": {
      "required": true,
      "service": null,
      "account": null,
      "env_fallback": null
    }
  },

  // --- 実値の出所記録(任意) ---
  "notes": "gitignore 実値(notion-config.json 等)の出所(Notion URL・社内資料など)。マシン移行・誤削除時の復元起点"
}
```

> `enabled_bundles` 例の `"skills-full"` について: bundle 中立名は [`構築手順.md`](構築手順.md) Phase 1-4 の改名 mapping(`xl-skills-full` → `skills-full` 等)で確定する。確定と同一 commit で本例を同期すること。

### フィールド定義

| フィールド | 必須 | 意味 | core 側の消費点 |
|---|---|---|---|
| `schema_version` | ○ | スキーマ版。破壊的変更で bump。 | `tenant-build` が互換判定 |
| `slug` | ○ | テナント識別子。`tenants/<slug>/` と一致。 | 全 tenant スクリプト |
| `display_name` | ○ | 表示名。 | doctor/レポート出力 |
| `keychain_prefix` | ○ | 資格情報ストアの名前空間。意味は「**OS 資格情報ストア上の名前空間 prefix**」であり macOS Keychain 専用ではない(フィールド名は互換のため維持 — 2026-07-11 ユーザー要望。§3)。**テナント横断で一意**(`tenant-init` が重複を拒否、`tenant-doctor` が全 `tenants/*/tenant.json` を横断検査)。 | §4 の全 Keychain resolver |
| `core_compat` | 方式B 必須(方式A 任意) | 検証済み harness の semver release tag。commit SHA は release manifest の `source_commit` で検証し、tag / manifest / install receipt の不一致は FAIL。 | `tenant-doctor`(バージョン整合検査) |
| `enabled_bundles` | ○ | 有効にする bundle 名の配列。各名は `.claude-plugin/bundles.json` に実在すること。 | `tenant-build`(配布目録選択・実在検証) |
| `overlays.notion_config` | ○ | Notion overlay の相対パス。 | 有効化で repo-root `.notion-config.json` へ |
| `overlays.party_a` | 契約書を使う企業のみ | 甲プロフィールの相対パス。 | `contract-generator/lib/config_auth.py` |
| `overlays.google_config` | contract-generator を使う企業のみ | Google 系 config の相対パス(§5)。 | `contract-generator/lib/config_auth.py::load_config` |
| `overlays.ref_company_rules` | 任意 | L0 company 基準ディレクトリ。 | evaluator の reference 注入 |
| `credentials` | ○ | 必要な秘密情報の宣言(§3)。 | `tenant-build`(Keychain 登録)/ `tenant-doctor`(検証) |
| `notes` | 任意 | gitignore 実値ファイルの出所記録(復元用)。 | 人間参照(スクリプトは不使用) |

## 3. `credentials` ブロック

`credentials` は「この企業が使う秘密情報を core にどう伝えるか」の契約。秘密値そのものではなく、資格情報ストアの service/account と env fallback の**参照規則**だけを書く。

> **現行実装と目標契約を分離する**: service 名 `<purpose>.<keychain_prefix>` は全 OS 共通の目標命名契約。実装正本 `plugins/skill-governance-automation/scripts/cross_platform_secret.py` の現行解決順は (1) tenant非依存の `XLSKILLS_SECRET_<KEY>` → (2) macOS Keychain / Linux `$XDG_CONFIG_HOME/xl-skills/secrets.json`(chmod 600の平文JSON) / Windows `%APPDATA%/xl-skills/secrets.bin`(base64・非暗号化) → (3) fail-closed である。Linux/Windowsを「OS native credential store対応済み」とは扱わない。
>
> 注意: (a) **Windows の現行 backend は納品不可**であり、Credential Manager または同等の暗号化 backend を Step R0 で実装・検証する。(b) Linux の平文JSONは企業セキュリティ基準ごとに許否を判定し、必要なら Secret Service 等へ置換する。(c) tenant非依存の `XLSKILLS_SECRET_<KEY>` は複数企業間で誤読し得るため Phase 2 で `HARNESS_SECRET_<SLUG>_<PURPOSE>` 相当へ名前空間化し、他slugへのfallbackを禁止する。旧名の WARN・期限管理・撤去は本節末尾の「互換 alias の一般規則」に従う。(d) `keychain_prefix` というフィールド名は互換のため維持する。

推奨形は purpose map である。理由は、重複検出が簡単で、`tenant-doctor` が required purpose を機械的に検証できるため。

```jsonc
"credentials": {
  "notion-api-key": {
    "required": true,
    "service": null,          // null の場合は "<purpose>.<keychain_prefix>"
    "account": null,          // null の場合は keychain_prefix
    "env_fallback": null,     // env fallback を許す場合だけ env 名を書く
    "notes": "全 Notion plugin 共通"
  },
  "mfkessai-api-key": {
    "required": false,
    "service": null,
    "account": null,
    "env_fallback": "MFK_API_KEY"
  },
  "gbizinfo-api-token": {
    "required": false,
    "service": null,
    "account": null,
    "env_fallback": null
  },
  "japanpost-da-api": {
    "required": false,
    "service": null,
    "account": null,
    "env_fallback": null
  },
  "google-sa": {
    "required": false,
    "service": null,
    "account": null,
    "env_fallback": null,
    "config_path_env": "GOOGLE_CONFIG_PATH"   // config パス上書き用 env(秘密値ではない)
  },
  "slack-bot-token": {
    "required": false,
    "service": "xl-skills-slack",   // 既存互換(稼働中エントリの実測値)。新規テナントは null で <purpose>.<prefix>
    "account": "contract-generate/bot-token",
    "env_fallback": null
  },
  "gdrive-service-account": {
    "required": false,
    "service": "xl-skills-gdrive",  // 既存互換が必要な場合だけ完全指定を許可
    "account": "contract-generate/service-account-json",  // 稼働中エントリの実測値(keychain_get_secret.py の既定)
    "env_fallback": null,
    "service_env": "GDRIVE_KEYCHAIN_SERVICE"  // Keychain service 名上書き用 env(秘密値ではない)
  }
}
```

解決規則:

| フィールド | null / 未指定の場合 | 指定された場合 |
|---|---|---|
| `service` | `<purpose>.<keychain_prefix>` | 指定値を使う |
| `account` | `keychain_prefix` | 指定値を使う |
| `env_fallback` | env fallback なし | 指定 env から**秘密値そのもの**の供給を許可 |
| `config_path_env` | パス上書きなし | 指定 env で config ファイルパスを上書き(秘密値ではない) |
| `service_env` | service 名上書きなし | 指定 env で Keychain service 名を上書き(秘密値ではない) |

`env_fallback` は**秘密値そのものを供給する env に限定**する。config パス・service 名の上書き env は `config_path_env` / `service_env` に分離し、`required` の充足判定(下記 fail-closed)には算入しない(パスや service 名が env にあっても秘密値が解決できるとは限らないため)。

fail-closed ルール:

- `required=true` で OS 資格情報ストア / env fallback のどちらにも値がなければ `tenant-doctor` は FAIL。
- `required=false` でも、plugin が有効化されているのに必要値がなければ WARN ではなく FAIL にできる。
- 他テナントの `keychain_prefix`、旧 `.xl-skills`、旧固定 service へ自動 fallback しない。
- `service` 完全指定は互換用途に限定し、将来的には `<purpose>.<prefix>` へ寄せる。
- 既存エントリ互換を意図する完全指定では、**`service` と `account` の両方を既存エントリと一致させる**(Keychain generic password は (service, account) の組で引くため、片方のみの一致では解決できない)。

### 互換 alias の一般規則

移行用の互換 alias(旧 env 名・旧 Keychain service 名・旧 prefix)は、個別に撤去条件を再定義せず本規則に従う:

1. **採用時に必ず WARN を出す**(alias 経由の解決を無音で通さない)。
2. **lint の allowlist で期限管理する**(allowlist 外の旧名参照は lint FAIL)。
3. **撤去条件の既定は Phase 6 全緑後の `schema_version` bump とする**(bump と同一 commit で alias 本体と allowlist entry を削除)。

## 4. config 解決の注入点マップ

core の各ローダは既に「env → overlay → 焼き込み既定」の順で解決する。テナント値は主に **env と `.notion-config.json`(overlay)** から注入される。genericize 後、`keychain_prefix` がすべての Keychain service を統べる。

`keychain_prefix` の供給順(**本節が正本**。構築手順 Phase 2-1 はここを参照する):

1. env `HARNESS_KEYCHAIN_PREFIX`(or `MFK_KEYCHAIN_SERVICE` 等、既存の個別 env による service 直接指定)
2. 有効テナント(§6)の `tenant.json` の `keychain_prefix`
3. どちらも無ければ **fail-closed**(黙って `xl-skills` 等の旧既定に落ちない)

### 4-1. Notion DB ID の注入点(env で個別上書き可)

| DB(role) | env 変数 | env 不在時のフォールバック先 |
|---|---|---|
| hearing-sheet | `INTAKE_NOTION_DATABASE_ID` | `skill-intake/notion-config.fixed.json`(焼き込み) |
| skill-list | `NOTION_DB_SKILL_LIST` | repo-root `.notion-config.json`(overlay 層。焼き込み既定なし・未設定は fail-closed) |
| improvement-request | `NOTION_DB_IMPROVEMENT_REQUEST` | repo-root `.notion-config.json`(overlay 層。焼き込み既定なし・未設定は fail-closed) |
| company-master | `COMPANY_MASTER_NOTION_DATABASE_ID` | `company-master/notion-config.fixed.json`(焼き込み) |
| parent page | `INTAKE_NOTION_PARENT_PAGE_ID` | (空・fail-closed) |
| mf-kessai 請求書チェック出力 | `MFK_DATABASE_ID`(Phase 2 で新設。現実装に env 上書きは無く、gitignore の `.mf-kessai-config.json` 差分のみ) | `mf-kessai-config.default.json` の `notion.database_id`(焼き込み) |
| mf-kessai 出力シート/照合 | `MFK_SHEET_DB_ID` / `MFK_RECONCILE_DB1_ID` / `MFK_RECONCILE_DB2_ID` | `mf-kessai-config.default.json`(焼き込み) |

「焼き込み」行は genericize(構築手順 Phase 2)後に placeholder 化される。mf-kessai の 2 行は構築手順 §2-2 の退避対象(`database_id` + 3 DB)と 1 対 1 に対応する。

- Notion config パス自体の上書き: env `NOTION_CONFIG_PATH`(最優先・明示)。有効テナントは repo-root `.notion-config.json` symlink 経由が既定。

### 4-2. Keychain purpose 一覧(genericize 後 `service = "<purpose>.<keychain_prefix>"`)

| purpose | 用途 | env 上書き | 備考 |
|---|---|---|---|
| `notion-api-key` | Notion トークン | `INTAKE_ALLOW_ENV_TOKEN=1` 時のみ `NOTION_TOKEN` | 全 plugin 共通 |
| `mfkessai-api-key` | MF 掛け払い | `MFK_KEYCHAIN_SERVICE` / `MFK_API_KEY`(非 mac/CI) | |
| `gbizinfo-api-token` | gBizINFO | **env fallback なし**(public API 厳格化) | company-master。env fallbackの無いpurposeは企業基準で承認したsecret backend必須。unknown OSはfail-closed |
| `japanpost-da-api` | 日本郵便 | `COMPANY_MASTER_JAPANPOST_*` | company-master |
| `google-sa` | Google SA 鍵 | `GOOGLE_CONFIG_PATH` 経由 | notion-gmail-send |
| `slack-bot-token` **(規約外 service 互換あり)** | contract-generator の Slack 通知/承認 | env なし(`google-config.json` の `slack_keychain_service` / `slack_keychain_account` で上書き) | 稼働中エントリ `xl-skills-slack` の互換が必要な場合だけ `credentials.slack-bot-token.service` で完全指定(§3) |
| `gdrive-service-account` **(規約外 service 互換あり)** | contract-generator の SA | `GDRIVE_KEYCHAIN_SERVICE` / `GDRIVE_KEYCHAIN_ACCOUNT` | 既存 service 名 `xl-skills-gdrive` の互換が必要な場合だけ `credentials.gdrive-service-account.service` で完全指定 |

規約外 service は `credentials.<purpose>.service` で完全指定する。ただし core の既定値として `xl-skills-gdrive` を残してはいけない。

### 4-3. 会社プロフィール(甲)の注入点

| 値 | 解決順 | 注入点 |
|---|---|---|
| party_a | env `PARTY_A_JSON_PATH`(正準)→ env `XL_PARTY_A_JSON_PATH`(期限付 alias。§3「互換 alias の一般規則」)→ `~/.config/contract-generator/party_a.json` → 同梱 `party_a.default.json`(placeholder) | `contract-generator/lib/config_auth.py::load_party_a` |

- `tenant-build` は `overlays.party_a` の絶対パスを `PARTY_A_JSON_PATH` / `XL_PARTY_A_JSON_PATH` の**両名でエクスポート**する(alias 撤去まで併記。or 同梱 default を tenant 値でレンダ)。

## 5. overlay ファイル契約

### `notion-config.json`(企業実値)
既存 `.notion-config.example.json` と同形。最低限: `parent_page`, 各 `databases.<role>.db_id`, `keychain_service`/`keychain_account`(= `keychain_prefix` から導出)。`notion-gmail-send` を有効にする場合は、現行ローダ契約どおり `notion_gmail_send.sender.impersonate` と `notion_gmail_send.sender.sa_keychain.{service,account}` を同じファイルに置く。**トークン・SA鍵は含めない**。

commit 対象は `notion-config.example.json` のみ。実値 `notion-config.json` は `.gitignore` 対象。

### `party_a.json`(契約書 甲)
必須キー(既存 `PARTY_A_REQUIRED_KEYS` に一致): `name` / `address` / `representative` / `title` / `rep_name`。

commit 対象は `party_a.example.json` のみ。実値 `party_a.json` は `.gitignore` 対象。

### Gmail 委任/impersonate の配置

独立した `gmail-config.json` は新設しない。現行 `notion-gmail-send/lib/notion_config.py` と `.notion-config.example.json` がすでに `notion_gmail_send.sender.impersonate` / `sa_keychain` を消費するため、`notion-config.json` を再利用する。必須値が無ければ doctor は fail-closed とし、旧ドメインや旧serviceへfallbackしない。将来、Gmail設定がNotion設定から独立する実需要が生じた場合だけ、overlay slot追加チェックリストを経て分離する。

### `google-config.json`(contract-generator の Google 系 config)
`contract-generator` の台帳・出力先フォルダ・Slack 通知先は、Keychain 秘密でも Notion DB でも甲プロフィールでもない企業固有 config 値である。現実装の正本は `~/.config/contract-generator/google-config.json`(`lib/config_auth.py::load_config`。env `GOOGLE_CONFIG_PATH` で明示上書き可)。`tenant-build` は `overlays.google_config` の絶対パスを `GOOGLE_CONFIG_PATH` としてエクスポートする。

必須キー(現実装 `config_auth.py::REQUIRED_KEYS` に一致): `spreadsheet_id` / `templates_folder_id` / `individual_folder_id` / `corporate_folder_id`。加えて `slack_channel`(通知先)と Keychain 参照名 `keychain_service` / `keychain_account` / `slack_keychain_service` / `slack_keychain_account`(§3・§4-2 の purpose 解決と対称)。**秘密値そのもの(SA 鍵・Bot Token)は置かない**(OS 資格情報ストアのみ — §3)。

commit 対象は `google-config.example.json` のみ(同梱 `references/google-config.sample.json` と同形)。実値 `google-config.json` は `.gitignore` 対象。必須キー欠落時は fail-closed(`load_config` が例外停止。旧 XLOCAL 値へ fallback しない)。

### `ref-company-rules/`(L0 company 基準)

### overlay 実値ファイルの drift 検査(契約)
実値ファイル(`notion-config.json` / `party_a.json` / `google-config.json`)は gitignore されるため、core 側で必須キーが増えても git diff に現れず無音で腐る。`tenant-doctor` は overlay ごとに「**example の必須キー集合 ⊆ 実値ファイルのキー集合**」を検査し、欠落があれば FAIL とする(example が契約のスキーマ版を兼ねる)。加えて、(a) 共通キーの **JSON 型一致**(example と実値で型が異なれば FAIL)、(b) example に無い**未知キーの WARN**(stale キー・タイポ検知)を検査する。将来は example からの JSON Schema 生成へ拡張する。

### overlay slot 追加時の更新チェックリスト
overlay slot を1つ増やしたら以下を**同一 commit**で更新する: 本書 §1 構成ツリー / §2 フィールド定義表 / §5 契約節 / §7 Git 管理方針表、構築手順.md Phase 3(_template と tenants/<slug> の内容物)、README.md §4 ツリー、企業別リポジトリ運用.md §2(企業 repo 標準構成ツリー)/ §6 Step R2(実値記入の対象一覧)。既存configに格納先がある値のために新slotを作らない。

## 6. 有効化(activate)

あるマシンで「今どの企業か」を選ぶ 2 手段(既存 4 層解決がそのまま拾う):

1. **symlink**: repo-root `.notion-config.json` → `tenants/<active>/notion-config.json`(4 層解決②で採用)。
2. **env**: `HARNESS_TENANT=<company>`(tenant スクリプトが対応 overlay パス群を導出・エクスポート)。

`tenant-build <company>` はこの有効化まで含めて 1 コマンドで行う。

競合時の優先順位と矛盾検出:

- 優先順位: env `NOTION_CONFIG_PATH`(最優先・明示)> env `HARNESS_TENANT` > repo-root symlink。
- `HARNESS_TENANT` と symlink が**異なるテナントを指す場合、`tenant-doctor` は FAIL** とする(黙ってどちらかを採用しない)。誤テナント DB への書き込み事故を fail-closed で防ぐ。

## 7. Git 管理方針

| path | Git 管理 | 理由 |
|---|---|---|
| `tenants/_template/tenant.json` | する | スキーマ雛形 |
| `tenants/_template/*.example.json` | する | 新規企業 scaffold |
| `tenants/<company>/tenant.json` | する | tenant の public contract |
| `tenants/<company>/notion-config.example.json` | する | 必須キーの共有 |
| `tenants/<company>/party_a.example.json` | する | 必須キーの共有 |
| `tenants/<company>/google-config.example.json` | する | 必須キーの共有 |
| `tenants/<company>/ref-company-rules/*.md` | 原則する | 秘密情報を含まない企業ルール |
| `tenants/<company>/notion-config.json` | しない | 実 DB ID |
| `tenants/<company>/party_a.json` | しない | 実会社情報 |
| `tenants/<company>/google-config.json` | しない | 実台帳/出力フォルダ ID・通知先 |
| `tenants/<company>/**/*.secret.*` | しない | 秘密情報 |

`.gitignore` はこの表と一致させる。例外が必要な場合は、例外理由をこの文書へ追記してから運用する。

本表は**方式A(モノレポ内 tenant)**の規定である。納品向けの**方式B(企業ハーネス = sanitized export + fresh history)**では、実値3ファイル(notion-config / party_a / google-config の `.json`)を企業 repo 側に commitする読み替えを行う(秘密 `**/*.secret.*` は方式Bでも管理しない)。方式BのGit管理方針・納品フローは [企業別リポジトリ運用.md](企業別リポジトリ運用.md) §2を正本とする。
