# 構築手順(Phase 0-7)

大元(master)+ テナント派生を段階的に構築する手順。各 Phase は独立にレビュー・マージ可能な単位。用語・全体像は [`README.md`](README.md)、移管前の棚卸しは [`移管計画.md`](移管計画.md)、`tenant.json` の詳細は [`テナント仕様.md`](テナント仕様.md) を参照。

> **前提**: master = この harness リポジトリ。`xl-skills` は汎用 plugin の**供給元**であり、かつ XLOCAL テナントの**実値の抽出元**でもある。「最新」は remote / local / dirty worktree の曖昧な呼称ではなく、移管計画 Source Snapshot Gate を通った `source_snapshot_sha` を指す。2026-07-12 の実行状態は [`実行記録.md`](実行記録.md) を参照。

## 進め方の原則

- **ゲート単位で実行する**: snapshot取込とローカル実装は開始済み。秘密登録・実運用cutover・releaseは各ゲート完了後に分離実行する。
- **承認済み snapshot だけを入力にする**: fetch 後の採用 SHA を固定し、供給元が behind / dirty の間は停止する。cache / worktree / local log / 秘密情報は持ち込まない。
- **取り込み対象の Git 管理外情報は先に棚卸しする**: harness 側の untracked / ignored は `git clean -n` で dry-run し、必要なものを退避してから削除対象を確定する。
- **fail-closed を壊さない**: 既存 config ローダは未設定時に `sys.exit(2)` で停止する。placeholder 化は「空値 + fail-closed」を維持し、黙って動く既定値を残さない。
- **1 Phase = 1 PR** を目安にする。Phase 2 と Phase 3 は「core から抜く」「tenant へ入れる」の表裏なので、**同一 PR を既定**とし、**退避の往復で欠落が出ないこと**を差分で担保する(順序の正は [`移管計画.md`](移管計画.md) §6 Step 5)。
- **XLOCAL を壊さない**: 全 Phase 完了後、`tenant-build xlocal` + 各 doctor が緑になること(Phase 6)を最終ゲートにする。
- **正本方向を固定する**: 移管完了後は harness 側 `plugins/` が正本。`xl-skills` は後続で symlink または mirror にする。

## Phase 2 — master の genericize(XLOCAL 焼き込み除去)【中核】

**目的**: core から企業固有値を完全に抜き、placeholder のみにする。Phase 3 と対で行う。

**実値移動の順序(正)**: 実値は**先に Phase 3 の場所(`tenants/xlocal/`)へ複製し、一致検証してから core から除去する**([`移管計画.md`](移管計画.md) §6 Step 5→6 と同順)。除去を先行させると実値がどこにも存在しない区間が生じるため、逆順にしない。

**baseline 取得**: genericize に着手する前に、XLOCAL 環境で各 doctor(`company_master.py doctor` / `mfk_doctor.py` / `contract-generator` / `notion-gmail-send` の setup_doctor)の出力を取得・保存する。Phase 6-1 のリグレッション比較はこの baseline を対象にする。

### 2-1. Keychain resolver の SSOT 化
vendored された `notion_config.py`(現 harness は `plugins/skill-creator/scripts/` / `plugins/skill-intake/scripts/` の 2 箇所。Phase 1 取込後は `skill-creator` → `harness-creator` の改名(置換)と `company-master` の追加で **3 箇所** = `harness-creator` / `skill-intake` / `company-master`)と `mf-kessai-invoice-check/lib/mfk_keychain.py`、`notion-gmail-send/lib/secrets.py` の `NOTION_SERVICE` / `NOTION_ACCOUNT` 定数(xl-skills 実物 26-27 行に焼き込みあり)、各 `keychain_get_secret.py` の `.xl-skills`/`"xl-skills"` 定数を、**`keychain_prefix` 由来の resolver** に置き換える。

- 実装方式: 物理的な 1 ファイルへの集約は層Aの自己完結規約(`CONVENTIONS.md`「層A 内の plugin 間参照は禁止」)と衝突するため行わない。**SSOT 1 実装 + 各 plugin へ vendored 配置 + `scripts/lint-intake-vendored-ssot.py` 方式の同期 lint で drift 検出**とする。
- 解決規則: `service = f"{purpose}.{keychain_prefix}"`, `account = keychain_prefix`。
- `keychain_prefix` の供給順の**正本は [`テナント仕様.md`](テナント仕様.md) §4**(env `HARNESS_KEYCHAIN_PREFIX` → 有効テナントの `tenant.json` → **fail-closed**。黙って `xl-skills` に落ちない)。
- 有効テナントの発見方法: `HARNESS_TENANT` env、または repo-root `.notion-config.json` symlink の実体パス(`tenants/<slug>/`)からの逆引き(優先順位は テナント仕様 §6)。
- 暫定運用: Phase 2-3 マージ後〜Phase 4(tenant-build)完成前は `tenants/` も自動発見も未整備のため、`HARNESS_KEYCHAIN_PREFIX` を手動 export して運用する(この間の実運用を `xl-skills` 側で継続する場合は影響なし)。

### 2-2. DB ID 焼き込みの退避

| ファイル | 退避する値 |

### 2-3. 会社プロフィール・ドメイン固有値の退避
- `plugins/contract-generator/references/party_a.default.json`(株式会社XLOCAL 一式)→ placeholder。
- party_a の env 名も改名対象: `PARTY_A_JSON_PATH` を正準とし、旧 `XL_PARTY_A_JSON_PATH` は期限付き alias(WARN 付き)として両対応する(正本は テナント仕様 §4-3)。WARN・allowlist 期限管理・撤去条件は [`テナント仕様.md`](テナント仕様.md)「互換 alias の一般規則」に従う。

- CI(`.github/workflows/`)と `.pre-commit-config.yaml` に追加。

## Phase 5 — 配布(企業別)

**目的**: 企業ごとに「core を install → overlay を適用」の 2 段導線に統一する。

**導入者ペルソナ**: 2社目以降の導入者は「macOS / Windows / Linux(WSLはLinux扱い)+ Claude Code + python3を使えるエンジニア」を想定する。ただしOS名だけで納品可能とは判定しない。macOSはKeychain、WindowsはR0で暗号化backendを実装後、Linuxは平文XDG backendを企業基準で承認するかSecret Service等へ置換後に解放する。CIのenv fallbackはtenant scoped化後だけ許可し、env fallbackを持たないpurposeは承認済みbackend必須。unknown OSはfail-closed。成熟度はREADME冒頭を正本とする。

1. 有効 plugin 集合は `tenant.json.enabled_bundles` から `tenant-build` が導出する(生成方式に一本化)。core の `.claude-plugin/bundles.json` へ企業別 bundle を追加する案は、企業固有値を core に置かない分離線(README §4)に違反し、`lint-tenant-isolation` の検査対象へ企業名を書き込むことになるため棄却。
2. **配布経路の設計判断**: 企業側マシンからprivate harnessへのGitHubアクセスは前提にしない。coreの配布可能pluginは版固定bundleとして生成し、private企業repoのGitHub Release assetへmanifest/checksumと共に添付する。`NEVER_DISTRIBUTE` 3本は含めない。install後はreceiptを残す。企業固有pluginは企業repo自身の独立marketplaceで配る。

## Phase 6 — 検証・XLOCAL リグレッション

**目的**: 「既存 XLOCAL が壊れていない」かつ「2 社目が分離されている」を実証する。

### 6-1. XLOCAL リグレッション
前提条件: Keychain の新名称 `<purpose>.xlocal` への再登録が完了していること(Phase 4 tenant-build の移行支援を使用)。

**baseline 再取得**: 本節の実施**直前**に、`xl-skills` 現行環境で各 doctor の baseline を再取得する。Phase 2 冒頭で保存した初期 baseline は Phase 3-5 の期間に本番進化で陳腐化しうるため、再取得により**移管起因の差分と本番進化の差分を区別**する(初期 baseline は差分の切り分け資料として保持)。
```bash
python3 scripts/tenant-build.py xlocal
python3 scripts/tenant-doctor.py xlocal    # 全 doctor が従来同等(OK / 想定内 SKIP)
```
- 受け入れ基準: XLOCAL の各 doctor が**直前に再取得した baseline** と同じ結果を出す(DB 到達・列一致・トークン解決)。

### 6-2. 2 社目の分離(config-only の成立証明)
```bash
python3 scripts/tenant-init.py demo-corp
# demo-corp の tenant.json に別の keychain_prefix を、notion-config.json にダミー DB ID を設定
python3 scripts/tenant-build.py demo-corp
```
- 受け入れ基準:
  1. **Keychain 非衝突**: `demo-corp` は `notion-api-key.demo-corp` を読み、`notion-api-key.xlocal` に触れない。
  2. **DB 非混入**: `demo-corp` の書き込み先が XLOCAL DB を指さない。ダミー DB ID では実 API 到達で検証できないため、tenant-doctor の**解決値ダンプ(dry-run)モード**で解決済み DB ID / Keychain service 一覧を出力し、xlocal 系文字列が含まれないことを機械照合する。
  3. **fail-closed**: 未設定の秘密/DB があると `demo-corp` の doctor が FAIL で停止し、XLOCAL の既定値に**フォールバックしない**。
  4. **core クリーン**: `scripts/lint-tenant-isolation.py` が緑(企業固有値が core に一切ない)。
- e2e ゲート(**2026-07-11 格上げ**): 2 社目で最低 1 plugin の e2e(実在するテスト用 Notion DB への書き込み + doctor 緑)を、**初回納品([企業別リポジトリ運用.md](企業別リポジトリ運用.md) Step R1)着手前までに 1 回必須**とする。本 Phase 時点での実施は任意だが、未実施のまま本 Phase を閉じる場合、証明範囲は「分離の成立」に限られ、受け入れ基準「config-only 稼働実証済み」が未達のままでは R1 に着手できない。

## Phase 7 — xl-skills への symlink 共有【後続】

**目的**: 移管後、`xl-skills` 側から harness 正本を参照できるようにする。これは初回移管とは分離し、Phase 1-6 が安定してから行う。

> **判断記録 — xl-skills retire 案との比較**: Phase 7 を行わず `xl-skills` を廃止(retire)する案は、XLOCAL 日次業務の実行パス・既存 Keychain 参照・ローカル運用スクリプトが `xl-skills` パスに依存し続けるため、cutover 直後の廃止はリスクが高く採らない。symlink 共有で参照互換を維持し、パス依存が解消された時点での retire を後続判断とする。

> **参照元の一般化**: 本Phaseは`xl-skills`を第一参照元として記述するが、同一マシン上の任意の参照元にも同じ「編集起点はharnessのみ・setup再生成・doctor検査」を適用する。正本はREADME §4。

**前提条件**: `xl-skills` 側の変更凍結を宣言し、凍結日時と基準 commit を記録すること。Phase 1 取込以降に `xl-skills` 側へ蓄積した差分は、切替前に再同期レビューで取り込み or 破棄を判定する([`移管計画.md`](移管計画.md) §6)。

> **判断記録 — Phase 6 以前に symlink を張らない理由**: (a) genericize 前の master は焼き込み config(`notion-config.fixed.json`・`.xl-skills` 定数)を持ち、XLOCAL 実運用が直結して Phase 2 の編集が本番業務へ即時波及する。(b) 正本切替前に二重編集起点が生じ、Step 6.5 の差分再同期が破綻する。早期の使い回しは `xl-skills` 側実体の直接参照で代替する。

**実装項目**:
- `scripts/link_master_plugins.py` の実装 — **構築手順 Phase 7（PR-6）で実装**する。Phase 7 完了確認が本 script を使うため、実装は [企業別リポジトリ運用.md](企業別リポジトリ運用.md) §6 Step R0 でなく本 Phase に属する。
- 業務系 skill 実行の前段 hook(または SessionStart hook)で `tenant-doctor --mode maintainer_symlink` を自動起動する(doctor の起動点はこの 1 箇所で確定する)。

1. symlink 対象を一覧化する:
   - 必須: `xl-skills/plugins/<plugin>` → `harness/plugins/<plugin>`（README §4 の `MASTER_LINK_SET` **11本だけ**）。
   - `BUSINESS_TEMPLATE_SET` 6本は一般の必須 symlink ではない。XLOCAL 日次業務の cutover で必要なものを `enabled_bundles` から導出し、追加対象として明示する。結果として XLOCAL は17本になり得るが、任意の参照元へ17本を強制しない。
   - `.claude/skills` と `.claude-plugin` は正本定義から再生成する派生物とし、個別 symlink の必須集合に含めない。生成後の drift check で検査する。
2. `xl-skills` 側の既存実体を退避し、symlink に置換する(既定は symlink。read-only mirror はツール互換問題が実測された場合のみの代替)。
   - **symlink の Git 管理方針**: symlink は **commit しない**(`xl-skills` 側 `.gitignore` に追加)。setup スクリプト(`scripts/link_master_plugins.py`(上記実装項目) — Python 標準ライブラリ(`os.symlink`/`pathlib`)。マスター harness のパスを引数 or env `HARNESS_ROOT` で受け、OS 判定(unknown は fail-closed 停止)を行う。旧記載 `.sh` からの是正理由は [企業別リポジトリ運用.md](企業別リポジトリ運用.md) §3 の判断記録(2026-07-11)を参照)で再生成する。他マシンでの復元は「clone → 同スクリプト実行」で完結させ、絶対パスをリンク実体に焼き込んで repo に残さない。
3. harness 側で編集した変更が `xl-skills` 側に見えることを確認する(即時反映 — 下記 4 の正式仕様)。
4. 逆方向の編集を禁止する運用を README / hook / doctor で明示する。これはファイルシステム上のread-only保証ではない。参照元から書き込めば harness 実体が変わるため、業務実行時は harness worktree dirty を doctor で FAIL にする。あわせて「harness 正本 worktree の HEAD が main（または明示許容した branch/tag）であること」を同 doctor で検査する(clean な非 main branch checkout が全機械検査を素通りする穴を塞ぐ)。tenant 側(`xl-skills` 現場)の改善は禁止ではなく **harness への PR** で受け付ける。
   - **即時反映の正式仕様化(2026-07-11・ユーザー決定)**: symlink は作業ツリー直結であり、harness 側の変更は**マージ・リリースを待たず参照元へ即時反映される**(これが正式仕様。「レビュー・リリース後に伝播」ではない)。**注意**: ブランチ切替・未コミット編集も参照元から見える。このため運用規約を守る: (a) **main を常時デプロイ可能に保つ**、(b) **実験・大規模変更は `git worktree` で隔離**し、main 作業ツリーに未コミット差分を残さない。リリース tag がゲートになるのは企業側マシン向け bundle 配布のみ(伝播範囲の正本は README §4 マトリクス)。

**cutover / rollback**:
- cutover 判定: Phase 6 全緑 + 一定期間の並行検証(従来 `xl-skills` 運用と harness 経由 `tenant-build xlocal` の結果一致)をもって、XLOCAL の日次業務を harness 側へ切り替える。
- rollback: 切替後に重大障害が出た場合、symlink を手順 2 で退避した実体コピーに戻して `xl-skills` 単独運用へ復帰する。退避物は rollback 不要が確定するまで削除しない。

**完了確認**（`MASTER_LINK_SET` 11本・派生物drift・「harness 正本 worktree の HEAD が main（または明示許容した branch/tag）であること」を検査する）:
```bash
python3 scripts/link_master_plugins.py --check --profile master-link-set
python3 scripts/build-claude-symlinks.py --check
python3 scripts/validate-plugin-completeness.py
python3 scripts/tenant-doctor.py xlocal --mode maintainer_symlink   # worktree clean + HEAD が main(または明示許容した branch/tag)
```

**受け入れ基準**:
- harness を消すと `xl-skills` 側は実体を失う。つまり正本方向が明確である。
- `xl-skills` 側に独自変更が発生しても、harness に自動逆流しない。
- symlink 対象外の tenant 実値・秘密情報は各環境に残す。
- symlink は commit されておらず、別マシンでも「clone → setup スクリプト」だけで再生成できる(手順 2)。
