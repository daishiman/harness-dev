# 企業別リポジトリ運用 — 構築・接続・納品

この文書は、**移管完了後(構築手順 Phase 6 全緑 + Phase 7 完了)** に企業を追加・納品するときの Runbook である。[移管計画.md](移管計画.md) §2 の非ゴール(実ファイル操作・symlink 作成をしない)は手順整理段階の制約であり、本書はその「将来やるとき」の手順を先に確定させる。**本書の Step R0-R7 も、移管完了までは実行しない。**

番号体系: 本書は Step R0-R7(R = per-Repo track。R0 = 前提実装)。構築手順 Phase / 移管計画 Step とは独立した納品トラックである(README 冒頭の対応表を参照)。

## 1. 決定(判断記録)

### 採用方式

**方式B: 企業ハーネスモデル(sanitized export + fresh history)。企業ごとに独立した private リポジトリを、harness の許可リスト付き export から新しい Git 履歴で構築する。共有 plugin は企業 repo に複製せず、保守者はマスター(`harness/plugins/`)を symlink 引用し、企業側マシンは版固定 bundle を導入する。企業固有 plugin は企業 repo 側に置き、企業 repo 自体を独立 marketplace として配布する。**

> **訂正記録(2026-07-11)**: 旧案の「harness を fork/コピーしてから XLOCAL 情報を削除」は、削除前の実企業値が Git 履歴に残り得るため撤回した。元 repo の全履歴が配布可能と別途証明された場合を除き、顧客 repo は sanitized allowlist export から fresh history で作る。

企業ハーネスの構成原則:

1. **器 = harness の sanitized scaffold**。必要な構造・設定・運用規約だけを許可リスト export し、`tenants/xlocal/`、実企業値、共有 plugin 実体、履歴を含めない。export manifest と checksum を保存する。
2. **共有 11 プラグイン(`harness-creator` / `plugin-dev-planner` / `prompt-creator` / `skill-governance-*` 7 本 / `skill-intake`)は企業 repo へ複製(commit)しない**。マスターを引用する:
   - **マスター保守者のマシン上**(自分の他ディレクトリ・企業 repo のローカルクローン): **symlink で引用**。symlink は commit せず、setup スクリプト(例: `scripts/link_master_plugins.py`)で再生成する(§3)。
   - **企業側マシン(形態(ii))**: `NEVER_DISTRIBUTE` 3本は提供しない。配布可能プラグイン(governance 7本・`skill-intake`・業務テンプレ6本等)は **marketplace/bundle で版固定配布**する(README §4 の plugin 集合と伝播マトリクス・構築手順 Phase 5)。
3. **秘密(APIキー等)は企業側環境で設定**する: `keychain_prefix` = 企業slugのsecret名前空間 + 企業repo内config。秘密値はrepoへcommitしない。OSごとの納品可否はREADME成熟度表とテナント仕様 §3を正本とする。

[テナント仕様.md](テナント仕様.md) が定めるモノレポ内 tenant(`tenants/<company>/` を harness で直接 Git 管理)は **方式A** と呼び、両方式は併存する。

| | 方式A(モノレポ内 tenant) | 方式B(企業ハーネス = fresh history + bundle)**← 納品標準** |
|---|---|---|
| 適する場面 | 自社運用のみのテナント(例: `xlocal`)、実験用 `demo-corp` | 企業へ**納品**するテナント、企業固有 plugin を持つテナント |
| 実値 config | gitignore(Git 管理外) | **企業 repo に commit**(private repo 前提) |
| 復元経路 | `tenant.json.notes` の出所記録のみ | 企業 repo の git 履歴そのもの |
| 納品単位 | 切り出し作業が必要 | repo がそのまま納品物 |
| 共有 plugin | harness 内の実体をそのまま使用 | **複製せず引用**(保守者マシン = symlink / 企業側マシン = 版固定 bundle) |

### 代替案と判定

| 案 | 判定 |
|---|---|
| 企業別 fork(core ごと複製) | **棄却**。同期コストに加え、fork 後の削除では元履歴にある実企業値を除去できない。元履歴全体の配布安全性を証明できた場合だけ例外審査する |
| sanitized export + fresh history | **採用**。許可リストで器を再現し、企業情報と元履歴を持ち込まず、core は版固定 bundle で更新する |
| git submodule(`tenants/<company>` を submodule 化) | 棄却のまま: harness 側に private 企業 repo への参照(URL・commit pin)が commit され、(a) harness を public 化する場合に企業の存在が漏れる、(b) tenant 側の変更のたびに harness 側で pin 更新 commit が必要になり、core と tenant の変更ライフサイクルが結合する |
| tenant repo に core も同梱(自己完結配布) | 棄却のまま: 共有 plugin の複製は fork の同期問題そのもの。core は harness 正本 + bundle 配布(構築手順 Phase 5)の一系統に保つ(企業ハーネスモデルとの違いは「plugin 実体を commit するか引用するか」) |
| 共通 distribution repo(sanitized core の read-only 配布 mirror + 共通 marketplace) | **不採用**(現段階 N≤1 では per-company Release asset で足りる)。3社目到達時に再評価 |

### 方式B が既存 findings を解消する点

- 実値 config(gitignore)の**バックアップ・復元経路が無い**問題(if思考): 企業 repo への commit で git 履歴が復元経路になる。
- harness を **public 化する場合の企業情報露出**(素人思考): 企業名・企業基準・実値はすべて企業 repo(private)側に隔離される。README §1 の「harness が private であること」前提を、方式B 全面移行後は README §1 の public 化チェックリストを満たしたうえで緩和できる。

### 変わらない原則(方式共通)

- **秘密(API token / SA鍵 / Bot Token)は方式Bでもrepoに置かない。** 企業基準で承認したsecret backendだけに置く。方式Bでcommitするのは実値config(DB ID・会社プロフィール・impersonate等)であって秘密ではない。
- `keychain_prefix` の一意性・fail-closed・有効化優先順位(`NOTION_CONFIG_PATH` > `HARNESS_TENANT` > symlink)・overlay 契約・drift 検査は、方式に依らず [テナント仕様.md](テナント仕様.md) が正本。
- core plugin は fork しない。企業差は overlay + `ref-company-rules` + opt-in 機能フラグで吸収(README §3)。

## 2. 企業 repo(企業ハーネス)の標準構成

```
<company>-harness/                    # private repo(例: acme-harness)。sanitized export から fresh history で構築
├─ README.md                          # 導入手順(納品先向け): 前提・接続・doctor・更新の受け方
├─ CONVENTIONS.md / Makefile / scripts/ / installers/ …   # harness から引き継ぐ構造・設定・運用規約
├─ scripts/link_master_plugins.py     # 共有 plugin symlink の再生成(マスター保守者マシン用。§3)
├─ tenants/<company>/                 # 自社 tenant(テナント仕様 §1 と同形)
│  ├─ tenant.json                     # テナント仕様 §2 と同一スキーマ(唯一の入口。core_compat 必須)
│  ├─ notion-config.json              # 実値。方式B では commit する
│  ├─ party_a.json                    # 実値。commit する
│  ├─ google-config.json              # 実値。commit する
│  ├─ *.example.json                  # 各 example(契約のスキーマ版)
│  └─ ref-company-rules/              # L0 company 品質/セキュリティ基準
├─ plugins/                           # 企業固有 plugin のみ commit(§4)
│  ├─ <company>-<name>/
│  └─ (共有 plugin の実体・symlink は commit しない — マスター引用。§1/§3)
├─ .claude-plugin/
│  └─ marketplace.json                # 企業 marketplace(企業固有 plugins/ がある場合のみ)
└─ .gitignore                         # *.secret.* / cache / run artifact / 共有 plugin symlink(実値 config は ignore しない)
```

- export allowlist に含めないもの(§1 原則1): 元 `.git/`、`tenants/xlocal/`、実企業値を含み得る `doc/` / `eval-log/`、共有 plugin 実体、cache / run artifact / secrets。必要な公開文書は値を検査したうえで個別に許可する。
- **Git 管理方針(方式B)**: テナント仕様 §7 の表のうち「しない」とされる実値3ファイルを「する」に読み替える。`**/*.secret.*` と cache/生成物、および**共有 plugin の symlink** は方式Bでも管理しない。それ以外はテナント仕様 §7 に従う。
- 企業 repo の可視性は **private 必須**。remote URL の形式だけでは public / private を判定できないため、`tenant-doctor` は認証済み GitHub API / `gh repo view` 等で visibility を検査する。API検証不能時は自動PASSにせず、手動attestation未提出としてFAILする。

## 3. マスターへの接続(共有 plugin の symlink 引用)

企業 repo は共有 plugin を commit しない(§1)。マスター保守者のマシンでは、ローカル clone 後に setup スクリプトで symlink を再生成して引用する。パスは環境変数で抽象化する: `HARNESS_ROOT` = マスター harness の作業ツリー、`COMPANY_ROOT` = 企業 repo のローカル clone 先。

```bash
# 企業 repo を任意の場所に clone(例)
git clone git@github.com:<org>/<company>-harness.git "$COMPANY_ROOT"

# 共有 plugin symlink を再生成(commit しない。.gitignore 対象)
python3 "$COMPANY_ROOT/scripts/link_master_plugins.py" "$HARNESS_ROOT"

# 検証
python3 "$COMPANY_ROOT/scripts/tenant-doctor.py" <company>
```

> 具体パスの例(作者マシン): `HARNESS_ROOT=~/dev/dev/個人開発/harness`、`COMPANY_ROOT=~/dev/tenants/<company>-harness`。Runbook 本文には絶対パスを焼き込まない。

- **判断記録(2026-07-11・ユーザー要望)— setup スクリプトの `.py` 化(既存規約違反の是正)**: 旧記載 `scripts/link-master-plugins.sh` は、設計書 [22-cross-platform-runtime.md](../ClaudeCodeスキルの設計書/22-cross-platform-runtime.md)「Python 標準ライブラリを実行基盤の正本とし、`.sh` / `.js` を新規生成しない・shell script を成果物として配布しない(macOS 向けでも)」の規約に違反するため、`scripts/link_master_plugins.py`(Python 標準ライブラリ。`os.symlink` / `pathlib`)へ全文書で改めた。同スクリプトは OS 判定(mac / linux / windows / unknown — WSL は Linux 扱い)を行い、**unknown は fail-closed 停止**する仕様として記述する。
- **判断記録(2026-07-11・ユーザー要望)— symlink 機構と Windows の関係**: 本節の symlink 引用は**マスター保守者マシン側の機構**であり、**企業側マシン(形態(ii))には不要**(企業側は版固定 bundle の配布を受けるため symlink を使わない — README §4 伝播範囲マトリクス)。したがって**企業側が Windows でも symlink 機構は障害にならない**。保守者や運用代行者が Windows マシンで symlink 引用を使う場合の注記: NTFS の symlink 作成には**開発者モードまたは管理者権限**が必要。WSL 内は Linux 扱いで通常どおり動作する。
- **参照元の一般化**: harness/plugins/へのsymlink引用は同一マシンの任意ディレクトリから許可する。編集起点はharnessに限定する運用規約とし、setupスクリプト再生成・doctor検査を必須にする。正本はREADME §4。
- 形態(i)(運用代行)で業務テンプレ系 plugin(`company-master` 等)が必要な場合も、同一の symlink 引用規約(README §4 の一般形)を適用する(企業 repo へ複製しない)。
- symlink 接続とテナント選択(activate)は別物である点に注意: 本節の symlink は「共有 plugin 実体の引用」、テナント仕様 §6 の symlink は「どの tenant を有効にするか」。有効化・競合検出は既存契約のまま(テナント仕様 §6)。
- **`.gitignore` の完全ルールセット**: 企業 repo 側は「共有 plugin symlink のパス(例: `plugins/harness-creator` 等 11 本)+ `**/*.secret.*` + cache/run artifact」を ignore し、実値 config は ignore しない。マスター harness 側は方式A の per-file ignore(テナント仕様 §7 の表)と一致させ、**否定例外(`!`)による再包含は用いない**(旧規定の `tenants/*` + `!tenants/xlocal/` 方式は企業ハーネスモデル移行で廃止。やむを得ず否定例外を使う場合は「`tenants/*` → `!tenants/<slug>/` → 実値ファイルの再 ignore」の順序を厳守する)。`tenant-doctor` は `.gitignore` とテナント仕様 §7 の表の突合を検査する。
- `tenant-doctor` の追加検査(方式B): (a) `MASTER_LINK_SET` 11本の存在・向き(`HARNESS_ROOT` の実体へ到達)・broken link・余剰、(b) 業務実行時に harness 作業ツリーが dirty なら FAIL、(c) `keychain_prefix` 一意性は自 repo と納品テナント台帳(§6 Step R6)を横断して検査。

### 配備モード（排他的）

`tenant-doctor` は実行場所に応じて検査を切り替え、symlink と bundle を同時に必須化しない。

| mode | 対象 | 必須検査 |
|---|---|---|
| `maintainer_symlink` | 同一マシンの保守者・運用代行者 | `MASTER_LINK_SET` 11本の存在・向き・broken link・余剰、harness worktree clean（業務実行時は dirty を FAIL）、harness 正本 worktree の HEAD が main（または明示許容した branch/tag）であること |
| `customer_bundle` | 企業側マシン | release manifest、plugin 集合、source commit、各 sha256、install receipt、`core_compat`、`NEVER_DISTRIBUTE` 3本の不在 |

「read 専用」はファイルシステムの強制ではなく、編集起点を harness に限定する運用規約である。symlink 参照元から書き込めば harness 実体が変わるため、doctor は検査時点の dirty を検出し、業務実行を停止する。

## 4. 企業固有 plugin

企業専用の業務 plugin(その企業でしか使わないもの)は core に入れず企業 repo の `plugins/` に置く。

| 規則 | 内容 |
|---|---|
| 置き場 | `<company>-harness/plugins/<company>-<name>/`(命名は `<company>-` prefix 必須。core plugin との名前衝突を構造的に防ぐ) |
| 配布 | 企業 repo 自体を Claude Code の **独立 marketplace** にする(`.claude-plugin/marketplace.json` を企業 repo に置き、`/plugin marketplace add <path-or-repo>` で登録)。core の marketplace / `.claude-plugin/bundles.json` には**載せない** |
| lint との関係 | `lint-tenant-isolation.py` の検査対象は core のみ。`tenants/` 配下は**除外(allowlist)対象であり検査されない**(構築手順 Phase 2-4 と同一の語義: allowlist = 検査から除外するパスの列挙)。企業 plugin 内の企業固有値は許容される。`validate-plugin-completeness.py` は core marketplace のみを対象とし、企業 marketplace には干渉しない |
| core 依存 | 企業 plugin が core plugin の lib を使いたい場合も、`CONVENTIONS.md` 層A の自己完結規約に従い **vendored コピー + SSOT 同期 lint 方式**(README §5)を踏襲する。core への直接参照(相対 path import)は禁止 |
| 昇格 | 企業 plugin に汎用性が出たら、genericize(企業固有値を overlay 化)して core へ PR で昇格する。方向は「企業 repo → harness への PR」の一方向(構築手順 Phase 7 の貢献フローと同じ) |

企業pluginのconfig解決・秘密解決もcoreと同じ契約に従う。実値はtenant overlay、秘密は`<purpose>.<keychain_prefix>`名前空間の承認済みbackendへ置く(テナント仕様 §3)。既存configに格納先がある値のためにoverlay slotを増やさない。

## 5. 納品フロー

納品形態は 2 つを想定し、企業 repo の README に採用形態を明記する。

| 形態 | 内容 | 納品物 |
|---|---|---|
| (i) 運用代行 | 弊社マシンで企業 repo クローン上の tenant を有効化して業務を実行し、成果物のみ納める | 成果物 + 企業 repo(閲覧権) |
| (ii) 企業内運用 | private企業repoのGitHub Releaseから版固定bundleを導入し、企業が実行する。**`NEVER_DISTRIBUTE` 3本は提供しない** | 企業repo権限 + Release asset(bundle/manifest/checksum) + 導入README |

### バージョン整合

- `tenant.json` の `core_compat` は検証済み harness の **semver release tag** に限定する。commit SHA は release manifest の `source_commit` に分離し、`tenant-doctor` が tag・manifest・install receipt の一致を検査する(不一致は FAIL)。
- 納品時は企業 repo に tag を打つ(`delivery-YYYYMMDD`)。以後の更新は tag 間差分で説明できる。

### 納品前チェックリスト

1. 納品形態に応じた `tenant-doctor --mode maintainer_symlink|customer_bundle <company>` が全緑(overlay drift・keychain_prefix・core_compat・remote visibilityを含む)。
2. 秘密の不在検査: `git -C "$COMPANY_ROOT" grep -iE '(api[_-]?key|token|private[_-]?key|BEGIN [A-Z]+ KEY)'` がヒット 0(example の placeholder を除く)。
3. 企業 plugin がある場合: `<company>-` prefix 準拠・vendored SSOT lint 緑・企業 marketplace から install できること。
4. クリーンなマシン(または一時ユーザー)で企業 repo README の手順だけを頼りに導入し、代表 1 業務の e2e が通ること(構築手順 Phase 6-2 の config-only 検証と同型)。
5. 形態(ii)はREADME冒頭のOS成熟度ゲートを満たす。macOSはKeychain、Windowsは暗号化backend実装・E2E後、Linuxは平文XDGの企業基準承認または安全なbackendへの置換後に納品する。導入READMEにはpython3、Windowsの`core.autocrlf` / long paths、WSLはLinux扱い、unknown OSはfail-closedを記載する。
6. 形態 (ii) の場合: 配布 bundle に `NEVER_DISTRIBUTE` 3本が含まれないこと(`validate-plugin-completeness.py` の非配布ロック)、release manifest と install receipt の checksum が一致すること、および共有 plugin 実体・symlink が企業 repo に commit されていないこと。

### 企業向け Quick Start（企業 repo README の必須冒頭）

1. private 企業 repo を clone する。
2. private企業repoの指定GitHub Releaseから、検証済みbundle・manifest・checksumを取得してinstallする。
3. tenant config を設定する。
4. OS 別の資格情報ストアへ秘密を登録する。
5. `tenant-doctor --mode customer_bundle <company>` を実行し、全緑を確認する。

## 6. Runbook(Step R0-R7)

**前提**: 構築手順 Phase 6 全緑(移管・genericize・tenant 機構が完成済み)+ **Phase 7 完了**(冒頭の規定の再掲)+ **config-only 稼働の e2e 実証済み**(構築手順 Phase 6-2 — 初回 R1 着手前までに 1 回必須)。それ以前は実行しない。

### Step R0: 前提実装(初回のみ・1 PR)
方式B が依存する機能を初回 R1 の前にまとめて実装する(構築手順 Phase 4 のスコープ外分の割り当て先):
- `tenant-doctor` の方式B拡張: `maintainer_symlink` / `customer_bundle` のmode別検査(§3)、GitHub API等による remote visibility 検査、`.gitignore` とテナント仕様 §7 の突合、納品テナント台帳との keychain_prefix 横断検査(R6)、**upstream(harness)新リリース検出**(R7)。URL形式だけでprivateと判定しない。
- `tenant-init --repo <company>`: sanitized allowlist export から fresh history の企業ハーネスを生成し、export manifest・source commit・checksumを記録する。`tenants/<company>/` を同時生成する。
- `scripts/link_master_plugins.py`: 構築手順 Phase 7（PR-6）で実装済みの `link_master_plugins.py` を方式B(企業 repo からの引用)観点で検証し、必要なら拡張する(実装正本は構築手順 Phase 7。Python 標準ライブラリ(`os.symlink`/`pathlib`)。OS 判定つき・unknown は fail-closed 停止。`.sh` からの是正は §3 判断記録)。
- **`cross_platform_secret.py` の Windows backend を暗号化**: 現状の `%APPDATA%/xl-skills/secrets.bin`(base64・非暗号化)は納品基準未達。Credential Manager またはDPAPI等を候補に、SA JSON等の容量も含めて実証し、Windowsへの形態(ii)納品の前提実装とする。
- **`tenant-doctor` / `tenant-init` / `tenant-build` の OS 判定対応**(2026-07-11・ユーザー要望): mac / linux / windows / unknown の 4 分類(WSL は Linux 扱い)。**unknown は fail-closed 停止**(設計書 22 章)。

### Step R1: 企業 repo(企業ハーネス)作成
`tenant-init --repo <company>` で sanitized export を生成し、新しい private repo を fresh history で初期化する。元 `.git/`、`tenants/xlocal/`、実企業値、共有 plugin 実体が export manifest に含まれないことを確認する。`slug` / `keychain_prefix`(全テナント横断で一意 — 納品テナント台帳(R6)と突合)を確定。

### Step R2: 外部リソースの準備 + 実値の記入と Keychain 登録
1. **外部リソースのプロビジョニング**(config 差し替えの前提。doctor は列一致を検査するが作成はしない): Notion DB(必要スキーマ/列の作成)・Google SA の発行と Workspace 委任・Slack App(Bot Token)。「config 差し替えだけで導入」の正確な範囲は **plugin コード変更ゼロ**であり、外部リソース準備は含まれない(README §2)。
2. overlay 実値(notion-config / party_a / google-config)を記入して commit。Gmail impersonate は notion-config 内に置く。秘密はOS資格情報ストアにのみ登録する(企業repo側マシンの名前空間 = 企業slug)。mac = Keychain / windows = 暗号化backend(R0実装後) / linux = 企業基準で承認したbackend。正本はテナント仕様 §3。unknown OS は fail-closed 停止。

### Step R3: マスターへ接続

- 形態(i)・保守者マシン: §3 の手順で `link_master_plugins.py` を実行し、`tenant-doctor --mode maintainer_symlink <company>` を全緑にする。
- 形態(ii)・企業側マシン: 対象releaseのbundleをinstallし、`tenant-doctor --mode customer_bundle <company>` を全緑にする。symlinkは作らない。

### Step R4: 企業固有 plugin(必要な場合のみ)
§4 の規則で企業 repo 内に開発し、企業 marketplace として登録・install 検証。

### Step R5: 納品前検証
§5 のチェックリストを全件 PASS させ、`delivery-*` tag を打つ。

### Step R6: 納品
形態 (i)/(ii) に応じて repo 権限・bundle・README を引き渡す。納品記録(日付・tag・core_compat・形態)を企業 repo の README に追記する。加えて、harness 側ローカルの**納品テナント台帳**(Git 管理外・gitignore 対象。例: `tenants/.registry.local.json`)に slug・repo URL・core_compat・形態・連絡先を記録する(R7 の通知先列挙と keychain_prefix 一意性検査の入力。harness→tenant 参照を Git に持ち込まないため Git 管理外とする)。

### Step R7: 更新運用
伝播範囲の正本は README §4 の「改善の伝播範囲」マトリクス(symlink 参照元 = 即時 / 企業側マシン = 本 Step の Runbook による)。

- **core 更新の検出**: `tenant-doctor` が upstream(harness)の新リリース tag を検出して WARN する。**判断記録(2026-07-11)**: 更新伝播の自動化代替案(GitHub Actions 自動起票 / bot 通知 / tarball+スクリプト / marketplace update)のうち、既存の doctor 実行動線に乗り実装コストが最小の **doctor upstream 検出**を選定。
- **core 更新の伝播(形態 (ii) 向け更新 Runbook — `brew upgrade` 相当)**:
  1. harness 側リリース(tag)。
  2. 配布可能bundleを版固定で再生成(`NEVER_DISTRIBUTE` 3本の不在を検証し、release manifestとsha256を生成)。
  3. private企業repoのGitHub Release assetを更新し、企業側マシンでbundleを再install。
  4. 企業 repo を pull(`core_compat` 更新 commit を含む)。
  5. `tenant-doctor --mode customer_bundle <company>` 全緑を確認(更新検証)。
  通知先は納品テナント台帳(R6)から列挙する。`core_compat` を更新して doctor で再検証 → tag。
- **rollback(更新検証 FAIL 時)**: 前 release の bundle を再 install し、`core_compat` を前 tag へ戻して `tenant-doctor --mode customer_bundle <company>` を再実行する。
- **tenant 側の改善**: 企業 plugin の汎用化・core への要望は harness への PR で受ける(一方向正本の維持。構築手順 Phase 7)。形態(ii)の企業側は private harness への read 権限を持たないため、改善要望は企業 repo の issue/PR で受け、マスター保守者が代理で harness へ PR する(2 hop 経路)。harness への PR 主体は保守者に限定する。tenant の improvement-request DB の巡回(形態(i))または企業 repo issue への転記(形態(ii))を納品テナント台帳(R6)の連絡先運用に含める。

## 8. 禁止事項

- 共有 plugin(`harness/plugins/` の共有実体)を企業 repo に複製(commit)しない。企業 repo の器は sanitized export、core は保守者 symlink / 企業側 bundle で供給する。
- `NEVER_DISTRIBUTE` 3本を企業側マシンへ提供しない(bundle に含めない。マスター保守者マシン上の symlink 引用は「配布」に当たらない — README §4)。
- 企業 repo に秘密(token / SA 鍵 / keychain dump / `.env`)を commit しない。実値 config と秘密の区別はテナント仕様 §5 の各契約に従う。
- harness の Git 管理に方式B テナントの実体・参照(submodule 化を含む)を入れない。
- 企業 marketplace の plugin を core の `.claude-plugin/bundles.json` / `marketplace.json` に載せない。
- `tenant-doctor` が FAIL のまま納品しない(fail-closed は納品ゲートにも適用。公開 remote FAIL(§2)を含む)。
