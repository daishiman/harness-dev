# xl-skills → harness 移管計画

この文書は、`~/dev/dev/xlocal/xl-skills` の内容をこの harness に反映する Runbook である。2026-07-12 に実行を開始した。採用 snapshot と進捗は [`実行記録.md`](実行記録.md)、恒久手順と禁止事項は本書を正とする。

---

## 1. ゴール

- harness を plugin/skill の正本にする。
- `xl-skills` のレビュー済み `source_snapshot_sha` を固定し、その Git 管理成果物を harness に取り込む。
- harness 配下の取り込み対象領域から Git 管理外の古い情報・生成物・cache を排除する。
- XLOCAL 固有値を `tenants/xlocal/` に分離し、core は企業非依存にする。
- 将来、`xl-skills` 側から harness 側へ symlink 共有できる構造にする。

## 2. 非ゴール

- この手順整理段階では、実ファイル削除・コピー・symlink 作成をしない。
- Keychain / 1Password / `.env` / Google SA json / API token はコピーしない。
- **plugin の fork(コード分岐)を作らない。** 企業 repo は harness の sanitized allowlist export から fresh history で構築し、共有 plugin 実体も元 Git 履歴も複製しない([企業別リポジトリ運用.md](企業別リポジトリ運用.md) §1)。
- XLOCAL 固有の実値を core plugin に残さない。

1 行目は「今はやらない」であって「やらない」ではない。移管完了後に企業を追加・納品するときの手順(企業ハーネス構築・マスター引用接続・企業固有 plugin・納品)は [企業別リポジトリ運用.md](企業別リポジトリ運用.md) に定める(実行は構築手順 Phase 6 全緑が前提)。plugin fork(コード分岐)禁止と秘密情報の非コピーは、その段階でも変わらない恒久原則である(§9)。

## 2.5 反映方式の判断記録(2026-07-12)

背景: これまで `xl-skills` → harness の反映は**手動コピー**で行われ、snapshot 固定もレビューもないまま古い部分コピーが作業ツリーに堆積し、版ずれ・重複・衝突(§4.5 の半移行状態)を生んだ。再発防止のため、反映方式そのものを判断記録として固定する。

| 方式 | 判定 | 理由 |
|---|---|---|
| 手動コピー(rsync / Finder 等) | **廃止** | snapshot 固定なし・レビューなし・「部分コピーの上に足し算」になり、§4.5 の半移行状態を再生産する。現存する堆積物(doc/ 全ファイル mtime 同一秒 = 一括コピー痕跡)はこの方式の産物 |
| git fork / merge(`xl-skills` と履歴共有) | **棄却** | (a) `xl-skills` の git 履歴には XLOCAL 実値(実 DB ID・会社プロフィール・`.xl-skills` Keychain 名)が焼き込まれており、fork すると genericize(Step 6)後も**履歴から実値を掘り出せる**状態が harness に永続する。企業 repo は harness から作るため、混入リスクが下流へ波及する。(b) harness は既に独自履歴と `xl-skills` に無い成果物(`doc/マルチ企業展開/` 等)を持ち、後付けの fork 関係は unrelated histories で成立しない。fork で作り直すなら「中立な新規 repo 案」の変種となり、README §3 の棄却理由がそのまま当たる。(c) fork の利点である衝突検出付き差分取り込みは、Source Snapshot Gate(§5)+ disposition 表 + Step 3 dry-run + Step 6.5 差分再同期で既に代替されている |
| 承認済み snapshot の `git archive` 取り込み(§6 Step 2) | **採用(本計画の正)** | 履歴を持ち込まず、レビュー済み `source_snapshot_sha` の Git 管理ファイルだけを版固定で反映できる。genericize(Step 6)・正本切替(Step 7)に接続し、切替後は方向が harness → `xl-skills`(symlink)に固定されるため往復コピーが構造的に消える |

**暫定例外(doc/ 領域のみ)**: `doc/` 配下のドキュメント保持領域(設計書・notion-schema・参考Skill の参照対象)に限り、移管本番前の暫定措置として diff レビュー付きの一方向同期(`xl-skills` → harness)を認める(2026-07-12 実施済み。[クリーンアップ計画.md](クリーンアップ計画.md) §0)。`plugins/` `scripts/` 等のコード領域はこの例外の対象外であり、必ず Step 2-4 で反映する。

## 5. 実行前ゲートと棚卸し手順

### Source Snapshot Gate

「最新」は作業ツリーを意味しない。次の5条件をすべて満たす commit を `source_snapshot_sha` として採用し、archive・比較・再同期・bundle provenance で同じ SHA を使う。

1. `git fetch --prune` 後である。
2. 採用 branch の local HEAD と upstream の関係を確認し、behind commit をレビュー済みである。
3. staged / unstaged / untracked の採否を決め、供給元 worktree が clean である。
4. 採用 SHA と tree hash を記録し、移管レビューで承認済みである。
5. §3 の core 出力 path を `git ls-tree $SOURCE_SNAPSHOT_SHA -- <path>` で全件非空確認し、空の path は §3 表を先に更新して archive 対象から外す。

2026-07-11 の最終実測では、`xl-skills` local HEAD `0126c5c` はremote `main` `662bc67`に5 commit behindで、worktreeは未コミット変更6件（業務plugin 5件 + plugin plan 1件）。調査中にも件数が変化したため、固定値ではなく実行直前の`git status`を正とする。この状態の`git archive HEAD`は「最新」ではないため実行禁止。指定11 plugin自体にremote差分がなくても、marketplace / config lock等を含む移管全体のsnapshot gateは省略しない。

```bash
git -C ~/dev/dev/xlocal/xl-skills fetch --prune
git -C ~/dev/dev/xlocal/xl-skills status --short
git -C ~/dev/dev/xlocal/xl-skills rev-list --left-right --count HEAD...@{upstream}
git -C ~/dev/dev/xlocal/xl-skills rev-parse HEAD
git -C ~/dev/dev/xlocal/xl-skills rev-parse 'HEAD^{tree}'
```

### 削除・保持 disposition

リポジトリ全体を「残す / 消す」の二択にしない。実行前に全 top-level path を次のいずれかへ分類し、未分類が1件でもあれば削除へ進まない。単一 disposition を与えられない path は、全 leaf が単一 disposition になるまで sub-path へ再帰分割する。「ユーザーの未コミット作業」は `git status --porcelain` の出力行を leaf として分類する。

| disposition | 対象例 | 条件 |
|---|---|---|
| `PRESERVE` | `.git/`, `doc/マルチ企業展開/`, ユーザーの未コミット作業 | 削除・上書き禁止 |
| `REPLACE_FROM_SNAPSHOT` | 承認済み `plugins/`, `scripts/`, `tests/`, `installers/` 等 | path限定dry-run、backup、hash確認後に入替 |
| `MERGE` | `.gitignore`, `README.md`, `CONVENTIONS.md`, `Makefile`, manifests | 単純上書き禁止。両側の意図をdiffレビュー |
| `RETAIN_AS_DEPENDENCY` | `doc/ClaudeCodeスキルの設計書/`, `doc/notion-schema/`, 必要な reference / fixture | 参照・実行依存がある間は保持 |
| `REGENERATE` | `.claude/skills`, indexes, link farm, cache可能な生成物 | 正本から再生成できることを先に実証 |
| `DELETE_AFTER_REVIEW` | stale部分コピー、旧 `skills/` / `docs/`、cache、run artifact | 参照0・復元経路あり・ユーザー承認後だけ削除 |
| `EXCLUDE_SECRET_LOCAL` | `.env`, SA key, token, OS資格情報ストア | import / commit / archive 禁止 |

harness に実在する未分類 4 項目は次のとおり分類する。

| path | disposition | 根拠 |
|---|---|---|
| `analysis/` | `REGENERATE` | OCR 中間生成物。`xl-skills` に同等生成物あり |
| `eval-log/` | `RETAIN_AS_DEPENDENCY`(2026-07-12) | criteria・coverage・content-review のテスト fixture / CI 証跡として実行時に参照されるため、固定 snapshot の追跡済み集合を取り込む。単なる一時ログは今後も対象外 |
| `plugin-plans/` | `RETAIN_AS_DEPENDENCY`(2026-07-12) | 可視・永続の tracked deliverable であり、plugin 実装の設計根拠・task graph・plan coverage gate が参照する。transient progress / intermediate だけを ignore する |
| `.worktrees/` | `PRESERVE` | git worktree 実体。除去する場合も `rm` でなく `git worktree remove` 経由のみ |
| `doc/【コード共有有】Agent Skill大全…/` | `DELETE_AFTER_REVIEW` | 正本は `xl-skills` 側と確定(2026-07-12)。参照 0・復元経路ありを確認済み。実行手順・検証ゲートは [クリーンアップ計画.md](クリーンアップ計画.md) §2-3 |

**判断**: 現時点で `doc/マルチ企業展開/` 以外を全削除してはならない。本ディレクトリだけでも `doc/ClaudeCodeスキルの設計書/` の4文書へ依存し、plugins / scripts / tests / installers も他の `doc/` を参照する。削除可能範囲は依存閉包を機械抽出してから確定する。

### 棚卸し

実行時は、必ず次の順で棚卸しする。

```bash
# 供給元の Git 管理ファイル一覧
cd ~/dev/dev/xlocal/xl-skills
git status --short
git ls-files > /tmp/xl-skills.git-files.txt

# 供給元の Git 管理外ファイル一覧
git ls-files --others --exclude-standard > /tmp/xl-skills.untracked.txt
git ls-files --ignored --others --exclude-standard > /tmp/xl-skills.ignored.txt

# harness 側の現状
cd ~/dev/dev/個人開発/harness
git status --short
git ls-files > /tmp/harness.git-files.txt
git ls-files --others --exclude-standard > /tmp/harness.untracked.txt
git ls-files --ignored --others --exclude-standard > /tmp/harness.ignored.txt
```

この段階では削除しない。`/tmp/*.txt` を見て、以下を判断する。

- harness 側の untracked がユーザ作業か、古い生成物か。
- `plugins/`, `.claude-plugin/`, `scripts/`, `tests/`, `installers/` のどこを取り込み対象にするか。
- `doc/`, `analysis/`, `eval-log/` の扱いは上記 disposition 表で解消済み。原則は docs と、テスト・CI が直接参照する証跡 / reference のみに絞る。

## 6. 反映順

### Step 1: harness 側の作業保護

目的は、削除前に「残すべき未追跡情報」を判定できる状態にすること。

```bash
cd ~/dev/dev/個人開発/harness
git status --short
git diff --stat
```

必要なら `git diff` と untracked 一覧を保存する。実行時に未コミットのユーザ作業がある場合は、移管対象外として保護する。

### Step 2: 承認済み snapshot から Git 管理ファイルだけを export

`rsync ~/dev/dev/xlocal/xl-skills/ ./` のような丸ごとコピーは禁止する。代わりに `git archive` を使う。

```bash
cd ~/dev/dev/xlocal/xl-skills
SOURCE_SNAPSHOT_SHA="<reviewed-commit-sha>"
test -z "$(git status --porcelain)" || { echo "STOP: xl-skills worktree is dirty"; exit 1; }
git archive --format=tar "$SOURCE_SNAPSHOT_SHA" \
  plugins scripts tests installers .claude-plugin .github references plugin-plans \
  CONVENTIONS.md Makefile config-version-lock.json .pre-commit-config.yaml sitecustomize.py \
  doc/GCP-Gmail送信設定手順.md \
  > /tmp/xl-skills.git-managed.tar

# Step 6.5 の差分再同期で使う基準 commit を記録する
git rev-parse "$SOURCE_SNAPSHOT_SHA" > /tmp/xl-skills.import-base.txt
git rev-parse "$SOURCE_SNAPSHOT_SHA^{tree}" > /tmp/xl-skills.import-tree.txt
```

ディレクトリの対象 path は §3 の core 出力 7 pathに、tracked 計画成果物 `plugin-plans/` を加えた定義である。repo-root 統治4ファイルと subprocess coverage 起動フック `sitecustomize.py`、plugin が直接参照する GCP 設定正本も依存閉包として同時に取り込む。harness 側の現物は §4.5 のとおり stale な未追跡コピーしかないため、ここで一緒に取り込む。対象を増減する場合は §3 の表を先に更新し、このコマンドを追随させる。

### Step 3: harness 側の Git 管理外情報を dry-run

```bash
cd ~/dev/dev/個人開発/harness
git clean -nd plugins scripts tests installers .claude-plugin .github references
git clean -ndX plugins scripts tests installers .claude-plugin .github references
```

`-n` の結果をレビューし、削除してよいものだけを確定する。実削除はこの文書の次段階で承認してから行う。

### Step 4: core を反映

承認後、Step 2 の archive を harness に展開する。展開後、企業固有値を core に残したまま完了扱いにしない。

**統治 4 ファイルの上書き禁止**: `CONVENTIONS.md` / `Makefile` / `config-version-lock.json` / `.pre-commit-config.yaml` は tar の単純上書きではなく **diff マージ**で反映する。特に構築手順 Phase 0-2 で `CONVENTIONS.md` に追記した**テナント軸の節**を、`xl-skills` 版(テナント軸を持たない)で上書き消失させない。テナント軸の節の残存を PR-2 の完了確認に含める(§8)。

### Step 5: XLOCAL 実値を tenant へ退避

core に存在する実 DB ID、会社プロフィール、Keychain service 固定名、実ドメインを `tenants/xlocal/` に移す。秘密情報はファイル化しない。

**順序の正**: 実値の移動は「tenant へコピー(Step 5)→ 一致検証 → core から除去(Step 6)」の原子順序で行う。本計画の Step 5 → Step 6 の順を正とし、構築手順 Phase 2(genericize)/ Phase 3(tenant 移設)も同一 PR 内でこの順序に従う。除去を先行させると実値がどこにも存在しない区間が生じる(復元が git 履歴頼みになる)ため、先抜きを禁止する。

### Step 6: genericize

core 側は placeholder + fail-closed にする。

- Notion DB ID は `tenant.json` / `notion-config.json` 由来で解決する。
- Keychain は `keychain_prefix` から `service = "<purpose>.<prefix>"` を導出する。
- 未設定時に `xlocal` / `xl-skills` へ戻らない。

### Step 6.5: 切替直前の差分再同期

Step 2-4 の取込後も、`xl-skills` は XLOCAL の実運用で稼働し変更が蓄積する。Step 7 の正本切替前に、Step 2 で記録した基準 commit からの差分をレビューし、1 件ずつ「harness へ取り込む / 破棄する」を判定する。判定対象には `doc/ClaudeCodeスキルの設計書/`・`doc/notion-schema/` など doc/ 差分も含め、取り込むか破棄するかを個別に判断する(§3 のとおり移管完了後の正本は harness 側)。

```bash
git -C ~/dev/dev/xlocal/xl-skills diff \
  "$(cat /tmp/xl-skills.import-base.txt)"..HEAD --name-status
git -C ~/dev/dev/xlocal/xl-skills diff --name-status
git -C ~/dev/dev/xlocal/xl-skills diff --cached --name-status
```

判定なしに Step 7 へ進むと、`xl-skills` 側の取込後 commit が黙って harness 版に置換される(サイレントロールバック)。差分ゼロを確認するか、再同期を終えてから切り替える。

**中間再同期の dampener(差分蓄積の自己強化ループ対策)**: 切替直前の単回レビューに差分を溜め込まない。基準 commit からの差分が **20 ファイルを超えた時点、または 2 週間経過のいずれか早い方**で中間再同期レビューを行い、取り込み / 破棄を判定して基準 commit(`/tmp/xl-skills.import-base.txt`)を更新する。これにより「差分蓄積 → レビュー負荷増 → 切替遅延 → さらに蓄積」のループを断つ。

### Step 7: 正本方向を固定

移管完了後の編集方向を固定する。

```text
harness/plugins/*  = 正本
xl-skills/plugins/* = 後続で symlink 化(既定)
```

symlink を既定とする。read-only mirror は、バックアップツール互換など symlink 起因の問題が実測された場合に限る代替とする(構築手順 Phase 7 と同一の決定)。symlink 化は別 PR/別手順に分離する。
